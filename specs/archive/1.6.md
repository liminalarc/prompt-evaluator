## Problem
Spec 1.1 delivered copy-in ingestion as **paste-only** — a user types or pastes a prompt's
text into the create / add-version forms. The 1.1 scope also named importing a prompt's text
**"from a file"**, which was split out here (deferral reconciliation, per *No silent
deferrals*). This spec adds file-based import so owners aren't hand-pasting large prompts, and
opens the door to bulk-importing several prompts/versions at once. "Shipped" means a user can
load prompt content from a file (and ideally many at once) into the registry through the SPA.

## Value
As a prompt owner I want to import prompt text from a file (and optionally several at once) so
that I don't have to hand-paste large or numerous prompts when copying them into the registry.

## Scope
**In:**
- Client-side **single-file import**: pick a text file, read its contents into the existing
  add-version flow (still one `POST` per version — reuses 1.1's API). The add-version form is
  the only content-bearing flow; `createPrompt` takes name/description only.
- **Bulk import** (confirmed in-scope at plan-time, built as a later slice): a structured JSON
  file describing multiple prompts and/or versions, imported in one action with per-row
  success/error reporting. Orchestrated **client-side** by looping the existing 1.1 POSTs — no
  new API endpoint, no `Prompt` aggregate change (honours the Out list + *No premature
  abstractions*). Exact JSON shape locked at slice 3's plan.
**Out:** Parsing app-specific prompt formats or scraping prompts out of the source apps (that
is capture territory — see 1.2). Zatomic sync (3.1). Any change to the `Prompt` aggregate.
Server-side batch API (explicitly rejected — bulk is client orchestration of existing POSTs).

## Acceptance criteria
- [x] A user can import a single prompt/version's content from a file via the SPA — proven by a unit test + one e2e
- [x] Malformed / oversized / non-text files are rejected with a clear message — proven by unit tests
- [x] (If bulk is in scope) multiple prompts/versions import in one action with per-row success/error reporting — proven by tests

## Plan (thin slices)
1. [x] Scope locked at plan-time: single-file **and** bulk both in scope, sequenced as slices;
   bulk is client-side orchestration (no new API). -> abd5e9c (plan)
2. [x] **Single-file import.** File picker in the add-version form (`prompt-detail.ts`) →
   `FileReader.readAsText` → existing `content` signal → unchanged `addVersion` POST. Pure
   `validateImportFile` helper (`import-file.ts`) gates non-text / oversized (>1 MB) / empty
   with a clear message via `ErrorState`. Unit tests (helper each-rejection + happy) + component
   test + one Playwright e2e (`setInputFiles`). -> 617ce9f
3. [x] **Bulk import (JSON).** JSON shape locked (array of prompts, each optional
   description + `versions[]`; folder omitted — imports into the org/folder in view). Import
   action on `/prompts` (`prompt-list.ts`): parse (`bulk-import.ts`) → validate → loop
   `createPrompt`/`addVersion` in the current org/folder → per-row success/error report
   (sequential, no transaction; partial-failure never stops later rows). Parser/schema unit
   tests + two component tests (happy + partial-failure) + e2e. -> ea8ed27

## Decisions
- 2026-07-12 — Split out of 1.1 by user decision during deferral reconciliation. 1.1 shipped
  paste-only; file/bulk import is tracked here as its own Phase-1 item.
- 2026-07-15 — Plan-time scope: user chose to keep **both** single-file and bulk in 1.6,
  sequenced as slices (single-file first). Nothing deferred.
- 2026-07-15 — Bulk is **client-side orchestration** of the existing 1.1 POSTs — a server-side
  batch endpoint was considered and rejected (avoids a `Prompt` aggregate/API change; *No
  premature abstractions*). Recorded on the Out list.
- 2026-07-15 — Single-file import targets the **add-version** form: it is the only
  content-bearing flow (`createPrompt` posts name/description only).

## Decisions (cont.)
- 2026-07-15 — Bulk JSON row does **not** carry a folder: imports land in the org + folder
  currently in view (mirrors single-prompt create); keeps the shape minimal.
- 2026-07-15 — Bulk import runs **sequentially** (rxjs `concatMap`), not in parallel — a clean
  per-row success/error report with no transaction; an early failure never blocks later rows.

## Verification / evidence
- `import-file.spec.ts` (7) — accepts text/* + extension-only .md/.prompt; rejects empty /
  >1 MB / explicit-non-text / unknown-extension, each with a clear message.
- `prompt-detail.spec.ts` (+2) — a picked text file fills the content signal; a non-text file
  raises an error and leaves content empty.
- `bulk-import.spec.ts` (7) — parses array + defaults; accepts no-versions; rejects invalid
  JSON / non-array / empty / missing-name / version-missing-content, naming the offending row.
- `prompt-list.spec.ts` (+3) — bulk import loops create/add-version POSTs with one report row
  each; a failing row is reported and the import continues; a malformed file issues no requests.
- `e2e/prompt-import.spec.ts` (2) — single-file import fills the version and adds it; bulk
  import of two prompts shows a 2-row success report and both land in the list.
- Full suite green: `npm run test:ci` 157 passing, `npm run lint` clean, `npm run build` OK.
- e2e smoke against the live compose stack (rebuilt web): both specs pass (3 passed).
