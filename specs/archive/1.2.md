## Problem
To evaluate a prompt we need inputs to run it against. The hard case — flagged as the main
risk for this project — is prompts whose input is the **output of an upstream SLM in the
app**. Synthetic-only inputs would test a distribution that doesn't match production and
produce a green dashboard that lies. We need a fixture model that is capture-first
(ground truth sampled from the apps) and uses synthetic generation only to fill coverage
gaps, seeded from the captured examples.

## Value
As a prompt owner I want datasets of realistic inputs — including inputs shaped like the
upstream SLM's output — so that my eval scores reflect how the prompt behaves in production,
not on invented data.

## Scope
**In:**
- `Dataset` aggregate containing `Fixture`s. A `Fixture` records the prompt's input plus
  optional upstream context (e.g. the SLM output it was derived from) and an optional
  expected/reference output. Origin is tagged: `captured` | `synthetic`.
- **Capture-ingestion format** — a documented JSON schema the apps emit (input → SLM
  output → the prompt's actual input → optional downstream result), plus an ingestion
  endpoint that lands captured tuples as fixtures. Anonymization/redaction hook at ingest.
- **Synthetic generation** — an eval-runner endpoint that, given N captured example
  fixtures for a dataset **plus operator guidance** (coverage goals, edge-case / adversarial
  variants to target, constraints), generates additional SLM-shaped inputs to fill gaps;
  results are tagged `synthetic` and linked to their seed examples. The generation prompt is
  **guided, not fixed** — the operator steers *what* gets generated, seeded by the captured
  distribution so it doesn't drift off-distribution.
- Angular: dataset list, fixture browsing, origin filter (captured vs synthetic), trigger
  synthetic generation.

**Out:** running prompts against datasets / scoring (1.3). Live in-app instrumentation is
the apps' responsibility — this spec owns the *format* and *ingestion*, not the app-side hooks.

### eval-runner ↔ .NET contract (locked at plan sign-off)
Wire format is **snake_case JSON** (`[JsonPropertyName]` on the .NET DTOs; Pydantic is
naturally snake_case). `POST /generate-fixtures`:
- req: `{ seed_examples: [{input, upstream_context?, expected_output?}], guidance:
  {coverage_goals?, edge_cases?, constraints?}, count }`
- resp: `{ fixtures: [{input, upstream_context?, expected_output?, seed_index?}] }`

## Decisions
- 2026-07-11 — Capture-first, synthesize-to-fill. Synthetic generation must be seeded from
  captured examples so the distribution matches; a fixture is treated identically by the
  harness regardless of origin, but origin is always recorded.
- 2026-07-12 — Distinguish three prompt roles: the **subject** prompt under test (registry,
  1.1), the **generator** prompt that produces test cases (this spec), and the **judge**
  rubric that scores outputs (1.3). The generator prompt is **guided** — operator guidance
  (coverage/edge/adversarial targets) on top of the seed examples steers what gets
  generated. The generator and judge prompts are themselves prompts and are candidates to
  live in the registry and be versioned/reviewed later — noted as a direction, not built here.
- 2026-07-12 — **Redaction is a concrete `FixtureRedactor` in Application** (pure regex PII
  scrubbing), injected into the capture handler — not a port. Redaction has no EF/HTTP
  dependency, so it needs no seam; per *No premature abstractions*, promotion to an
  `IRedactor` port is deferred until a second redaction strategy actually appears. (User
  decision at plan-time.)
- 2026-07-12 — Wire contract with eval-runner is **snake_case JSON** via `[JsonPropertyName]`
  on the .NET DTOs (the skeleton `/echo` DTOs only had single-word fields, so no
  snake/camel mismatch had surfaced before). Contract locked at sign-off (see Plan).
- 2026-07-12 — `Dataset` mapped as an EF Core **owned collection** of `Fixture`s (table
  `fixtures`), mirroring the 1.1 `Prompt`/`PromptVersion` decision; the repository always
  loads a dataset with its full fixture list. `GET` reads call `IDatasetRepository`
  directly (no premature query handlers), matching the 1.1 read convention.
- 2026-07-12 — eval-runner synthetic generation uses the Anthropic SDK with **structured
  output** (`output_config.format`, never free-text parsing); default model
  `claude-opus-4-8`, overridable via `EVAL_RUNNER_MODEL`. The Anthropic client is mocked at
  the boundary in pytest — no live calls in the suite.
- 2026-07-12 — **Capture-schema mapping:** `promptInput`→`Input`, `slmOutput`→`UpstreamContext`,
  `downstreamResult`→`ExpectedOutput`. The schema's leading `input` (the SLM's own upstream
  input) is accepted for completeness but not persisted as a separate fixture field — the
  fixture keeps the spec's three text fields.
- 2026-07-12 — **Generator output is defensively validated:** a generated fixture with blank
  input (e.g. an "empty input" edge-case variant an LLM may emit) is skipped rather than
  aborting the whole batch — found via close-out smoke testing (`a57ee0c`).
- 2026-07-12 — **AC #6 generate-trigger e2e** (user decision at close-out): built a
  deterministic browser e2e against a **stubbed eval-runner** (`EVAL_RUNNER_STUB`, enabled by
  `docker-compose.e2e.yml`) so CI needs no live model/key. The stub is test-only and gated;
  the spec's e2e is skipped by default and runs via `E2E_EVAL_RUNNER_STUB=1`. No deferral —
  AC #6 is met in full by automated e2e.
