## Problem
With prompts (1.1) and datasets (1.2) in place, we need the harness that actually runs a
prompt version over a dataset and scores each output. Scoring must combine fast
deterministic checks with LLM-as-judge for fuzzy quality, under one abstraction so scorers
compose per dataset.

## Value
As a prompt owner I want to run a prompt version against a dataset and get scored results
so that I have objective evidence of how well that version performs.

## Scope
**In:**
- `EvalRun` aggregate: a `Prompt × Version × Dataset` execution producing one `Score` per
  fixture per scorer. Every run persisted; nothing overwritten. Owns `FixtureRun` entities
  (captured `ModelOutput` + latency/cost per fixture), each owning its `Score`s.
- `IScorer` abstraction with two implementations this spec:
  - **Deterministic** (in-process): regex match, JSON-schema validation, exact/fuzzy
    match, latency/cost capture.
  - **LLM-judge** (→ eval-runner): rubric-based structured scoring via Claude, returned as
    a structured verdict (tool-use / structured output, never free-text parsing).
- Scorer selection/config **persisted per dataset** — a `ScorerConfig` aggregate keyed to
  a dataset (`scorer_configs` table), configured once and reused by every run so version
  comparisons stay like-for-like. Does **not** modify the archived 1.2 `Dataset` aggregate.
- **Judge model selection**: the LLM-judge's Claude model is configurable (Opus 4.8 default;
  Sonnet 5 / Haiku 4.5 for cheaper high-volume judging) and is **part of the Scorer's
  identity** (`ScorerDescriptor` = Kind + Config + JudgeModel, stable `Identity`). Changing
  the judge model yields a distinct scorer — and therefore a distinct score series — so
  trend/regression comparisons (1.4) only ever compare like-for-like.
- API to trigger a run and fetch results; Angular view of a run's per-fixture scores plus
  scorer-config + trigger UI.
- The prompt output itself is produced by calling the prompt's target model through the
  eval-runner (the same service that hosts the judge).
- The 0.1 walking-skeleton echo round-trip is preserved via a dedicated `POST /api/echo`
  seam check (home page repoints to it), freeing `EvalRun` to become the real aggregate.

**Out:** human review as a scorer (2.1). Trend/regression analytics across runs (1.4).
Non-Claude judge/subject execution (1.5) — `TargetModel` stays opaque, per 1.1.

## Acceptance criteria
- [x] An `EvalRun` executes a prompt version over every fixture in a dataset and persists a `Score` per fixture per scorer — proven by Application + Infrastructure tests
- [x] Deterministic scorers (regex, JSON-schema, exact/fuzzy, latency/cost) work and are pure-unit-tested — proven by Domain/Application tests
- [x] The LLM-judge scorer returns a structured verdict from a rubric — proven by an eval-runner test (model mocked) + an Application test against a faked `IEvaluationRunner`
- [x] Scorers are selected/configured per dataset and compose — proven by a run using both scorer types
- [x] The Angular UI shows a run's per-fixture scores — proven by unit + e2e
- [x] Runs are append-only; re-running never overwrites a prior run — proven by a test
- [x] The judge model is configurable and forms part of the Scorer identity; two runs judged by different models record as different scorer series — proven by a test

### Seam contract (locked at sign-off)
`IEvaluationRunner` gains: `ExecutePromptAsync(content, targetModel, input, upstreamContext) → (output, latencyMs, costUsd?)` (→ `/execute-prompt`); `JudgeAsync(rubric, input, output, expected, judgeModel) → (score 0–1, passed, rationale)` (→ `/judge`, structured output). Judge models via the `claude-api` skill — default `claude-opus-4-8`, alternatives `claude-sonnet-5` / `claude-haiku-4-5`; never hardcoded from memory.

## Decisions
- 2026-07-11 — One `IScorer` abstraction; deterministic in-process, LLM-judge delegated to
  eval-runner. Judge verdicts are structured (tool-use), never parsed from prose.
- 2026-07-12 — The judge model is part of Scorer identity (Prompt × Version × Dataset ×
  Scorer). Changing it starts a new score series rather than silently shifting an existing
  one, so regression detection only compares like-for-like. Subject-model selection lives on
  the `PromptVersion` (1.1); non-Claude judge/subject execution is spec 1.5. Consult the
  `claude-api` skill for current model ids — never hardcode from memory.
- 2026-07-12 — Scorer config is **persisted per dataset** (new `ScorerConfig` aggregate +
  `scorer_configs` table), not passed ad hoc per run, so re-runs and version comparisons
  reuse an identical scorer set — the stable foundation 1.4 regression tracking needs. The
  archived 1.2 `Dataset` aggregate is left untouched. (User decision at plan-time.)
- 2026-07-12 — `EvalRun` is **repurposed** into the real aggregate (its 0.1 comment always
  foretold this). The walking-skeleton echo round-trip is preserved via a dedicated
  `POST /api/echo` seam check that proxies eval-runner `/echo`; the home page +
  `round-trip.spec.ts` repoint to it. Keeps one clean `EvalRun` concept. (User decision.)
- 2026-07-12 — Latency and cost are modelled as deterministic **scorers** (per spec scope),
  so a run's per-fixture latency/cost captures appear as `Score`s with their own identity —
  uniform with every other scorer and directly comparable across runs.
