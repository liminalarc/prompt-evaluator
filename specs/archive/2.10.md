## Problem
The fields where we author real markdown — a prompt version's **Content** (the system prompt; Golf/Stormboard
prompts are all `##`-headed markdown with lists) and an LlmJudge **Rubric** — are plain `<textarea>`s. There's
no structure while writing and no way to see the rendered result, so multi-section prompts are hard to scan,
review, and get right. **Shipped** = those fields use a reusable markdown editor with an optional **preview**
that renders the markdown (read-only), the raw text staying the source of truth.

## Value
As a prompt author I want a markdown editor with a preview toggle so I can write and review structured
prompt/rubric content comfortably instead of eyeballing raw text in a bare textarea.

## Scope
**In:**
- A reusable **markdown-editor** component: a textarea (source) + an **Edit ⇄ Preview** toggle that renders
  the markdown to sanitized HTML (read-only). Source text is authoritative; preview never mutates it.
- Wire it into the **version Content** field ("+ Add version" / new-version-seeded-from-latest) and the
  **LlmJudge Rubric** field (dataset scorers). Existing immutability rules unchanged (content stays
  immutable-by-add; the editor is an authoring aid, not a new mutation path).
- Rendering is **self-contained + sanitized** (XSS-safe): a vetted markdown renderer + HTML sanitizer.
**Out:** WYSIWYG / rich-text editing (source-markdown only); markdown in short description fields (dataset /
fixture / prompt descriptions — a later, optional extension); AI-assisted authoring → [[2.7]]; syntax
highlighting beyond markdown; a full split-pane live preview (a toggle is enough for v1 — revisit if wanted).

## Acceptance criteria
- [x] Version **Content** and LlmJudge **Rubric** render through the markdown editor with an Edit⇄Preview toggle — unit + e2e
- [x] Preview renders markdown to **sanitized** HTML; a `<script>`/`onerror` payload in the source does not execute — test
- [x] Source round-trips unchanged (what you save === what you typed; preview is display-only) — test
- [x] Immutability unchanged — content still can't be edited in place; the editor only appears on add/new-version — test

## Decisions
- 2026-07-19 — **Renderer: `marked`@18 + `DOMPurify`@3** (not `ngx-markdown`). Both self-contained, no external
  requests. Sanitized HTML is bound via `[innerHTML]` (a plain string), so **Angular's built-in sanitizer runs
  on it too** — DOMPurify + Angular = defense in depth; no `bypassSecurityTrustHtml`.
- 2026-07-19 — Preview is a **toggle** (Edit ⇄ Preview tabs), not a live split pane, for v1 (no premature
  complexity), per the original stance.
- 2026-07-19 — The editor renders **only for markdown-bearing fields**: version Content (always) and the
  **LlmJudge Rubric**. Regex/JsonSchema scorer configs are patterns/schemas, not prose, so they stay plain
  textareas (the editor is gated on `isJudge()`/`editIsJudge()`). The source `<textarea>` keeps its original
  `id`/`data-testid` so import + e2e selectors (`#content`, `scorer-config`) are unchanged.

## Verification / evidence
- `cd web && npm run test:ci` — 233 green. New: markdown-editor unit tests (edit default, typed-source
  round-trip, preview structure, **XSS payload inert** — a real `javascript:` anchor href is stripped by
  DOMPurify and `<script>`/`onerror` never execute); prompt-detail immutability (editor only on add-version,
  `<pre>` on existing versions); dataset-detail rubric-uses-editor / Regex-stays-plain.
- `npm run build` — AOT clean; bundle 887 kB (warning threshold 800 kB, **error threshold 1.2 MB** — well
  within). marked+DOMPurify add ~78 kB.
- e2e `web/e2e/markdown-editor.spec.ts` — preview renders `##`/list structure, Edit round-trips the source
  unchanged, and an `<img onerror>` payload fires **no dialog** and does **not** mutate `window`.

## Deferrals
- markdown in short description fields (dataset/fixture/prompt), WYSIWYG, syntax highlighting beyond markdown,
  and a full split-pane live preview remain **out** (as scoped) — no spec home needed (explicit v1 boundaries,
  revisit only if wanted). AI-assisted authoring stays homed at [[2.7]].
