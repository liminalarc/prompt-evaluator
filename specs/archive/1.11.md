## Problem
Regression detection (1.4) flags a version only when the mean score drop exceeds a threshold (0.05)
**and** a paired t-test finds it significant (p < 0.05). The significance guard returns `null` for
fewer than 2 matched fixtures, so a **catastrophic drop on a single fixture** (e.g. 1.0 ‚Üí 0.0, a
pass flipping to a fail) is **not flagged at all** ‚Äî surfaced during dogfooding, where a deliberately
worse prompt version dropped the only labeled case from 1.0 to 0.0 and the Regressions panel stayed
empty. Two issues: (1) the empty-state copy is misleading, and (2) whether severe small-sample drops
should be surfaced is an open product call. "Shipped" means the analytics view communicates
small-sample drops honestly (and, if decided, surfaces them as unverified).

## Value
As an operator I want the analytics to tell me clearly when a version got worse but the sample is
too small to be statistically certain, so that a real regression on one or two fixtures isn't
silently hidden.

## Scope
**In:**
- **Fix the misleading empty state.** "No regressions beyond the threshold" renders even when the
  drop far exceeds the threshold but significance couldn't be established (n < 2). Distinguish the
  cases: "no meaningful drop" vs. "drop detected but not statistically significant ‚Äî add more
  fixtures to confirm."
- **Small-sample surfacing (product decision ‚Äî see open question).** Decide whether a drop that
  clears the threshold but lacks significance (n < 2, or n ‚â• 2 with p ‚â• alpha) is surfaced as an
  **unverified / possible** regression (distinct styling) rather than omitted. Keep the confirmed
  flag semantics unchanged.
- Wherever the trend/regression/comparison views disagree in what they imply (the trend line and
  Compare view both show the drop; Regressions shows nothing), reconcile the messaging so they read
  consistently.

**Out:** Changing the core detector math (threshold + paired t-test + zero-variance‚Üísignificant
short-circuit) unless the product decision requires it. New scorers. Backend perf.

## Decisions
- 2026-07-14 ‚Äî Raised during dogfooding: v2 (naive prompt) dropped the single labeled fixture 1.0‚Üí0.0;
  Regressions panel said "No regressions beyond the threshold" though the drop was total. Root cause:
  `RegressionDetector.OneSidedDropPValue` returns null for n < 2, so no flag. Note the existing
  zero-variance short-circuit means adding a 2nd consistently-regressing fixture *would* flag it.

## Open questions
- ~~Surface an insignificant-but-threshold-clearing drop as unverified, or just message "not enough
  fixtures"?~~ **Resolved: surface as unverified** ‚Äî a distinct, muted `--sb-warning` "Possible ‚Äî
  not enough data to confirm (add more fixtures)" treatment, never the confirmed error styling.

## Verification / evidence
- **Detector** (`RegressionDetector`): drops within threshold still skipped; a threshold-clearing
  drop is now always returned, tagged `RegressionConfidence.Confirmed` (p<alpha) or `Unverified`
  (pValue null for n<2, or p>=alpha). Core math (threshold, paired t-test, zero-variance
  short-circuit) unchanged. Application.Tests cover single-fixture 1.0‚Üí0.0 ‚Üí Unverified, noisy
  ‚â•2-fixture drop ‚Üí Unverified, within-threshold ‚Üí none, and confirmed stays Confirmed.
- **Api**: `RegressionFlagResponse` carries `confidence`; Api.Tests assert `"Confirmed"` on the
  significant case and `"Unverified"` on a single-fixture drop.
- **Web**: confirmed regressions render as before; unverified render in a muted `--sb-warning` block
  (`regression-unverified`); the "No regressions" empty state shows only when there's no
  threshold-clearing drop at all. Dashboard "open regressions" summary filtered to Confirmed.
- **Suites:** backend Domain 99 / Application 80 / Infrastructure 35 / Api 65; web Karma 126 + build;
