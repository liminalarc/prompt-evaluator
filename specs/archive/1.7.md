## Problem
The registry (1.1) is a **flat list** of prompts, and everything *about* a prompt lives apart
from it: datasets are global (`/datasets`, unattached to any prompt) and analytics are a
separate dashboard keyed by a `Prompt √ó Dataset` pair. Owners can neither organize prompts nor
see a prompt as a whole.

This spec makes the **prompt the unit everything hangs off** ‚Äî its versions, its datasets, and
its analytics **live together** ‚Äî and organizes prompts into a **folder tree** (top-level folders
with **optional subfolders**) so they're browsable by area. It also establishes the boundary
multi-user access is later granted on: **permissions attach to the top-level folder** (subfolders
inherit), which spec 4.1 builds on.

"Shipped" means: a `Dataset` belongs to exactly one `Prompt`; a prompt can live in a folder, be
browsed by folder, and be moved between folders; and a single prompt-workspace page shows that
prompt's versions, datasets, and analytics together ‚Äî all through the SPA.

## Value
As a prompt owner I want each prompt to be a self-contained workspace (versions + datasets +
analytics in one place), organized into folders by area, so that I can find and manage everything
about a prompt at a glance ‚Äî and so access can later be granted per top-level folder.

## Scope
**In:**
- **`Folder` aggregate** forming a tree: name, optional parent. A prompt belongs to exactly one
  folder, with a **default/root folder** for otherwise-unfiled prompts.
- The **top-level folder is the permission unit** ‚Äî the model resolves a prompt's top-level
  ancestor cheaply (store the root ref or walk parents); 4.1 grants access there and subfolders
  inherit.
- **`Dataset` ‚Üí `Prompt` ownership** (the reshape) ‚Äî a `Dataset` gains an owning `PromptId`; it is
  created under a prompt, and running a prompt against a dataset from a *different* prompt is
  rejected. Retrofits the 1.2 dataset model to single-owner.
- **API:** create/rename/move folders; move a prompt to a folder; list prompts by folder; fetch
  the folder tree; datasets nested under a prompt; analytics scoped to a prompt.
- **Angular:**
  - A folder tree / sidebar in the prompts browse UI, create-folder, move-prompt, and a
    breadcrumb; the 1.1 list becomes folder-scoped.
  - A **unified prompt-workspace page** ‚Äî one prompt's versions, its datasets, and its analytics
    on a single screen.

**Out:**
- The permissions/auth themselves (**4.1** ‚Äî this spec only fixes the boundary).
- **Shared / cross-prompt datasets (many-to-many)** ‚Äî deliberately excluded; re-homed to **1.8**
  (cloning is the likely future shape). See front-matter `deferrals`.
- Automatic foldering from captured app data (1.2). Moving *versions* between prompts.

## Acceptance criteria
- [x] Folders form a tree (top-level + optional subfolders); a prompt belongs to exactly one folder ‚Äî proven by Domain + Infrastructure tests (`FolderTests`, `FolderRepositoryTests`)

- [x] A dataset belongs to exactly one prompt; a run rejects a dataset owned by another prompt ‚Äî proven by `DatasetTests` + `EvalHarnessTests.Run_rejects_a_dataset_owned_by_a_different_prompt`

- [x] The SPA browses prompts by folder (tree navigation + breadcrumb, move a prompt) ‚Äî proven by `PromptList` unit spec + `e2e/folders.spec.ts`

## Decisions
- 2026-07-12 ‚Äî Grouping is a **folder tree with optional subfolders**; the **permission
  boundary is limited to the top-level folder** (subfolders inherit) ‚Äî user decision.
- 2026-07-13 ‚Äî **Datasets belong to a prompt** (single owner), not global and not folder-owned.
  Rationale: fixtures are inputs shaped to one prompt's contract, and the genuinely reusable
  cross-prompt concern (scorers) is already composable per dataset (`IScorer` √ó `ScorerConfig`);
  version-comparison (1.4) already covers "old vs new prompt." Sharing across prompts is excluded
  as a premature abstraction and re-homed to **1.8** (likely via *cloning* rather than
  many-to-many, per user). ‚Äî user decision.
- 2026-07-13 ‚Äî Scope grew to include a **unified prompt-workspace page** (versions + datasets +
  analytics together), kept in **one spec** rather than split ‚Äî user decision.

## Verification / evidence
- **Backend tests green:** Domain 89, Application 76, Infrastructure 21, Api 37 (xUnit;
  Infrastructure + Api use Testcontainers Postgres).
- **Web green:** 54 Karma unit tests + `ng build`; Playwright e2e (`folders`, `datasets`,
  `prompt-workspace`, `prompt-registry`) pass against the compose stack (4 passed, generate spec
  self-skips without the model stub).
- **End-to-end smoke (compose stack):** created a folder + subfolder, created a prompt and moved
  it into the folder, created a dataset under the prompt, listed datasets-by-prompt, fetched the
  folder tree ‚Äî all via the live API (`http://localhost:4240`).
- **Top-level ancestor (4.1 boundary)** resolves at any depth via a recursive CTE; folder-move is
  a single-row update (no denormalized root ref); folder-move cycles rejected via a descendants CTE.
- **Migration note (pre-prod):** `AddFoldersAndPromptOwnedDatasets` adds `datasets.PromptId` as
  non-null + FK. On a **fresh** DB (CI, Testcontainers) it applies cleanly. A dev volume holding
  pre-reshape dataset rows must be reset (`docker compose down -v`) since those rows have no owning
  prompt ‚Äî acceptable per the spec's pre-prod scope (no hosted data; production deploy is 3.2).
