## Problem
LitmusAI stamps a git-tag-derived version into the API assembly and eval-runner env and exposes an
aggregated `GET /version`, but **the web UI surfaces nothing** and **no service carries an
environment/channel signal**. Once releases are promoted to hosted **upper environments** (staging/prod
— spec 3.2), a user can't tell from the app whether they're on prod or a dev build. Stormboard
(`C:\Development\stormboard`, specs 142/163/164) solved exactly this, and the user asked to replicate it.
Its key lesson: **`ASPNETCORE_ENVIRONMENT` is not a reliable dev/prod discriminator** (its host reported
`Production` everywhere), so it added a **`channel`** signal (`DEPLOY_CHANNEL`, baked at build from a CI
build-arg by git-ref), shown as a **footer build chip** + **toolbar env badge**. "Shipped" means the SPA
shows the running version+environment the same way, channel-ready for the environments 3.2 introduces.

## Value
As a user/operator I want the running version + environment visible in the app so I can tell at a glance
which build and which environment (local/dev/staging/prod) I'm on — "this is production" vs. "this is a
dev build" must be unmistakable once releases are promoted to hosted environments.

## Scope
**In:**
- **Deploy-channel plumbing** — a `DEPLOY_CHANNEL` signal baked at build from a CI build-arg by git-ref
  (`v*` tag → `prod`, `main` → `dev`, local → `local`). Kept an **open string** so `staging`/further
  targets drop in when 3.2 wires real deploys; this spec is the *mechanism*, not the environments.
- **API payload** — expose `channel`, `environment` (`IHostEnvironment.EnvironmentName` — reported but
  NOT the discriminator), and `buildTime`, reachable by the SPA via its `/api` proxy.
- **Web version service** — a root signal service fetching the version once at startup; a failed/absent
  fetch leaves it null so indicators just don't render (no error surfaced).
- **Footer build chip** — subtle fixed monospace pill, channel-keyed: `prod`→`v<ver> · <sha7>`,
  `dev`→`dev · <sha7>`, `local`→`local`; full detail in a tooltip. Dev shows channel+sha (not a semver)
  so it never looks "behind" a freshly-tagged prod.
- **Topbar env badge** — uppercased channel (`LOCAL`/`DEV`/`STAGING`); **none in prod**; channel wins.
- **CI + Dockerfiles** — `ci.yml` derives `APP_CHANNEL` from git-ref; API `Dockerfile` maps it to
  `DEPLOY_CHANNEL`. No registry/deploy yet, so it's build-arg + ENV plumbing validated by `compose-smoke`.
- **Docs** — CLAUDE.md *Releasing* section + README; preserve "assert shape, never a literal".

**Out:**
- Hosted environments / deploy targets / registry / publish pipeline — **3.2**; this only makes display
  channel-ready (defaults `local`/`dev`/tag→`prod`). 3.2 sets `APP_CHANNEL` per target (incl. `staging`).
- Changing version derivation (`git describe` → `APP_VERSION`) — unchanged. Auth (endpoint stays anon).

## Verification / evidence
- **API** (`VersionEndpoints.cs`): flat `GET /api/version` → `{version, commit, buildTime, environment,
  channel}`, anonymous, no DB/runner dependency. `channel`=`DEPLOY_CHANNEL` (default `local`); `buildTime`
  =assembly last-write UTC. Aggregated `/version` unchanged. Api.Tests: 200+shape, channel default + flow.
- **Plumbing**: `Dockerfile` `ARG APP_CHANNEL=local`→`ENV DEPLOY_CHANNEL`; compose passes it; `ci.yml`
  derives from `$GITHUB_REF` and asserts `GET :4240/api/version` (through the web proxy) carries a version
  + `dev|prod` channel.
- **Web**: `VersionService` root signal via a 2nd `provideAppInitializer`; failure→null→nothing renders.
  `buildLabel`/`envBadge` channel-keyed computeds; badge test proves `channel:'dev'` beats
  `environment:'Production'`. `--sb-*` tokens only (theme-aware). Shell tests cover render + omit.
- **Suites (final):** backend Domain 99 / Application 80 / Infrastructure 35 / Api 68; eval-runner 24;
