## Problem
Individual run scores (1.3) only become useful when you can see movement across
improvements. We need to persist score history and surface trends and regressions per
`Prompt × Version × Dataset × Scorer`, so a prompt owner can answer "did v3 actually beat
v2, and did it regress anything?"

## Value
As a prompt owner I want to see how a prompt's scores change across versions so that I can
tell whether an edit was an improvement or a hidden regression before shipping it.

## Scope
**In:**
- Aggregation/query layer over persisted `EvalRun`/`Score` data keyed by
  `Prompt × Version × Dataset × Scorer`.
- **Regression detection**: compare a version against the prior version over the same
  dataset+scorer; flag statistically/threshold-meaningful drops. Configurable threshold.
- API endpoints for trend series and regression flags.
- Angular **dashboard**: score trend charts over versions, a regression-flag list, and a
  version-vs-version comparison view. Charts follow the `dataviz` skill.

**Out:** advisory / rewrite suggestions (2.2). Human review scores exist as just another
scorer once 2.1 lands and flow through the same analytics automatically.

## Acceptance criteria

- [x] Regression detection flags a version whose score dropped beyond the configured threshold vs. the prior version — proven by tests with crafted data

### Analytics contract (locked at sign-off)
- **Trend series** keyed `Prompt × Dataset × Scorer` (scorer by `ScorerDescriptor.Identity`).
  Each **point** is one **version**: the mean of the per-fixture `Score.Value`s of that
  version's **latest run** over the dataset (newest `CreatedAt` wins when a version has been
  re-run), plus pass-rate and fixture count. Points ordered by `PromptVersion.VersionNumber`.
- **Regression flag** on a version when **both** hold vs. the prior version in the same
  series: (a) `priorMean − currentMean > threshold` (default `0.05`, `?threshold=`), and
  (b) a **paired t-test** over per-fixture score deltas (matched by `FixtureId` across the
  two versions' latest runs) gives `p < alpha` (default `0.05`). Zero-variance consistent
  drop ⇒ significant; `n < 2` pairs ⇒ not flagged, `pValue: null`.
- **Endpoints:** `GET /api/analytics/trends?promptId=&datasetId=`,
  `GET /api/analytics/regressions?promptId=&datasetId=&threshold=`,
  `GET /api/analytics/comparison?promptId=&datasetId=&fromVersionId=&toVersionId=`.
- Read-only: **no Domain change, no migration.** Analytics are Application read-models over
  the existing append-only aggregates.

## Decisions
- 2026-07-11 — Score identity is `Prompt × Version × Dataset × Scorer`; all runs are
  append-only so trends are reconstructable and regressions provable against history.
- 2026-07-12 — **Statistical significance built in-scope** (not deferred): regression
  requires threshold **and** a paired t-test (`p < alpha`) over matched per-fixture deltas,
  so noisy LLM-judge series don't false-flag. Adds `MathNet.Numerics` (pure-managed, not an
  adapter) to Application for the Student-t CDF. (User decision at plan-time.)
- 2026-07-12 — A trend point uses a version's **latest run** (newest `CreatedAt`), not an
  average across all its runs — reflects the current measured state ("did my latest eval of
  v3 beat v2"). (User decision.)
- 2026-07-12 — Charts use **`@swimlane/ngx-charts`** (SVG, Angular-native, CSS-themeable)
  over inline SVG, sourcing series colors from computed `--sb-*` brand tokens (no hardcoded
  hex) so light/dark follows `data-theme`. Pulls `@angular/cdk` + `@angular/animations`
  (Angular-family peers, bundled — no CDN). (User decision.)
- 2026-07-12 — Analytics live as **Application read-models** (`TrendSeries`/`TrendPoint`/
  `RegressionFlag`) with pure computation, not Domain aggregates — they enforce no write-side
  invariants, and Domain stays about the append-only write model. No migration (read-only).

## Decisions (contd.)
- 2026-07-13 — **Latent 1.3 persistence bug fixed here** (`165ec2f`): `RunEvaluationHandler`
  reused one `ScorerDescriptor` CLR instance across every fixture's `Score`, but EF Core owned
  types can't be shared across owners — so any run over a 2+ fixture dataset threw
  `null value in column "scorer_kind"`. Only single-fixture runs had ever been persistence-tested,
  so it stayed hidden until 1.4's multi-fixture analytics tests. `Score` now copies the value-object
  descriptor. Not a scope narrowing — a fix required for 1.4 to function; surfaced to the user.
