## Problem
The Walking Skeleton (0.1) stood up the full stack under docker-compose as the **local + CI
runtime only** and explicitly deferred production orchestration ("hosting/k8s… established
later"). Nothing yet deploys LitmusAI to a reachable, hosted environment, and
`/flow-ship` therefore stops at a tagged, verified build. This spec establishes a real
deployment so releases reach users. "Shipped" means a tagged release is deployed to a hosted
environment and reachable, with migrations applied and health verified.

**This is a dev-only deployment** (no prod target yet), modeled directly on **Prism**
(`C:\Development\Prism`) — the sibling dev-only app that reuses the shared `liminalarc` AWS
account's resources. Prism's pattern is **"share the network, own your data + compute."**

## Value
As an operator I want LitmusAI deployed to a hosted environment so that users can reach
the app and each release is promoted there rather than only running locally.

## Scope
**In:**
- **Host = AWS App Runner + ECR + Terraform** (the Stormboard/Prism model). Account `973221168142`,
  region `us-east-1`. IaC is a single flat `infra/` stack (Prism shape), state in its own S3 bucket.
- **Reuse shared resources** (look up / pass by ARN — never create): the account-wide GitHub OIDC
  provider, and the `stormboard-dev` **VPC connector + VPC/subnets** (App Runner egress).
- **Own resources** litmus creates: ECR repo `litmus-ai`; `litmus-github-deploy` OIDC role (trust
  `repo:liminalarc/litmus-ai:*`, `iam:PassRole` scoped to `litmus-*`); App Runner access + instance
  roles; **its own RDS Postgres** (litmus is Postgres, the shared RDS is SQL Server — cannot reuse);
  Secrets Manager entries; App Runner services; tfstate bucket `litmus-ai-tfstate-973221168142`.
- **Two App Runner services** (Prism collapses client+server into one image; litmus does the same,
  eval-runner stays separate):
  - `litmus-ai` — **combined web+api image**: the .NET API serves the built Angular SPA from
    `wwwroot` (single origin; SPA calls `/api` same-origin; no nginx in the deployed image). Port
    8080, HTTP `/health` check. VPC-connector egress so it reaches Postgres.
  - `litmus-ai-eval-runner` — the existing Python `eval-runner/Dockerfile`, public URL, **protected
    by `EVAL_RUNNER_SERVICE_TOKEN`**; the api calls it via that URL with the existing
    `ServiceTokenHandler`. Default public egress (needs the Anthropic API); no VPC connector.
- **Postgres placement:** own `db.t4g.micro` Postgres **inside the shared `stormboard-dev` VPC's
  private subnets** (discovered via data sources by tag), new SG allowing `5432` only from the shared
  connector's app SG. Additive only — no Stormboard resource modified.
- Config + secrets in the target: `ConnectionStrings__Postgres` (built from the RDS endpoint +
  generated password), `EvalRunner__ServiceToken`, `ANTHROPIC_API_KEY` — all via Secrets Manager,
  injected as App Runner `runtime_environment_secrets`; never committed. DB migrations apply on
  deploy (API already migrates on startup).
- **Auth hardening for a multi-instance deploy (re-homed from 4.1).** (1) **immediate live-session
  invalidation on password reset** — switch cookie sign-in to `SignInManager` + wire the
  `SecurityStampValidator`; (2) **Data-Protection key persistence to Postgres** so the auth cookie is
  valid across replicas. See `specs/archive/4.1.md`.
- Health/readiness wiring and a post-deploy smoke (`/health`, `/api/version`) against the deployed
  URL. `APP_CHANNEL` per deploy target (the channel-aware version display itself is spec **3.3**,
  already done).
- CI/CD `deploy` job (Prism model): OIDC → ECR login → build+push both images → `start-deployment`
  on both services → poll RUNNING → post-deploy smoke. Extend `/flow-ship` + CLAUDE.md's *Releasing*
  section with the deploy step (main → dev; there is no prod target yet).

**Out:** App features. A prod environment (dev-only for now). Autoscaling/DR/observability depth
beyond a basic health smoke (later). **Concrete email provider — re-homed to 4.2** (dev deploy
doesn't need reset email; it belongs with the identity/SSO work).
