## Problem
A dataset can carry several scorers (deterministic + LLM-judge + human), each producing its own
independent series. There's no single number that says "did this version get better *overall*" — you
compare per-scorer deltas by eye (5.1's daily-briefing v1→v2: judge 0.55→0.88 **and** regex 5/5, read
separately). As scorer sets grow, that doesn't scale for ranking versions or spotting regressions at a
glance.

## Value
As a prompt owner I want a weighted composite score across a dataset's scorers so that I can track one
"overall quality" number per version, rank versions at a glance, and watch composite movement over time —
without eyeballing every scorer series.

## Scope
**In:**
- **Per-scorer weights** on the dataset's scorer config ([[1.3]] `ScorerConfig`); **default equal** (a new
  scorer joins at the same weight as the rest).
- A **weighted composite** per run — the weighted mean of each scorer's 0–1 value (pass/fail scorers
  contribute 1.0 / 0.0). Persisted with the run (or derivable), treated like any score.
- The composite as its own **Analytics trend series** ([[1.4]]) alongside the per-scorer series.
- A high-level **version-over-version change table**: rows = a prompt×dataset's versions, columns = each
  scorer's value + the weighted composite, with deltas — "what changed across versions," scorer-by-scorer
  and overall, at a glance. Extends [[2.8]]'s two-version compare to N versions.
- **Feed the weighted composite into [[1.16]]'s backport signals — both eligibility AND the single-target
  pick.** 1.16 ships *unweighted* interim rules: a version is eligible when it's ≥ Current on every shared
  same-scorer-config series and strictly higher on ≥1, and the **single backport target** is the eligible
  version with the highest *average of its per-scorer means over the series shared with Current*. Both
  have a known confound on a prompt whose **scorer config changed mid-history** (e.g. round-debrief's
  documented rubric swap): comparing raw means across versions scored on different configs/subsets is not
  apples-to-apples, so the target can land on an older high-on-the-old-yardstick version (v2) instead of
  the newer one the maintainer iterated toward (v7). Once weights exist, key eligibility **and the single
  target** off the **normalized weighted composite vs Current**, so a high-signal scorer (LLM judge)
  outweighs a low-signal one (RegEx) and cross-config comparison is normalized — the determination 1.16
  deferred here (origin: 1.16 plan-time + dogfood, user 2026-07-19).

**Out:**
- **Auto-tuning** weights (learn them from human labels) — future.
- **Must-pass / veto gating** (a scorer that zeroes the composite on fail) — deferred; the composite here
  is a *mean*, not a gate. Revisit once weighted means are in use.
- Changing any individual scorer's semantics.

## Acceptance criteria
- [x] A dataset's scorers carry weights (default equal) and are editable — proven by tests
- [x] Each run computes a weighted-mean composite over its scorer values — proven by Application tests
- [x] Analytics shows the composite as a trend series + a version-over-version change table (per-scorer + composite deltas) — proven by unit + e2e
- [x] The composite is correct when scorers are added/removed between versions (weights renormalize) — proven by a test
- [x] **[[1.16]] backport signals key off the normalized weighted composite** — both **eligibility** and
      the **single backport target** are determined by the weighted composite vs Current (replacing the
      interim unweighted rules), so a high-signal scorer (LLM judge) outweighs a low-signal one (RegEx)
      and cross-config comparison is normalized — **proven by a regression test on the scorer-config-change
      confound** (a round-debrief-shaped history where the raw-mean rule picks the old high-on-old-yardstick
      version but the weighted composite picks the maintainer-iterated one). This is the determination 1.16
      deferred here; it is **required scope**, not optional.

## Decisions
- 2026-07-18 — Split from [[2.8]] as its own spec (user call): weighting is a real capability (domain +
  analytics), not a UX fix. Composite = **weighted mean of 0–1 scorer values**; weights **per
  dataset-scorer, default equal**. Must-pass/veto gating and weight auto-tuning explicitly deferred.
  Origin: 5.1 dogfooding — with 2+ scorers per dataset, per-scorer deltas don't give one "overall" number.
- 2026-07-20 (plan sign-off) — **Weight lives on `ScorerConfig` (per dataset-scorer), default `1.0`
  (⇒ equal), and is excluded from `ScorerDescriptor.Identity`** so a reweight never forks a score
  series (weight is config metadata, not identity).
- 2026-07-20 (plan sign-off) — **Two composites, computed differently on purpose.** (1) The *headline*
  composite (trend series + change table) is per version per dataset = `Σ(wₛ·meanₛ)/Σ(wₛ)` over the
  scorers **in that version's run** — dividing by present weights auto-renormalizes on add/remove
  (AC4); answers "how good overall." (2) The *backport signal* ([[1.16]]) compares only over series
  **shared with Current** (same `Identity` = same config/rubric) as a **weighted delta normalized by
  Current's total configured weight**: `Σ_{s∈shared} wₛ·(candₛ−curₛ) / Σ_{s∈Current} wₛ`. Different
  questions ("overall quality" vs "safe real improvement on the shipped yardstick").
- 2026-07-20 (plan sign-off) — **Backport rule = safety floor + weighted target.** *Eligible* iff no
  per-scorer regression on any shared same-config series **and** weighted composite delta > 0 (retains
  1.16's "never flag a per-scorer regression" guarantee); the single *target* = argmax weighted delta
  vs Current, ties → later version. Equal weights reproduce 1.16's interim behavior, so existing
  `VersionStatusTests` stay green; the confound case flips old-v2 → iterated-v7 via the judge weight.
- 2026-07-20 (plan sign-off) — **Runs-table headline (`EvalRunSummaryResponse.MeanScore`) left as the
  2.19 heuristic** (judge-mean-else-overall), out of scope here — tighter slices, no risk to 2.19.
- 2026-07-20 (plan sign-off) — The spec's **Out** items (weight auto-tuning; must-pass/veto gating)
  name no spec home yet; left as declared-Out and **reconciled at close-out** per the deferral protocol
  (decide spec homes at the DONE-gate), not homed pre-emptively.
- 2026-07-20 (close-out, user decision) — both **Out** items re-homed into a single new stub
  **[[2.9a]]** (Composite scoring — advanced): must-pass/veto gating + weight auto-tuning. Recorded as
  `deferrals:` (both `to: 2.9a`); cross-linked. Chose one combined spec over two, since both extend
  the same composite machinery.
