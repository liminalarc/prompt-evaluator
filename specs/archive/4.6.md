## Problem
The admin Users page (4.3) manages **existing** users only — toggle admin, add/remove org
membership, set password (every endpoint is `/api/admin/users/{id}/…`). There's no way for an admin
to **create** a user; the only onboarding path is self-service `/register`. An admin should be able
to create an account directly (email + display name + an initial password), with **no email
required** — matching 4.3's no-email pattern. Invite-by-email onboarding is separate (see 4.2).

## Value
As a workspace admin I want to create user accounts directly so I can onboard people without relying
on self-registration or an email provider.

## Scope
**In:**
- `POST /api/admin/users` (global-admin gated, like the other admin-user endpoints) — creates a user
  via `IUserDirectory.RegisterAsync(email, displayName, password)`; returns the new user in the same
  `UserAccountDetail` shape the list uses (id, email, displayName, isAdmin=false, empty memberships).
- Validation surfaced as `400`: duplicate email + the Identity password policy (reuse
  `RegisterAsync`'s existing result errors) → inline errors in the UI.
- Web: a **New user** form on `/admin/users` (email, display name, password) that creates the user
  and shows it in the list. Org/role and admin are then granted with the **existing per-row
  controls** (no need to duplicate them in the create form).

**Out:** Invite-by-email / any email-based onboarding — **re-homed to 4.2** (needs the transactional
-email provider 4.2 wires). Bulk/CSV user import. Setting admin flag or org membership *at create
time* (use the existing per-row controls right after creating).

## Acceptance criteria
- [x] An admin can create a user (email + display name + password); it appears in the list and can log in — proven by `AdminUsersEndpointTests` (create → list → login) + Karma form spec
- [x] A non-admin calling `POST /api/admin/users` gets `403` — proven by a test
- [x] Duplicate email / weak password return a clear `400` the UI shows inline — proven by xUnit (400) + Karma (inline error)

## Decisions
- 2026-07-18 — Split from the admin-user-management surface (4.3) as a follow-on: 4.3 was scoped to
  "admin flag, org membership, passwords — no email" and excluded user creation. Origin: user request.
- 2026-07-18 — **Invite-by-email (Option B) re-homed to 4.2** (user decision): it needs the email
  provider 4.2 owns. This spec is the no-email, admin-types-a-password path only.
