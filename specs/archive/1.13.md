## Problem
[[1.5]] gave the eval-runner multi-provider support (Anthropic + OpenAI, routed by model id),
but model ids are still managed free-form in three disconnected places:
- **Target/subject model** is a **free-text `<input>`** (`web prompt-detail`) — a typo
  (`gpt4o`) is accepted, then fails at eval time as an opaque provider 400.
- **Judge model** is a **hardcoded frontend constant** (`dataset-detail.ts` `JUDGE_MODELS`),
  Claude-only — so an OpenAI judge can't be chosen in the UI even though the backend supports it.
- The eval-runner carries its own routing + pricing table + defaults.

There is no single source of truth for "which models are available," their provider, or which
roles they can serve. "Shipped" means: a managed **Model Catalog** drives **droplists** for
target and judge selection (no typing), stays in sync with what the providers actually support,
and an admin can manage the list without a code change.

Origin: the stale judge dropdown + free-text target were surfaced by 1.5 (whose scope was
eval-runner-only); this spec closes the UI/management gap.

## Value
As an operator/admin I want the available models managed in one place and selectable from
dropdowns (target **and** judge) so people pick valid models instead of typing ids, and I can add
a new provider model (e.g. a new OpenAI or Claude release) without shipping code.

## Scope
**In:**
- A **Model Catalog** on the .NET side (persisted, Postgres via EF Core), seeded with the
  currently-supported models — Claude Opus 4.8 / Sonnet 5 / Haiku 4.5, GPT-4o / GPT-4o-mini —
  each with: `id`, display name, provider, and the **roles** it can serve (subject / judge /
  generator). Carries **display-only pricing** (input/output $ per MTok, nullable).
- `GET /api/models` returns the catalog; the web replaces the free-text target-model input
  (`prompt-detail`) and the hardcoded `JUDGE_MODELS` constant (`dataset-detail`) with dropdowns
  fed by it — filtered to the right role (judge dropdown shows judge-capable models, etc.).
- **Provider availability** reflected in the catalog: a model whose provider has no configured
  credentials is not offered (or is clearly marked). The eval-runner stays the authority on which
  providers are configured — it exposes that (`GET /providers`), and .NET reflects it.
  (Enforcement still happens in the eval-runner: an unavailable model → clear 400.)
- **Admin management page** to add / edit / deactivate catalog entries. Gated by a new
  **workspace-level global-admin flag** (`AppUser.IsAdmin`) — the app's first role-based gate,
  distinct from the per-org roles of [[4.1]] (which are membership-only and never read for
  authorization today). Non-admins only consume the droplists.
- **Backward-compat:** existing prompt versions carrying an arbitrary target-model string still
  display and run; the catalog is additive, and a legacy/unknown id is surfaced (not silently
  dropped).

**Out:**
- eval-runner **routing/pricing** behaviour — 1.5 owns provider routing; pricing stays in the
  eval-runner. The catalog carries display-only metadata but does not drive eval-runner logic.
- **Live model discovery** (auto-syncing the catalog from provider APIs) — manual/seeded
  management here; discovery is a future spec (see `deferrals:`).
- **Per-org catalogs** — a single workspace-wide catalog for now; per-org customization is future
  (see `deferrals:`).
- A live **Modal** model in the catalog — depends on [[1.12]] landing the adapter first.

## Acceptance criteria
- [x] `GET /api/models` returns the seeded catalog (id, display name, provider, roles) — proven by an Api test
- [x] The target-model field is a dropdown sourced from the catalog (no free-text typing); a non-Claude model (e.g. `gpt-4o-mini`) is selectable — proven by a web test
- [x] The judge-model dropdown is sourced from the catalog (not a hardcoded constant) and offers judge-capable models across providers — proven by a web test
- [x] Only models whose provider is available/configured are offered (or clearly marked) — proven by a test (ties provider availability to the eval-runner)
- [x] An admin can add / edit / deactivate a catalog entry via an admin-only page; the change persists and drives the droplists; non-admins cannot — proven by Api + web tests
- [x] Existing prompt versions with a legacy free-text target model still display and run — proven by a test

## Decisions
- 2026-07-16 — Created as the follow-up to 1.5 (droplists + management for the multi-provider
  models).
- 2026-07-16 — Resolved at implement-time (user decisions):
  - (a) **Source of truth** — .NET-owned catalog; the eval-runner remains the provider-availability
    authority (`GET /providers`) and the execution pricing/routing authority. Catalog pricing is
    display-only, decoupled from the eval-runner `_PRICING` table.
  - (b) **Admin CRUD ships in this spec** (not split out), which means introducing role-based
    authorization for the first time.
  - (c) **Display-only pricing is included** (nullable input/output $ per MTok).
  - **"Admin" for the workspace-wide catalog** = a new **global-admin flag** (`AppUser.IsAdmin`),
    distinct from the per-org `OrgRole` (which is stored but never read for gating). Bootstrap admin
    seeded `IsAdmin = true`; surfaced to the SPA via `/api/auth/me`.
  - **Roles persistence** — three boolean columns rather than a `List<enum>`/child table (no
    `List<enum>`/JSONB precedent in the codebase; three fixed roles make `WHERE Can…` filtering
    clean).
  - **Seed** — model ids verified via the `claude-api` skill: `claude-opus-4-8` (5/25),
    `claude-sonnet-5` (3/15), `claude-haiku-4-5` (1/5) [Anthropic]; `gpt-4o`, `gpt-4o-mini`
    [OpenAI, prices null — no authoritative source, pricing optional]. All seeded with all roles.
