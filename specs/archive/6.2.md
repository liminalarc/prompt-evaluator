## Problem
6.1 introduces a versioned, config-backed .NET pricing table (input / output / cache-write / cache-read
+ rate version) made **authoritative for the ledger**. It deliberately left two other pricing sources
untouched (6.1 plan-time decision, "ledger table only"): the eval-runner `_PRICING` table
(`eval-runner/app/execution.py` — execution cost → `FixtureRun`) and the Model Catalog's display-only
`InputPricePerMTokUsd` / `OutputPricePerMTokUsd` (`src/Domain/ModelCatalogEntry.cs`, spec 1.13). So the
same model's price can differ across three places, and the catalog UI can show a rate that disagrees
with what the ledger charges. Shipped = the Model Catalog's display price **derives from** the
authoritative 6.1 pricing table, so catalog and ledger show identical rates from one source.

## Value
As a workspace admin, I want the price shown in the Model Catalog to match what the AI-usage ledger
actually charges, so I'm not reconciling three different numbers for the same model.

## Scope
**In:**
- The Model Catalog's **displayed** input/output price is sourced from the 6.1 authoritative pricing
  table (`AiUsagePricingOptions`) as the single source, rather than its separately-seeded DB columns.
- **Decision (pre-baked, non-destructive): keep `ModelCatalogEntry`'s price columns as an optional
  per-model override.** The `ModelResponse` displayed price is `override ?? tableRate` — when the entry
  has an explicit price it wins (author intent preserved), otherwise the authoritative table's rate is
  shown. No column is dropped; no data is lost; no destructive migration. Expose the effective price
  (and, optionally, a flag for whether it came from the table vs an override) on `ModelResponse`.
- **Document** the eval-runner `_PRICING` relationship: the .NET table is authoritative for what the
  catalog *shows* and what the ledger *charges*; the eval-runner keeps computing execution `cost_usd`
  for `FixtureRun` from its own `_PRICING` (a known, documented duplication — converging it is out).

**Out:**
- The ledger, cost snapshotting, and everything else in 6.1 (already shipped there).
- Dropping the `ModelCatalogEntry` price columns or any destructive schema change.
- Converging the eval-runner `_PRICING` table (documented, not removed).
- Cache-rate display in the catalog UI unless trivial.

## Acceptance criteria
- [x] Catalog display price for a model equals the 6.1 authoritative table's rate for that model — test
- [x] No third divergent price source remains unaccounted for (each of catalog / ledger / eval-runner
      has a documented owner) — inspection
- [x] Backend + web tests green; EF migration if columns change

## Decisions
- 2026-07-19 — **(pre-baked, non-destructive) Catalog displayed price = `override ?? authoritative
  table rate`.** `ModelCatalogEntry`'s `InputPricePerMTokUsd`/`OutputPricePerMTokUsd` columns are kept
  as an optional per-model override (author intent wins); when absent, the catalog shows the 6.1
  `AiUsagePricingOptions` rate — the same source the ledger charges against. No column dropped, no
  destructive migration (in fact **no migration at all** — the change is a read-time derivation).
- 2026-07-19 — **Effective price exposed on `ModelResponse`.** Added `EffectiveInputPricePerMTokUsd`,
  `EffectiveOutputPricePerMTokUsd`, and `PriceSource` (`override` / `table` / `none`). The existing
  override columns stay on the DTO (the admin edit form still edits the override). A new
  `IUsagePricing.GetRate(model)` port method surfaces the table rate without re-deriving it from
  token counts.
- 2026-07-19 — **eval-runner `_PRICING` left in place (documented, not converged).** It remains the
  owner of the per-execution `cost_usd` on `FixtureRun`; the .NET table is authoritative for the
  catalog display + the ledger. Documented inline in `eval-runner/app/execution.py`. Converging it is
  explicitly out of scope.

## Verification / evidence
- Port: `IUsagePricing.GetRate` (`Application/Ports/IUsagePricing.cs`) + `UsageRate`; implemented in
  `ConfigUsagePricing`.
- Mapper: `ModelResponse.From(entry, configuredProviders, pricing)` computes `override ?? table` +
  `PriceSource` (`Api/Models/ModelDtos.cs`); all `ModelEndpoints` call sites inject `IUsagePricing`.
- Tests: `ModelResponsePricingTests` (3 pure unit tests — table/override/none) + a `ModelsEndpointTests`
  assertion that a non-overridden model's displayed price is the table rate (`table` source). Web:
  `model.ts` DTO + model-admin table shows the effective price + a source badge, with a Karma test.
- Gates: backend `dotnet test` (ModelResponsePricingTests 3/3, ModelsEndpointTests 9/9), web
  `npm run test:ci` (269) + build + prettier, eval-runner pytest — all green.
