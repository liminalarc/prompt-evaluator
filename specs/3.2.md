## Problem
The Walking Skeleton (0.1) stood up the full stack under docker-compose as the **local + CI
runtime only** and explicitly deferred production orchestration ("hosting/k8s… established
later"). Nothing yet deploys LitmusAI to a reachable, hosted environment, and
`/flow-ship` therefore stops at a tagged, verified build. This spec establishes a real
deployment so releases reach users. "Shipped" means a tagged release is deployed to a hosted
environment and reachable, with migrations applied and health verified.

## Value
As an operator I want LitmusAI deployed to a hosted environment so that users can reach
the app and each release is promoted there rather than only running locally.

## Scope
**In:**
- Choose the hosting target (managed containers / k8s / PaaS) and image registry.
- Build + publish per-service images (the `GIT_COMMIT` build-arg already bakes the commit).
- Config + secrets management in the target (connection string, `EvalRunner__BaseUrl`,
  `ANTHROPIC_API_KEY`); DB migrations applied on deploy (API already migrates on startup).
- **Email provider for auth (re-homed from 4.1).** 4.1 builds the full password-reset/forgot flow
  behind an `IEmailSender` seam (NoOp/log in dev, SMTP configurable by env) but does not wire a
  concrete hosted provider — there was no environment to run it in. This spec picks and wires the
  real SMTP/email provider (credentials via env/secrets) so reset emails actually send in the
  deployed environment. See `specs/4.1.md`.
- **Auth hardening for a multi-instance deploy (re-homed from 4.1).** Two items that only matter once
  the app runs deployed/multi-replica: (1) **immediate live-session invalidation on password reset**
  — switch cookie sign-in to `SignInManager` + wire the `SecurityStampValidator` so a reset kills
  concurrent sessions (4.1 rotates the stamp but issues the cookie manually and doesn't validate it);
  (2) **Data-Protection key persistence to Postgres** so the auth cookie is valid across replicas
  (4.1 uses per-process keys). See `specs/4.1.md`.
- Health/readiness wiring and a post-deploy smoke (`/health`, `/version`).
- Extend `/flow-ship` (via CLAUDE.md's *Releasing* section) with the deploy step this adds.
**Out:** App features. Autoscaling/DR/observability depth beyond a basic health smoke (later).

## Acceptance criteria
- [ ] A tagged release deploys to the hosted environment and the app is reachable — proven by a post-deploy smoke against the deployed URL
- [ ] Migrations apply on deploy and secrets/config are supplied by the environment (never committed) — proven by a clean deploy
- [ ] `CLAUDE.md`'s Releasing runbook gains the deploy step; `/flow-ship` no longer stops at the tag — proven by inspection
