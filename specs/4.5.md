## Problem
[[4.4]] builds the **global-admin** organization surface ‚Äî a workspace admin can list all orgs
and manage each org's members from the Admin area (gated by the global-admin flag). But an org's
own **Owner** ([[4.1]] `OrgRole.Owner`) has no way to manage their org's membership from the org's
own page ‚Äî today membership is editable only from the admin surfaces ([[4.3]] Users page, [[4.4]]
Admin ‚Ä∫ Organizations). This spec adds an **owner-facing** member view on the regular org detail
page: an Owner (or a global admin) can see who belongs to the org, add/remove members, and set
roles ‚Äî scoped to that one org, without the workspace-admin flag.

Origin: split from [[4.4]] (user decision, 2026-07-17 ‚Äî 4.4 stays admin-only; owner-facing member
management is a different permission model played next).

## Value
As an org owner I want to manage my organization's members (add, remove, set role) from my org's
own page, so I don't need a workspace admin to adjust who has access.

## Scope
**In (to refine at implement-time):**
- Member view on the org detail page (the member-scoped `/api/organizations` surface): list members
  (email, display name, role); add member, remove member, set role.
- **Permission model = owner-or-admin, per org** (not the global-admin flag): gated by
  `OrgRole.Owner` on the target org **or** `IsGlobalAdminAsync`. Distinct from [[4.4]]'s
  global-admin-only endpoints.
- Reuse `IUserDirectory` grant/revoke/role operations already used by [[4.3]]/[[4.4]].

**Out:**
- Global-admin org-entity management (list-all / create / rename / delete orgs) ‚Äî [[4.4]].
- User-side membership management (from the Users page) ‚Äî [[4.3]].
- Inviting non-existent users / email ‚Äî users self-register ([[4.3]] decision); SSO ‚Äî [[4.2]].

## Acceptance criteria
- [ ] An org owner can list their org's members with roles from the org page ‚Äî proven by a test
- [ ] An org owner can add / remove a member and set a role, scoped to their org ‚Äî proven by an Api test
- [ ] A non-owner, non-admin member cannot manage members (403 / no UI) ‚Äî proven by a test

## Plan (thin slices)
1. [ ] Backend: owner-or-admin member endpoints on `/api/organizations/{id}/members`
   (GET list; POST add-by-email; PUT set-role; DELETE remove). New
   `IUserDirectory.GetUserMembershipsAsync`; `OrgAccess.CanManageOrgMembersAsync` (Owner-or-admin);
   `Role` added to the `/api/organizations` list payload; last-owner guard (400) on demote/remove.
   Tests (`OrganizationMembersEndpointTests`). -> <sha>
2. [ ] Web: `/organizations/:id` org detail page (member list + per-row role dropdown + remove,
   add-by-email form), owner-or-admin gated UI; topbar **Manage** link beside the org switcher.
   `Organization.role`, `OrganizationsApiService` member methods. Tests (`org-detail.spec.ts`). -> <sha>

## Decisions
- 2026-07-17 ‚Äî Created as the re-home of 4.4's owner-facing member-management scope (user split it
  out; 4.4 stays global-admin-only).
- 2026-07-17 ‚Äî Plan approved. UI: a **new `/organizations/:id` detail page** (no org detail page
  existed ‚Äî orgs were only the topbar switcher), reached via a **Manage** link beside the switcher,
  shown when the caller is Owner-or-admin. **Add-by-email**, not a user dropdown: an owner isn't a
  global admin and can't enumerate the admin-gated user directory ([[4.3]]); the server resolves the
  email (users self-register ‚Äî [[4.3]]). **Last-owner guard**: demoting/removing an org's final Owner
  via these endpoints ‚Üí 400; the escape hatch is a global admin acting through [[4.4]]'s admin
  endpoints (no guard there). Client UI-gating uses a new `Role` field on the `/api/organizations`
  list payload; the server enforces owner-or-admin authoritatively regardless.
