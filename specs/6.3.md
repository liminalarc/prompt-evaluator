## Problem
The high-effort code review of 6.1/6.2 (2026-07-19) fixed the confirmed correctness bugs inline
(OpenAI cache double-charge, echoed-vs-requested model id, inclusive `to`-date, Anthropic structured
refusal crash, budget scope-value validation, the CSV formula-injection security finding). Two
residual findings were left as deliberate follow-ups rather than rushed into the review pass:

1. **Phantom usage-less ledger rows (PLAUSIBLE).** `EvalRunnerClient.RecordAsync` writes a `Success`
   `AiUsageRecord` even when a 200 response carries no usage block (`usage == null`) — today this only
   happens on the **stub path** (`EVAL_RUNNER_STUB`, e2e-only) and any provider/response that omits
   usage. Such rows have 0 tokens / 0 cost and inflate `CallCount` / `SuccessRate` and drag
   `AvgTokensPerCall` toward zero. In production every real call returns usage, so the practical
   blast radius is small, but the ledger should not record a "call" that made no model call.

2. **Full-ledger in-memory aggregation at scale (CONFIRMED, documented tradeoff).**
   `AiUsageQueryService.{Summary,Breakdown,TimeSeries}Async` each `ToListAsync()` the whole filtered
   set and aggregate in C# (the class docstring already scopes this to "the ledger sizes this admin
   tool targets"), and `BudgetStatusHandler` calls `SummaryAsync` once per budget. Sum/Count/GroupBy
   are SQL-translatable; only the latency **percentiles** genuinely need row-level data (and Npgsql
   won't translate `percentile_cont` without a raw fragment). On a large ledger the dashboard +
   per-budget status become slow / memory-heavy.

## Value
As a workspace admin on a busy workspace, I want spend metrics that (a) count only real model calls
and (b) stay fast as the ledger grows, so the dashboard stays trustworthy and responsive.

## Scope
**In:**
- Decide + implement how a usage-less 200 is ledgered: skip the record, or record it as a distinct
  non-billable marker so it doesn't pollute call counts / averages. Keep failure rows (they signal
  waste).
- Push the additive aggregations (cost/token sums, counts, success rate) into SQL; compute percentiles
  either via a raw `percentile_cont` fragment or a documented in-SQL approximation. Reduce
  `BudgetStatusHandler` from N full materializations to grouped/aggregated queries.

**Out:**
- Any change to the captured DTOs' shape beyond what the above requires.
- Re-opening the pricing-source model (owned by 6.1/6.2).

## Acceptance criteria
- [ ] A usage-less 200 no longer produces a billable `Success` row that inflates call count / averages
      — unit test on `EvalRunnerClient`
- [ ] Summary / breakdown / time-series aggregate in SQL (no full-set materialization for the additive
      metrics); percentiles remain correct — test against a seeded dataset
- [ ] `BudgetStatusHandler` no longer materializes the whole month per budget — test/inspection
- [ ] Backend tests green
