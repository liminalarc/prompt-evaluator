## Problem
Piece by piece the app works, but it **feels disjointed / disconnected** in places (user, 2026-07-19) —
navigation flow, where things live, and consistency *between* screens don't yet read as one coherent product.
This is distinct from the isolated nits in [[2.18]] (a banner, a label, a dropdown): it's a **cohesion**
problem across the whole surface — the seams between Dashboard, Prompts, workspace, Dataset, Analytics, Admin,
and the run/review flows. The app-shell + design-system foundation ([[2.4]]) and the dogfood-driven rounds
([[2.5]]/[[2.8]]) got the primitives right; this is the pass that makes the *whole* hang together.

## Value
As an operator I want the app to feel like one connected product — predictable navigation, consistent
placement/patterns across screens, and clear paths between related things (a prompt ↔ its datasets ↔ its runs
↔ its analytics) — so I'm never disoriented about where I am or how to get to the related thing.

## Scope
**Two standing lenses (user, applied to every screen):** (1) UI/UX judgment for the **section itself and the
overall application surface** together — not just the widget in isolation; (2) **containerization / IA** — is
this thing in the right place, or does it belong on a different screen / behind a reveal / in its own view?

**Approach (not yet a fixed feature list — this starts as a walkthrough):**
1. **Drive the app screen by screen** (user + agent), or score it with `/flow:validate` against the design
   system + task-flow, and **catalog** what actually reads as disjointed: navigation dead-ends, inconsistent
   placement/labels, broken cross-links between related entities, redundant/again-from-scratch flows, visual
   inconsistency between screens.
2. Turn the catalog into **concrete, prioritized fixes** — sharpen this spec's AC/plan from findings, or spin
   sub-specs if a theme is big enough.

**In (candidate themes — confirm during the walkthrough):** cross-entity navigation (prompt ↔ datasets ↔ runs
↔ analytics), consistent information architecture + placement, screen-to-screen visual/pattern consistency,
breadcrumbs/back-paths, empty/loading/error coherence.
**Out:** the three concrete nits already homed in [[2.18]]; new *features* (this is cohesion of what exists);
brand-token adoption ([[2.6]]).

## Walkthrough catalog (findings as we drive each screen)

### Prompt workspace — Version history (2026-07-19)
- **W1 — "Source app" column is always "—" (unwired + redundant).** `PromptVersion.SourceApp` is never
  populated by any UI path (add-version and both import paths pass `null`), and it duplicates the **org**
  (which *is* the app). → **drop the column** (or wire it from the org / a form field; recommend drop).
- **W2 — Raw ISO timestamps everywhere.** "Created" shows `2026-07-19T00:52:05.825008+00:00` — microsecond
  precision + `+00:00`, unreadable at a glance. **Format human-friendly** (+ relative "2h ago"). Recurs
  app-wide (runs, fixtures, eval-runs) → a cross-cutting date-format primitive.
- **W3 — Add-version Content editor is too small; wants a code editor, not WYSIWYG.** 8 rows for a ~55-line
  system prompt. Enlarge + resizable, and ideally a **syntax-highlighting source editor** (CodeMirror). NOT
  WYSIWYG: prompts need **exact source fidelity** (the model reads raw text; round-debrief has a literal
  `[DRILLS_JSON]`+JSON block a WYSIWYG would mangle). Upgrades 2.10's Edit side. → candidate own spec.
- **W4 — Immutable content `<pre>` has no max-height.** A long version's content dominates the whole page
  (endless scroll). Cap height + internal scroll, or collapse.
- **W5 — No version-status signal (best / current-in-source / regressed).** 7 versions, nothing marks v7 as
  the winner or what's deployed; the "which version won" story lives only in Analytics, disconnected here. →
  this is exactly **[[1.16]]** (version-status badges) — connect it on the version row.
- **W6 — No cross-navigation from a version.** From a version row you can view content + edit label, but
  can't jump to its **runs / compare / analytics** — the version ↔ runs ↔ analytics links are missing (the
  core "disconnected" feeling).

### Prompt workspace — Compare versions (content diff) (2026-07-19)
- **W7 — Name collision: two "Compare versions."** The workspace card (this one — diffs the version **content**
  via `VersionDiff`) and the Analytics card (diffs **scores** + now rationale) share the *exact* heading. Two
  different meanings, same name, two separate From/To pickers. → distinguish ("Compare content" vs "Compare
  scores"), or **unify into one Compare surface** with tabs: **Content · Scores · Rationale** (pick From/To
  once). Strong cohesion win — directly the "disconnected" feeling.
- **W8 — Diff shows unchanged regions.** It renders both full contents; a small change (v6→v7 = a few rules
  low in the prompt) means scrolling the *entire identical* prose to find the delta. → **collapse unchanged
  runs** ("N unchanged lines", jump-to-change), like a GitHub diff.
- **W9 — Containerization: doesn't belong inline-and-open on the workspace.** Heavy + occasional; at minimum
  collapse behind a reveal (like the other cards), better move into the unified Compare view/route/drawer.
- **W10 — Workspace is overloaded (IA).** Version history + content-compare + run + datasets + analytics all
  stacked and mostly expanded = a very long page. Consider the workspace as a leaner **hub** that links out to
  focused views (Compare, Analytics, Runs), rather than everything inline. (Overall-surface lens.)

### Prompt workspace — Run a version (2026-07-19)
- **W11 — The app's *primary action* is buried.** Running an eval is the whole point, but it's a collapsed
  reveal card mid-page among many. As the core CTA it should be **prominent + always available** (workspace
  header / hub primary button). It's a *quick* action (2 dropdowns → go), so **not a drawer** — a compact
  inline form or small modal. (Boundary for D1: drawers are for heavy/focused/*browse* surfaces; a transient
  quick action is elevated + compact, not drawered.)
- **W12 — Two run surfaces.** Running also lives on the **Dataset detail** page ("Run evaluation", version-
  only). Same split-brain as the two "Compare"s (W7): pick **one canonical Run control** invoked from either
  context (prompt or dataset), pre-filled by where you launched it.
- **W13 — "no page hop" subtitle is now self-contradictory.** Copy says "no page hop" but a run **navigates to
  the results**. It referred to not hopping to the dataset page to run; re-word or drop.
- **W14 — This flow changes with async runs ([[2.17]]).** Today: run → immediately navigate to the eval-run
  page. With async, "run" becomes a **job you watch** (kick off, poll, progress) — the action *and* the
  results screen change together. The user flagged the results screen will be reviewed + likely change; design
  **Run + [[2.17]] + the results view** as one coupled flow, not three.

## Candidate patterns / cross-cutting directions (emerging from the walk)
- **D1 — Right-side drawer for heavy, focused, occasional surfaces (user idea).** Slide-over panel from the
  right that keeps the page context behind it — a good home for **Compare** (content/score/rationale), an
  expanded **eval-run** view, and possibly the large **add-version editor**. Fixes the W9/W10 overload:
  the workspace stays a lean hub; heavy sub-tasks open in a drawer instead of stacking inline. **Success
  conditions (or it *adds* to the disjointedness):**
  - **One shared `Drawer` primitive** — consistent side, width, header, close/Esc, focus-trap, scroll — used
    everywhere; never one-off drawers.
  - **Right surfaces only** — heavy/focused/occasional (compare, run detail, big editor). *Not* primary
    always-on data (the version table), and *not* a replacement for the existing inline reveal/expand pattern
    (2.4/2.8) for small add/edit forms. Don't over-drawer.
  - **Linkable state** — for shareable/deep surfaces (a specific comparison), **sync drawer state to the URL**
    (query param or child route) so it's back-button-friendly and shareable; otherwise a drawer loses the one
    thing a route gives you. (Compare specifically may be better as a **route** `/prompts/:id/compare?from=&to=`
    that *renders* in a drawer-style panel — linkable + focused.)
  - **Responsive** — full-screen/bottom-sheet on narrow widths.
  - Pairs naturally with **W7 unify**: one **Compare drawer** (From→To once) with **Content · Scores ·
    Rationale** tabs is the synthesis of the drawer idea + the unify idea — likely a flagship cohesion fix.
