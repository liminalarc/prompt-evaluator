## Problem
Piece by piece the app works, but it **feels disjointed / disconnected** in places (user, 2026-07-19) —
navigation flow, where things live, and consistency *between* screens don't yet read as one coherent product.
This is distinct from the isolated nits in [[2.18]] (a banner, a label, a dropdown): it's a **cohesion**
problem across the whole surface — the seams between Dashboard, Prompts, workspace, Dataset, Analytics, Admin,
and the run/review flows. The app-shell + design-system foundation ([[2.4]]) and the dogfood-driven rounds
([[2.5]]/[[2.8]]) got the primitives right; this is the pass that makes the *whole* hang together.

## Value
As an operator I want the app to feel like one connected product — predictable navigation, consistent
placement/patterns across screens, and clear paths between related things (a prompt ↔ its datasets ↔ its runs
↔ its analytics) — so I'm never disoriented about where I am or how to get to the related thing.

## Scope
**Two standing lenses (user, applied to every screen):** (1) UI/UX judgment for the **section itself and the
overall application surface** together — not just the widget in isolation; (2) **containerization / IA** — is
this thing in the right place, or does it belong on a different screen / behind a reveal / in its own view?

**Approach (not yet a fixed feature list — this starts as a walkthrough):**
1. **Drive the app screen by screen** (user + agent), or score it with `/flow:validate` against the design
   system + task-flow, and **catalog** what actually reads as disjointed: navigation dead-ends, inconsistent
   placement/labels, broken cross-links between related entities, redundant/again-from-scratch flows, visual
   inconsistency between screens.
2. Turn the catalog into **concrete, prioritized fixes** — sharpen this spec's AC/plan from findings, or spin
   sub-specs if a theme is big enough.

**In (candidate themes — confirm during the walkthrough):** cross-entity navigation (prompt ↔ datasets ↔ runs
↔ analytics), consistent information architecture + placement, screen-to-screen visual/pattern consistency,
breadcrumbs/back-paths, empty/loading/error coherence.
**Out:** the three concrete nits already homed in [[2.18]]; new *features* (this is cohesion of what exists);
brand-token adoption ([[2.6]]).

## Walkthrough catalog (findings as we drive each screen)

### Prompt workspace — Version history (2026-07-19)
- **W1 — "Source app" column is always "—" (unwired + redundant).** `PromptVersion.SourceApp` is never
  populated by any UI path (add-version and both import paths pass `null`), and it duplicates the **org**
  (which *is* the app). → **drop the column** (or wire it from the org / a form field; recommend drop).
- **W2 — Raw ISO timestamps everywhere.** "Created" shows `2026-07-19T00:52:05.825008+00:00` — microsecond
  precision + `+00:00`, unreadable at a glance. **Format human-friendly** (+ relative "2h ago"). Recurs
  app-wide (runs, fixtures, eval-runs) → a cross-cutting date-format primitive.
- **W3 — Add-version Content editor is too small; wants a code editor, not WYSIWYG.** 8 rows for a ~55-line
  system prompt. Enlarge + resizable, and ideally a **syntax-highlighting source editor** (CodeMirror). NOT
  WYSIWYG: prompts need **exact source fidelity** (the model reads raw text; round-debrief has a literal
  `[DRILLS_JSON]`+JSON block a WYSIWYG would mangle). Upgrades 2.10's Edit side. → candidate own spec.
- **W4 — Immutable content `<pre>` has no max-height.** A long version's content dominates the whole page
  (endless scroll). Cap height + internal scroll, or collapse.
- **W5 — No version-status signal (best / current-in-source / regressed).** 7 versions, nothing marks v7 as
  the winner or what's deployed; the "which version won" story lives only in Analytics, disconnected here. →
  this is exactly **[[1.16]]** (version-status badges) — connect it on the version row.
- **W6 — No cross-navigation from a version.** From a version row you can view content + edit label, but
  can't jump to its **runs / compare / analytics** — the version ↔ runs ↔ analytics links are missing (the
  core "disconnected" feeling).

### Prompt workspace — Compare versions (content diff) (2026-07-19)
- **W7 — Name collision: two "Compare versions."** The workspace card (this one — diffs the version **content**
  via `VersionDiff`) and the Analytics card (diffs **scores** + now rationale) share the *exact* heading. Two
  different meanings, same name, two separate From/To pickers. → distinguish ("Compare content" vs "Compare
  scores"), or **unify into one Compare surface** with tabs: **Content · Scores · Rationale** (pick From/To
  once). Strong cohesion win — directly the "disconnected" feeling.
- **W8 — Diff shows unchanged regions.** It renders both full contents; a small change (v6→v7 = a few rules
  low in the prompt) means scrolling the *entire identical* prose to find the delta. → **collapse unchanged
  runs** ("N unchanged lines", jump-to-change), like a GitHub diff.
- **W9 — Containerization: doesn't belong inline-and-open on the workspace.** Heavy + occasional; at minimum
  collapse behind a reveal (like the other cards), better move into the unified Compare view/route/drawer.
- **W10 — Workspace is overloaded (IA).** Version history + content-compare + run + datasets + analytics all
  stacked and mostly expanded = a very long page. Consider the workspace as a leaner **hub** that links out to
  focused views (Compare, Analytics, Runs), rather than everything inline. (Overall-surface lens.)

### Prompt workspace — Run a version (2026-07-19)
- **W11 — The app's *primary action* is buried.** Running an eval is the whole point, but it's a collapsed
  reveal card mid-page among many. As the core CTA it should be **prominent + always available** (workspace
  header / hub primary button). It's a *quick* action (2 dropdowns → go), so **not a drawer** — a compact
  inline form or small modal. (Boundary for D1: drawers are for heavy/focused/*browse* surfaces; a transient
  quick action is elevated + compact, not drawered.)
- **W12 — Two run surfaces.** Running also lives on the **Dataset detail** page ("Run evaluation", version-
  only). Same split-brain as the two "Compare"s (W7): pick **one canonical Run control** invoked from either
  context (prompt or dataset), pre-filled by where you launched it.
- **W13 — "no page hop" subtitle is now self-contradictory.** Copy says "no page hop" but a run **navigates to
  the results**. It referred to not hopping to the dataset page to run; re-word or drop.
- **W14 — This flow changes with async runs ([[2.17]]).** Today: run → immediately navigate to the eval-run
  page. With async, "run" becomes a **job you watch** (kick off, poll, progress) — the action *and* the
  results screen change together. The user flagged the results screen will be reviewed + likely change; design
  **Run + [[2.17]] + the results view** as one coupled flow, not three.

### Prompt workspace — Datasets + Analytics cards (2026-07-19)
- **W15 — Datasets card → orphaned destination.** The card lists the prompt's datasets + inline create;
  clicking a dataset routes to `/datasets/:id`, which has **no nav presence** (Datasets was demoted in 2.4) —
  reachable *only* from here, an orphan with no coherent "where am I / how back". Fine as a related-list; the
  *destination* needs a real home.
- **W16 — Embedded Analytics is a partial duplicate.** This card is a **subset** (trend chart only) of the
  dedicated `/analytics` page — which now also has regressions, compare, **variance**, and **rationale-diff**.
  So there are two per-prompt analytics surfaces and one is a stale partial. "Looks like embedded functionality
  from another area" (user) = exactly right. Pick **one canonical** per-prompt analytics home.
- **General:** these cards blur *related-list* vs *embedded-feature* vs *quick-action* — the user "has no idea
  what to do with these," which *is* the finding. The kitchen-sink workspace (W10) is the root cause.

### Dataset detail — Fixtures (top) (2026-07-19)
- **W17 — "Fixture" is jargon that fights the mental model.** Users read the rows as **test cases** (correct —
  a fixture *is* one test input), but "fixture" doesn't obviously map, and the scorer lives at the **dataset**
  level (a separate card), so the term obscures the real structure. The dataset description even calls them
  "profiles". → strongly consider renaming **Fixtures → "Test cases"** (or "Scenarios") across the UI, or at
  minimum define it inline. (Domain term is `Fixture` / spec 1.2 — a rename is UI-label-level, keep the
  aggregate name; big enough to be its own decision.)
- **W18 — Fixtures ⟂ Scorers relationship is invisible.** The dataset page shows Fixtures here and Scorers as a
  disconnected card lower down; nothing states **dataset = test cases × scorers → a score per case per scorer**.
  That missing model *is* the user's confusion. Surface the structure (a one-line model, or an IA that shows
  cases + scorers + what a run does together).
- **W19 — INPUT column is raw truncated JSON.** Noisy (`## Round Scorecard { "course": …`); the **label** is the
  real identifier. De-emphasize/soften the raw-input preview (or drop it to the expanded row).
- Minor: **SEED** column is all "—" here (only set for synthetic-from-captured) — another mostly-empty column,
  same smell as W1 source-app.

### Dataset detail — Scorers (2026-07-19)
- **W20 — CONFIG column dumps the entire rubric.** The LlmJudge row shows the full ~15-line rubric inline in a
  table cell → one giant row next to two tiny Regex rows; a wall of text where a table wants a scannable
  summary. → show a **summary** ("LlmJudge · Opus 4.8 · rubric") and reveal the full text on
  expand/click. Same class as W4/W8/W19 (see **D3**).
- **W21 — Rubric editor is too small** (W3 again). The edit-scorer Rubric uses the 2.10 markdown editor (good —
  Edit/Preview is there) but only a few rows for a 15-line rubric. Enlarge/resize; a code-editor upgrade helps
  here too.
- **W22 — Editing a scorer expands inline and reflows the table.** Clicking a scorer inserts a tall form
  (select + big rubric editor + judge model + Save/Cancel/Remove) between rows, shoving the others down —
  jarring. Editing a rubric is a **focused task with a big editor** → strong **D1 drawer** candidate; keeps the
  table stable.
- **Good, worth reusing:** the Scorers subtitle — *"Configured once per dataset; every run scores with this
  set"* — is exactly the kind of model-statement the **Fixtures** card lacks (W18). The clarity exists here;
  connect the two cards and give Fixtures its equivalent.

### Dataset detail — Run evaluation + Runs table, and the Eval-run page (2026-07-19)
- **W23 — Runs table shows no SCORE.** Columns are Run/Version/Model/Scorers/Fixtures — but **not how the run
  did**. The result is the whole point; you must click into each run to learn anything. → add an **aggregate
  score (+ pass rate)** column so runs are scannable at a glance.
- **W24 — A run is identified by a raw ISO timestamp** (W2 again) — and that's the clickable "name"
  (`2026-07-19T15:52:48.695759+00:00`). Give runs a **readable identity**: "v7 · Jul 19, 3:52 PM" or a run #.
- **W25 — SCORERS cell renders "LlmJudgeRegex"** (kinds concatenated, no separator) — a formatting bug; use
  chips or a comma. MODEL column wraps to 3 lines (width).
- **W26 — Runs table is unbounded + ungrouped.** 11+ rows, no limit/pagination, no per-version grouping —
  ties to **[[2.14]]** variance: repeated runs of a version should group/aggregate here, not list as noise.
- **W27 — Eval-run page lacks run context in its header.** "Eval run · 4 fixtures · 12 scores" — but not the
  **version/model** or the **run's mean score**. → "Eval run · v7 · Sonnet 4.6 · mean 0.84".
- **W12 confirmed** — this "Run evaluation" card is the *second* run surface (workspace has "Run a version").
- **✅ Bank this as the target quality:** the **Eval-run detail page** is the app done *right* — a real
  **breadcrumb** (Dashboard / prompt / dataset / Eval run), **cross-links** (View dataset / View prompt), and
  **summarize-then-expand** per-fixture rows (D3). It's how the orphaned pages (W15) *should* behave. Use it as
  the reference for the redesign.

### Eval-run — expanded fixture (2026-07-19)
- **✅ This is the quality bar.** Side-by-side **Fixture input** / **Model output**, a **metrics** row
  (latency / tokens / cost), and a clean **per-scorer table** (scorer · judge model · value · passed · detail)
  with the **rationale** in Detail. Information-dense, legible, and it's the *right* place for the full rationale
  (a detail view, per D3). Reuse this layout as the **drawer/detail template**.
- **✅ Cohesion win:** the rationale here is the same data as the 2.14 Compare **rationale-diff** — consistent,
  one source of truth for "why".
- **W28 — Model output is a small fixed-height scroll box.** It's the primary artifact (what the model
  produced); in this focused/expanded view D3 permits showing more — give it more height / let it expand.
- **W27 confirmed at fixture level** — scores are per-fixture; still no **run-level mean** anywhere on the page.
- Minor: **COST "—"** (null metric) — same empty-field question as W1/W19 (SEED/source-app). Decide a
  consistent null-metric treatment (hide vs "—" vs "n/a").
- **Cohesion idea:** from a fixture here, a link to **that fixture's score across versions** (its per-fixture
  variance, [[2.14]]) would connect the run view to the trend/variance view — optional, low priority.

### Global Analytics page (2026-07-19)
- **W29 — Duplicate/stale scorer identities, all labelled just "Regex"/"LlmJudge".** Analytics groups by scorer
  **identity** (correct — you can't compare across different scorer configs; the same-scorer-config rule,
  [[1.16]]). But editing a scorer's config mid-walk **mints a new identity**, so the dataset accumulates
  several — and trend/stability/compare then render **multiple indistinguishable tables** (the "0.000, v1 only"
  Regex block is a *stale* identity from an old/removed config). → (a) **label each scorer block by its
  config/identity**, not just kind; (b) hide or mark **stale** identities (not on the dataset's current scorer
  set). Affects trend, compare, **and the new 2.14 variance view**.
- **W30 — Variance is noise for deterministic scorers (self-critique of 2.14).** Regex/deterministic scorers
  are always `1.000 ± 0.000` — run-to-run spread is meaningless for them; variance is only signal for
  **stochastic** scorers (LlmJudge). The Score stability card over-shows: it should **focus on stochastic
  scorers** (hide/collapse the always-zero-spread deterministic ones). Good live catch.
- **W31 — Score trend silently omits versions with no run** (v4 is absent). Minor, but "where's v4?" is
  confusing; consider showing a gap/marker.
- **W32 — Page is very long** (trend + N stability tables + regressions + compare stacked). Same length/stacking
  issue (W10/D3); the stability section (a full table per scorer identity) balloons it — W29+W30 fixes shrink it.
- **Confirms:** this page **is** the fuller version of the embedded workspace Analytics (**W16**) → becomes the
  prompt's **Analytics tab** under D2 (or cross-prompt browse). And "Compare versions" here = the **score**
  compare vs the workspace **content** compare — **W7 collision** again. The **2.14 rationale-diff ("Why")** is
  present and working here. ✅

### Header / topbar nav (2026-07-19)
- **W37 — Nav has "Analytics" (global) but not "Datasets" — inconsistent, and it shapes the IA.** Both are
  per-prompt entities, yet one is top-nav and the other is orphaned (W15). Global **Analytics** as a top item
  reinforces the split (global vs embedded, W16) — it's really per-prompt (you must pick a prompt+dataset to use
  it). Under **D2** the nav simplifies to **Dashboard · Prompts · Admin**, with Analytics/Datasets as **prompt
  tabs** (and any cross-prompt roll-up living on the Dashboard). The nav is the lever that makes the tabbed-hub
  model legible. Otherwise the header is clean ✅ (org switcher, DEV badge, theme toggle, account/logout all
  good). Minor: the greyed **"Manage"** link's purpose/enablement is unclear.

### Dashboard (2026-07-19)
- **W33 — Misleading "1.00" prompt score.** Both prompt cards show `1.00`, but round-debrief's meaningful
  (LlmJudge) score is ~0.84 — the badge is **inflated by the always-pass Regex scorers** (same root as W30:
  deterministic scorers distort an aggregate). Show the **meaningful** score (LlmJudge / per-scorer) and/or a
  **trend ↑↓ + best/deployed version**, not a mean dominated by pass/fail scorers.
- **W34 — Dashboard should center on "what needs attention."** Recent-runs has no score (W23 again); the
  dashboard's real value is **backport-pending ([[1.16]]) + open regressions + failing prompts** first. It's
  close; sharpen around *attention → status → activity* (see recommendation below).
- **✅ Dates are formatted here** ("Jul 19, 2026, 11:52:48 AM") — proving **W2 is *inconsistent***, not
  universally raw (dataset Runs table is raw ISO). Fix = make this good format the shared default everywhere.
- **Minimal-dashboard recommendation (user wants minimal, no wall of text):** three scannable sections —
  1) **Needs attention** — backport-pending + open regressions + failing prompts (calm "all good" when empty);
  2) **Prompts** — compact per-prompt status: *meaningful* score + trend + best/deployed version;
  3) **Recent activity** — recent runs *with score*. No charts, no walls; each row links into the detail.

## Candidate patterns / cross-cutting directions (emerging from the walk)
- **D3 — "Summarize in lists, reveal in focus" for long content (recurring theme).** The app repeatedly dumps
  long text **inline**: version content `<pre>` (W4), the content diff (W8), fixture INPUT JSON (W19), the
  rubric CONFIG cell (W20). Adopt one rule everywhere — **lists/tables show a scannable summary; the full long
  content opens in a focused surface** (expand row, or a D1 drawer for the heavy/editable ones). This single
  discipline fixes a big chunk of the "everything is a wall of text / endless scroll" feeling.
- **D2 — Prompt workspace as a hub with tabs (the backbone IA fix).** `/prompts/:id` = persistent prompt
  header + **tabs: Versions · Datasets · Analytics · Runs** (Compare opens in the D1 drawer). Each area gets
  its own space and becomes **canonical in-context**, which resolves most of the prompt-page findings at once:
  kills the kitchen-sink stacking (W10), the embedded partial-analytics duplicate (W16), the orphaned dataset
  destination (W15); gives the primary **Run** action a clear home (W11); and makes the split-brain Compare
  (W7) a single tab/drawer. The **global** `/analytics` + `/datasets` become **cross-prompt browse** views
  (their real distinct job), while per-prompt work lives in the tabs. This is likely the structural change the
  whole "disjointed" feeling points to; almost everything else hangs off it. (Pattern: GitHub-repo-style tabs.)

- **D1 — Right-side drawer for heavy, focused, occasional surfaces (user idea).** Slide-over panel from the
  right that keeps the page context behind it — a good home for **Compare** (content/score/rationale), an
  expanded **eval-run** view, and possibly the large **add-version editor**. Fixes the W9/W10 overload:
  the workspace stays a lean hub; heavy sub-tasks open in a drawer instead of stacking inline. **Success
  conditions (or it *adds* to the disjointedness):**
  - **One shared `Drawer` primitive** — consistent side, width, header, close/Esc, focus-trap, scroll — used
    everywhere; never one-off drawers.
  - **Right surfaces only** — heavy/focused/occasional (compare, run detail, big editor). *Not* primary
    always-on data (the version table), and *not* a replacement for the existing inline reveal/expand pattern
    (2.4/2.8) for small add/edit forms. Don't over-drawer.
  - **Linkable state** — for shareable/deep surfaces (a specific comparison), **sync drawer state to the URL**
    (query param or child route) so it's back-button-friendly and shareable; otherwise a drawer loses the one
    thing a route gives you. (Compare specifically may be better as a **route** `/prompts/:id/compare?from=&to=`
    that *renders* in a drawer-style panel — linkable + focused.)
  - **Responsive** — full-screen/bottom-sheet on narrow widths.
  - Pairs naturally with **W7 unify**: one **Compare drawer** (From→To once) with **Content · Scores ·
    Rationale** tabs is the synthesis of the drawer idea + the unify idea — likely a flagship cohesion fix.
