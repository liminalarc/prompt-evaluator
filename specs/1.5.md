## Problem
The eval loop is Claude-only today — Claude is the judge and the synthetic-fixture
generator, and a prompt's target model is assumed to be Claude. But the apps we evaluate
already run non-Claude models (e.g. a Stormboard SLM on Modal), so the *subject* model a
prompt runs against is heterogeneous, and we may later want to judge or generate with a
chosen provider. We need the eval-runner to execute and judge against multiple providers
behind the existing `IEvaluationRunner` seam — without changing Domain or Application.

## Value
As a prompt owner I want to evaluate prompts that run on non-Claude models (and optionally
choose the judge/generator provider) so that the harness covers every model our apps
actually use, not just Claude.

## Scope
**In:**
- A **provider abstraction inside the eval-runner** that routes a request to Anthropic /
  OpenAI / (Modal SLM / others) by model identifier — either a thin per-provider adapter
  layer or a library like LiteLLM (decided at slice 1).
- **Per-provider structured output**: judge verdicts and guided generation return structured
  results (tool-use / JSON schema) on every provider — never free-text parsing.
- **Subject execution across providers**: run a `PromptVersion`'s target model (1.1) on the
  right provider; or score captured outputs with no live call (capture-first, 1.2).
- **Config/secrets**: per-provider credentials via env (`ANTHROPIC_API_KEY` exists; add
  `OPENAI_API_KEY`, Modal creds, etc.), documented in `README.md` / `.env.example`.
- **Judge/generator provider** is optional and, per 1.3, part of Scorer identity — switching
  it starts a new score series.

**Out:** any Domain/Application change (the `IEvaluationRunner` seam + eval-runner absorb
this entirely). Prompt-engineering advisory (2.2). Making the generator/judge prompts
themselves registry-managed (a direction noted in 1.2/1.3, not built here).

## Acceptance criteria
- [ ] eval-runner executes a prompt against at least one non-Claude provider selected by model id — proven by an eval-runner test (provider clients mocked)
- [ ] Judge verdicts and guided generation return structured output per provider (no free-text parsing) — proven by eval-runner tests
- [ ] Per-provider credentials are env-configured and documented; missing creds fail clearly — proven by a config test + README
- [ ] No Domain/Application type references a provider; the only seam is `IEvaluationRunner` — proven by inspection
- [ ] Claude stays the default judge/generator; switching provider or model yields a distinct Scorer series — proven by a test (ties to 1.3)

## Plan (thin slices)

## Decisions
- 2026-07-12 — Multi-provider lives entirely in the eval-runner behind `IEvaluationRunner`;
  .NET and Domain stay provider-agnostic. Claude remains the default judge and generator;
  other providers are additive. Weigh LiteLLM vs thin per-provider adapters at slice 1.
- 2026-07-12 — Deliberately sequenced after 1.1 and 1.3: the registry (subject model on
  `PromptVersion`) and the harness (judge model as part of Scorer identity) define the
  contract this spec implements across providers. Consult the `claude-api` skill for Claude
  model ids; never hardcode from memory.
