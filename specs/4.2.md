## Problem
4.1 ships first-party authentication (cookie sessions over ASP.NET Core Identity, self-service
registration, password reset). It deliberately left **SSO / OAuth** out ("possible later
addition"). This spec is the home for that follow-on: let users sign in with an external identity
provider (e.g. Google / Microsoft / a generic OIDC provider) instead of — or in addition to — a
local password. Origin: re-homed from 4.1 at plan-time by user decision (2026-07-14). "Shipped"
means a user can authenticate through at least one external provider and land in the same
organization-scoped session 4.1 established.

## Value
As an operator I want users to sign in with their existing corporate/identity-provider accounts so
that we avoid managing separate passwords and onboarding is faster.

## Scope
**In:** (to refine when picked up)
- External sign-in via OIDC/OAuth (provider set TBD — likely Google + Microsoft + generic OIDC).
- Account linking: map an external identity to a 4.1 `AppUser` (link on first sign-in / to an
  existing account by verified email).
- Session issuance reuses 4.1's cookie + `ICurrentUser`/claims; org membership/authZ unchanged.
- SPA: "Sign in with …" affordance + callback route alongside 4.1's local login.
- **Transactional-email provider for auth (re-homed from 3.2).** 4.1 built the password-reset/forgot
  flow behind the `IEmailSender` seam (NoOp/log in dev, SMTP configurable by env) but wired no concrete
  hosted provider. 3.2 originally owned this (email needs a deployed environment to run in); at 3.2's
  AWS-deploy plan-time the user folded it here (2026-07-17) since the dev deploy doesn't need reset
  email and email pairs naturally with the identity/SSO work. Pick + wire a real SMTP/email provider
  (credentials via env/secrets) so reset (and any SSO-invite) email actually sends. See
  `specs/3.2.md` / `specs/archive/4.1.md`.

**Out:** SCIM / directory provisioning, per-provider role mapping beyond org membership (later).

## Acceptance criteria
- [ ] A user can sign in through an external provider and get a normal org-scoped session — proven by tests + a run
- [ ] External identities link to a single `AppUser` (no duplicate accounts by email) — proven by tests
