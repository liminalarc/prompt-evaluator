## Problem
LitmusAI already stamps a git-tag-derived version into the API assembly (`-p:Version=$APP_VERSION`)
and the eval-runner env, and exposes an aggregated `GET /version` (service, version, commit, and
dependency versions). But **the web UI surfaces nothing** — `web/package.json` `version` is an unused
`0.0.0` placeholder — and **no service carries an environment/channel signal**. A user looking at the
running app can't tell which build or which environment they're on. This becomes essential the moment
releases are promoted to hosted **upper environments** (staging/prod — spec 3.2): there's currently no
way to distinguish "am I on prod or a dev build?" from the UI.

Stormboard (`C:\Development\stormboard`, specs 142/163/164) solved exactly this and the user asked to
replicate it here. Its key lesson: **`ASPNETCORE_ENVIRONMENT` is not a reliable dev/prod discriminator**
(their host reported `Production` everywhere), so it added a **`channel`** signal — a `DEPLOY_CHANNEL`
env var baked at image build from a CI build-arg driven by git-ref — and surfaced it as a **footer build
chip** plus a **toolbar environment badge**. "Shipped" means the LitmusAI SPA shows the running version
and environment the same way, and the channel plumbing is in place so it works correctly across upper
environments when 3.2 introduces them.

## Value
As a user/operator I want to see the running version and environment right in the app so that I can tell
at a glance which build and which environment (local / dev / staging / prod) I'm looking at — the
difference between "this is production" and "this is a dev build" must be unmistakable once releases are
promoted to hosted environments.

## Scope
**In:**
- **Deploy-channel plumbing (the "upper environments" support).** Introduce a `DEPLOY_CHANNEL` signal
  baked into the image at build from a CI build-arg driven by git-ref — mirroring Stormboard: a `v*` tag
  push → the promoted channel (default `prod`), a `main` push → `dev`, a local/non-CI build → `local`.
  Channel is an **open string** so `staging` (and any further upper environments) drop in when 3.2 wires
  real deploy targets — this spec establishes the *mechanism*, not the hosted environments themselves.
- **Extend the API version payload** with `channel`, `environment` (`IHostEnvironment.EnvironmentName` —
  included but explicitly **not** the dev/prod discriminator, per Stormboard's lesson), and `buildTime`,
  alongside the existing version/commit/dependency fields. The endpoint must be reachable by the SPA
  through the `/api` proxy the SPA already uses (see Open questions for endpoint shape).
- **Web version service** — a root-provided signal service that fetches the version once at startup
  (app initializer). A failed/absent fetch leaves the signal null and the indicators simply don't render
  (never an error, never a 500 surfaced to the user).
- **Footer build chip** — a subtle, fixed, monospace pill (screen corner) whose label is channel-keyed:
  `prod` → `v<version> · <sha7>`; `dev` → `dev · <sha7>`; `local` → `local`. Full detail in a tooltip
  (`<channel> · v<version> · <commit> · built <buildTime>`). A dev build shows channel+sha, not a semver,
  so it never looks "behind" a freshly-tagged prod.
- **Toolbar env badge** — `LOCAL` / `DEV` / `STAGING` (uppercased channel) next to the brand in the navy
  hero topbar; **no badge in prod** (a prod badge would be noise). Channel wins over `environment`.
- **CI + Dockerfiles** — `ci.yml` computes `APP_CHANNEL` from git-ref alongside the existing `APP_VERSION`
  and passes it as a build-arg; the API `Dockerfile` maps `APP_CHANNEL` → `DEPLOY_CHANNEL` env. Since
  there is no registry/deploy step yet, this is build-arg + ENV plumbing validated by `compose-smoke`.
- **Docs** — update CLAUDE.md's *Releasing* / version section to document the channel signal and the web
  version display; keep the "assert the version's **shape**, never a literal" rule so a release never
  breaks a test.

**Out:**
- The actual hosted environments / deploy targets / image registry / publish pipeline — that is **3.2
  (Production Deployment)**; this spec only makes the version display channel-ready and defaults `local`/
  `dev`/tag→`prod`. 3.2 sets `APP_CHANNEL` per real target (incl. `staging`).
- Changing the version-derivation mechanism (`git describe` → `APP_VERSION` → assembly/env) — unchanged.
- Auth/permission changes — the version endpoint stays anonymous (it already is).
- eval-runner UI display — the SPA reads the API's channel (see Open questions re: eval-runner channel).

## Acceptance criteria
- [ ] The API version endpoint returns `channel`, `environment`, and `buildTime` alongside the existing
  version/commit/dependency fields — proven by an Api.Tests **shape** assertion (fields present; never a
  literal version/channel value).
- [ ] The SPA shows the running version in a **footer build chip** and an **environment badge** in the
  topbar, both channel-keyed (no badge when `channel = prod`), fed by one app-initializer-loaded signal
  service; a failed/absent version fetch renders nothing and surfaces no error — proven by a web unit
  test including the **channel-wins-over-`environment`** case, plus inspection.
- [ ] `DEPLOY_CHANNEL` flows git-ref → CI build-arg → Docker ENV → API payload: `v*` tag → `prod`,
  `main` → `dev`, local/non-CI build → `local` — proven by inspection + `compose-smoke` asserting the
  channel field is present (shape, not a literal).
- [ ] CLAUDE.md's *Releasing* section documents the channel signal + web version display; the shape-only
  (never-literal) version assertion rule is preserved.

## Open questions
- **Endpoint shape for the SPA.** Extend the existing aggregated `GET /version` (and ensure the SPA
  reaches it via the `/api` proxy) vs. add a flat `GET /api/version` mirroring Stormboard? Lean: **extend
  the existing `/version`** (single source of truth; avoids a divergent second endpoint) and expose it
  through the web proxy. Decide at slice 1.
- **Channel vocabulary for upper environments.** `dev` / `staging` / `prod` (+ `local`)? The final list
  belongs with 3.2's deploy targets; this spec keeps channel an **open string** and only defaults
  `local`/`dev` and tag→`prod`.
- **eval-runner channel.** Bake `APP_CHANNEL` into eval-runner too (payload consistency) or API-only (the
  UI only needs the API's channel)? Lean: **API-only**, optional eval-runner. Decide at slice 2.
