## Problem
6.1 introduces a versioned, config-backed .NET pricing table (input / output / cache-write / cache-read
+ rate version) made **authoritative for the ledger**. It deliberately left two other pricing sources
untouched (6.1 plan-time decision, "ledger table only"): the eval-runner `_PRICING` table
(`eval-runner/app/execution.py` — execution cost → `FixtureRun`) and the Model Catalog's display-only
`InputPricePerMTokUsd` / `OutputPricePerMTokUsd` (`src/Domain/ModelCatalogEntry.cs`, spec 1.13). So the
same model's price can differ across three places, and the catalog UI can show a rate that disagrees
with what the ledger charges. Shipped = the Model Catalog's display price **derives from** the
authoritative 6.1 pricing table, so catalog and ledger show identical rates from one source.

## Value
As a workspace admin, I want the price shown in the Model Catalog to match what the AI-usage ledger
actually charges, so I'm not reconciling three different numbers for the same model.

## Scope
**In:**
- Make the Model Catalog's displayed input/output price read from the 6.1 authoritative pricing table
  (single source), rather than its own separately-seeded DB columns.
- Decide + implement the fate of `ModelCatalogEntry`'s price columns (drop, or keep as an override
  layer over the table) — a design decision at implement.
- Reconcile / document the eval-runner `_PRICING` relationship (execution cost path) — at minimum a
  clear statement of which table owns what; ideally converge.

**Out:**
- The ledger, cost snapshotting, and everything else in 6.1 (already shipped there).
- Cache-rate display in the catalog UI unless trivial.

## Acceptance criteria
- [ ] Catalog display price for a model equals the 6.1 authoritative table's rate for that model — test
- [ ] No third divergent price source remains unaccounted for (each of catalog / ledger / eval-runner
      has a documented owner) — inspection
- [ ] Backend + web tests green; EF migration if columns change
