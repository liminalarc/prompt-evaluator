## Problem
Walking a heavier prompt through the loop (round-debrief, [[5.1]]) surfaced a batch of eval-loop gaps that
[[2.8]] (round 2) didn't cover: a run can **fail silently** on timeout, heavy runs **time out at all**
because execution is synchronous, the analytics dataset picker **leaks across prompts**, one rubric
**unfairly caps** mixed-richness fixtures, and a **single noisy run misleads** a version comparison.
**Shipped** = the eval loop runs reliably (async, loud on *any* failure), stays correctly scoped, and scores
fairly and stably enough to trust a version-over-version call.

## Value
As an operator I want eval runs that don't time out or fail silently, a prompt-scoped picker, rubrics that
score each fixture fairly, and a stable/variance-aware score, so the loop is trustworthy as prompts and
datasets get heavier.

## Scope
**In (round-debrief findings — see `specs/5.1/findings.md`):**
- **[B8] Analytics dataset picker → prompt-scoped.** Filter the picker by the selected prompt
  (`Dataset.PromptId`), not just the org — a dataset belongs to one prompt, so foreign-prompt datasets must
  not appear. Same fix shape as B3 (the run picker, shipped in 2.8).
- **[R2] Loud on _any_ run failure.** Surface a banner for a timeout / gateway / non-JSON 5xx, not only the
  structured `502 {error}` — B2's loud-error path has a hole for timeouts (a run failed with nothing on screen).
- **[R1] Async eval execution.** Kick off a run as a job, return immediately, poll for status/results —
  removes the ~100 s synchronous HttpClient/gateway timeout ceiling that made round-debrief flaky, and
  unblocks bigger datasets + heavier prompts. **(Heavy slice — architecture; may split to its own spec.)**
- **[R3] Data-conditional / per-fixture rubric scoring.** Let a rubric (or fixture metadata) express "don't
  expect X when the input can't supply it," so a mixed rich+sparse dataset scores each fixture fairly (a
  single rubric capped the sparse fixture at 0.62 vs 0.90 once made conditional). Alt shape: per-fixture criteria.
- **[R4] Score stability.** A multi-run / variance view (repeat a run N×, show mean ± spread) so a ±0.1
  wobble isn't mistaken for signal; and a **rationale-diff** in Compare — surface the judge's rationale delta,
  not just the score delta (a real prompt gain scored flat; only the rationale showed it).
- **[R5] Hold the subject model constant across versions.** Add-version **defaults Target model to the
  latest version's** (holding the model is the default, not a thing to remember) and **warns** on a change;
  Analytics/Compare **flags a cross-model comparison**. On round-debrief v2/v3 silently ran on Sonnet 5 while
  v1 ran on Sonnet 4.6, confounding the prompt result. Sibling to 1.16's same-scorer-config rule: hold both
  identity axes (subject model + scorer config) when comparing versions.

**Out:** version-status / `Regressed` / backport badges (→ [[1.16]]); the advisory that *consumes* these
signals to suggest next versions (→ [[2.2]]); weighted composite scoring (→ [[2.9]]).

## Acceptance criteria
- [ ] Analytics dataset picker shows only the selected prompt's datasets — unit + e2e [B8]
- [ ] A run that times out / returns a non-`{error}` 5xx shows a clear banner, never a silent no-op — test [R2]
- [ ] A run executes asynchronously (job + poll); a run longer than the old sync ceiling completes — test [R1]
- [ ] A rubric can score a fixture against what its input supports; a sparse fixture isn't penalized for absent analyses — test [R3]
- [ ] Analytics can show per-version mean ± spread over repeated runs, and a rationale delta in Compare — unit + e2e [R4]
- [ ] Add-version defaults Target model to the latest version's + warns on change; Compare flags cross-model — unit + e2e [R5]
