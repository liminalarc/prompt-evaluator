## Problem
Every user is auto-assigned a **"Default" organization** on sign-up. In practice it's a placeholder
that never holds content (users create real orgs), so it reads as clutter/waste in the org rail
(2.20 W39). Three coupled gaps (user, 2026-07-19):
1. **You can't cleanly get rid of it.** Deleting the Default org should be allowed like any other —
   but today it's the auto-assignment target, so deleting it risks leaving new/other users with no
   landing org (or a dangling reference).
2. **A user with *no* orgs isn't reflected in the app.** The rail/dashboard/prompts assume ≥1 org; a
   zero-org state should show a real **empty state / onboarding**, not a phantom default.
3. **There's no way to join someone else's org.** Onboarding today can only *create* an org. A user
   who belongs to a team needs to **request access** to an existing organization, and its owner needs
   to **approve/deny** — the missing half of the onboarding story (user, 2026-07-19).

## Value
As a user I want my org list to reflect *my* real organizations — no placeholder I can't remove — and
if I have none, a clear path to either **create** an org **or request to join** an existing one. As an
org owner I want to review and approve access requests, so membership is controlled without me having
to know every teammate's email up front (the 4.5 add-by-email path is the inverse; this is pull, not
push).

## Scope (to confirm at plan-time)
- **Auto-assignment policy.** Decide what sign-up does: (a) keep auto-creating a personal "Default" org
  but make it fully deletable; (b) stop auto-assigning and drop new users into the no-org onboarding;
  (c) auto-assign only when invited to an existing org. Backend (Identity / org-membership seeding) +
  the `IUserDirectory`/org grant path.
- **Deletable Default org.** Ensure the Default org has no special protection — the 4.4 admin delete and
  the owner/last-owner rules apply uniformly. Guard against orphaning: if deleting an org would leave a
  user with none, that's fine *if* the app has a no-org state (below); the last-owner guard still keeps
  a surviving org with ≥1 owner.
- **No-org empty state / onboarding (front end).** `OrgContextStore` already resolves to `null` when
  there are zero orgs and the rail shows "No organizations yet" — extend this into a real first-run
  surface: dashboard/prompts show an onboarding call to action offering **two paths — create an org or
  request to join one**. Ensure `?org=` / localStorage resolution and all org-scoped pages degrade
  cleanly to the empty state.
- **Data integrity.** What happens to a user whose only org is deleted (by them or an admin); re-entry
  flow; no broken `currentOrgId`.
- **Request-to-join access system (the pull path).** A user can **request access** to an existing
  organization; the org's **Owner** (or a workspace admin) sees pending requests and **approves**
  (grants membership at a chosen role) or **denies**. Design decisions to confirm at plan-time:
  - **Discovery:** how does a requester name the org — pick from a directory of all orgs, enter an
    exact name/slug, or only via a shared invite link/code? (Privacy: a full org directory may leak
    org names; an invite-code or exact-name match is tighter.)
  - **Backend:** a new `OrgAccessRequest` aggregate (requester, org, requested role, status:
    Pending/Approved/Denied, timestamps) with endpoints to create a request, list an org's pending
    requests (owner-gated, sibling to 4.5's member endpoints), and approve/deny (approve → the existing
    `IUserDirectory` grant path). Domain rules: no duplicate open request; can't request an org you're
    already in; approve is idempotent.
  - **Frontend:** a **"Request access"** entry in the no-org onboarding (and reachable from the rail);
    an owner **Requests** surface — likely a tab or section on the org detail page ([[2.20]] W40's
    Overview · Members → Overview · Members · **Requests**), with a pending count.
  - **Notification:** owner-notified-of-request and requester-notified-of-decision ride the existing
    `IEmailSender` seam (dev logs; real provider at deploy — [[3.2]]). In-app is the baseline; email is
    a nice-to-have.

**Out:** multi-org invitations/SSO auto-provisioning beyond the auto-assignment decision ([[4.2]]);
approval *workflows* beyond a single owner/admin approve-deny (no multi-step review).

## Acceptance criteria
- [ ] The "Default" org can be deleted like any other org (no special-casing), subject to the existing
      last-owner guard.
- [ ] A user with **zero** organizations sees a coherent onboarding across the shell (rail), Dashboard,
      and Prompts — **create an org or request to join one** — never a phantom/placeholder org.
- [ ] Sign-up's org-assignment policy is decided and implemented (see Scope), with no orphaned
      `currentOrgId` / broken `?org=` state.
- [ ] **Request-to-join:** a user can request access to an existing org; an owner/admin sees pending
      requests and approves (grants membership at a role) or denies; approved → the user is a member.
      Domain rules hold (no duplicate open request; can't request an org you're in; idempotent approve).
- [ ] Tests: backend (assignment + delete rules; access-request lifecycle + gating), web (no-org
      onboarding with both paths; owner request queue; rescope to null).
