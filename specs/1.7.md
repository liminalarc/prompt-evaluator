## Problem
The registry (1.1) is a **flat list** of prompts, and everything *about* a prompt lives apart
from it: datasets are global (`/datasets`, unattached to any prompt) and analytics are a
separate dashboard keyed by a `Prompt × Dataset` pair. Owners can neither organize prompts nor
see a prompt as a whole.

This spec makes the **prompt the unit everything hangs off** — its versions, its datasets, and
its analytics **live together** — and organizes prompts into a **folder tree** (top-level folders
with **optional subfolders**) so they're browsable by area. It also establishes the boundary
multi-user access is later granted on: **permissions attach to the top-level folder** (subfolders
inherit), which spec 4.1 builds on.

"Shipped" means: a `Dataset` belongs to exactly one `Prompt`; a prompt can live in a folder, be
browsed by folder, and be moved between folders; and a single prompt-workspace page shows that
prompt's versions, datasets, and analytics together — all through the SPA.

## Value
As a prompt owner I want each prompt to be a self-contained workspace (versions + datasets +
analytics in one place), organized into folders by area, so that I can find and manage everything
about a prompt at a glance — and so access can later be granted per top-level folder.

## Scope
**In:**
- **`Folder` aggregate** forming a tree: name, optional parent. A prompt belongs to exactly one
  folder, with a **default/root folder** for otherwise-unfiled prompts.
- The **top-level folder is the permission unit** — the model resolves a prompt's top-level
  ancestor cheaply (store the root ref or walk parents); 4.1 grants access there and subfolders
  inherit.
- **`Dataset` → `Prompt` ownership** (the reshape) — a `Dataset` gains an owning `PromptId`; it is
  created under a prompt, and running a prompt against a dataset from a *different* prompt is
  rejected. Retrofits the 1.2 dataset model to single-owner.
- **API:** create/rename/move folders; move a prompt to a folder; list prompts by folder; fetch
  the folder tree; datasets nested under a prompt; analytics scoped to a prompt.
- **Angular:**
  - A folder tree / sidebar in the prompts browse UI, create-folder, move-prompt, and a
    breadcrumb; the 1.1 list becomes folder-scoped.
  - A **unified prompt-workspace page** — one prompt's versions, its datasets, and its analytics
    on a single screen.

**Out:**
- The permissions/auth themselves (**4.1** — this spec only fixes the boundary).
- **Shared / cross-prompt datasets (many-to-many)** — deliberately excluded; re-homed to **1.8**
  (cloning is the likely future shape). See front-matter `deferrals`.
- Automatic foldering from captured app data (1.2). Moving *versions* between prompts.

## Acceptance criteria
- [ ] Folders form a tree (top-level + optional subfolders); a prompt belongs to exactly one folder — proven by Domain + Infrastructure tests
- [ ] A prompt's top-level folder (the permission boundary) is resolvable/queryable — proven by a test (consumed by 4.1)
- [ ] A dataset belongs to exactly one prompt; a run rejects a dataset owned by another prompt — proven by Domain + Application tests
- [ ] Folders, prompt-placement, and datasets-under-a-prompt are creatable/movable/browsable via the API — proven by API tests
- [ ] The SPA browses prompts by folder (tree navigation + breadcrumb, move a prompt) — proven by unit tests + one e2e
- [ ] The prompt-workspace page shows the prompt's versions, its datasets, and its analytics together — proven by a unit test + one e2e

## Decisions
- 2026-07-12 — Grouping is a **folder tree with optional subfolders**; the **permission
  boundary is limited to the top-level folder** (subfolders inherit) — user decision.
- 2026-07-13 — **Datasets belong to a prompt** (single owner), not global and not folder-owned.
  Rationale: fixtures are inputs shaped to one prompt's contract, and the genuinely reusable
  cross-prompt concern (scorers) is already composable per dataset (`IScorer` × `ScorerConfig`);
  version-comparison (1.4) already covers "old vs new prompt." Sharing across prompts is excluded
  as a premature abstraction and re-homed to **1.8** (likely via *cloning* rather than
  many-to-many, per user). — user decision.
- 2026-07-13 — Scope grew to include a **unified prompt-workspace page** (versions + datasets +
  analytics together), kept in **one spec** rather than split — user decision.
