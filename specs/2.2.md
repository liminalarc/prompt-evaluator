## Problem
Once we can score prompts and track versions, the natural next value is advice: help the
prompt owner write a *better* next version. This is the "prompt engineering layer on top"
from the original concept. It is deliberately deferred — it only becomes cheap and
trustworthy once the registry, harness, and score history exist to ground its suggestions
in real evidence rather than generic best-practice platitudes.

## Value
As a prompt owner I want evidence-grounded suggestions for improving a prompt so that I can
draft a better next version instead of guessing.

## Scope
**Sibling:** [[2.7]] owns *proactive, best-practice* authoring help (at the keyboard, no eval history);
this spec owns the *evidence-grounded* half below (run-derived). Scope split, not overlapping.

**In (when picked up):**
- Given a prompt version and its eval history, generate rewrite suggestions grounded in
  the actual failing/low-scoring fixtures — not generic advice.
- Explain score movement: "v3 beat v2 on extraction because … / regressed on tone because …"
  using the run data from 1.4.
- Suggestions land as a *draft* new version in the registry (1.1) for the owner to accept,
  edit, or reject — never auto-applied.
- Angular surface: advisory panel on a prompt version showing suggestions + rationale +
  "create draft version".
- **Diagnose from judge _rationales_, not just scores** — a real quality gain can leave the aggregate
  **flat** (5.1 R4: round-debrief v2 removed fabricated benchmarks/predictions but the number didn't move).
  The advisory must read the per-fixture rationale, not only the delta.
- **Advise on the rubric + dataset composition, not only the prompt.** A mis-calibrated rubric or a
  heterogeneous dataset caps scores independent of prompt quality (5.1 R3: a single rubric demanded
  front/back-nine analysis a sparse fixture couldn't supply, capping it). The advisory should flag
  "this rubric penalizes what some fixtures can't produce → make it data-conditional, or split the dataset."

**Out:** anything that edits prompts in the source apps directly. **Autonomous prompt-tuning loops — but
see D2:** the 5.1 dogfooding (this is literally the manual loop a human runs) makes a *semi-autonomous*
"propose → run → read rationale → propose again" loop the obvious next want; scope it deliberately, don't
let it creep in.

## Acceptance criteria
- [ ] Suggestions cite the specific fixtures/scores that motivate them — proven by tests over seeded eval history
- [ ] Score-movement explanations reference real per-scorer deltas from 1.4 — proven by tests
- [ ] Accepting a suggestion creates a draft `PromptVersion` (never auto-applied) — proven by Application tests
- [ ] The advisory panel renders suggestions + rationale and can create a draft — proven by unit + e2e

## Decisions
- 2026-07-11 — Deferred until Registry (1.1), Harness (1.3), and Analytics (1.4) exist;
  advisory must be grounded in real eval evidence, and suggestions are draft-only.
- 2026-07-18 (from the 5.1 round-debrief walk) — concrete eval-grounded checks the advisory should ship
  with: (1) **data-starvation → fabrication** — on sparse fixtures models substitute generic benchmarks,
  make score predictions, and narrate missing data; suggest adaptive structure ("skip analyses the data
  can't support"), explicit bans, and a **few-shot** when instructions alone don't land (v2→v3);
  (2) **rationale over score** — surface the judge's per-fixture rationale, since a real fix can score flat
  (R4); (3) **rubric/dataset advisory** — detect a rubric that penalizes impossible-for-this-fixture
  criteria and recommend data-conditional rubrics or a homogeneous split (R3). Pairs with 2.7's authoring
  checks (length-as-ceiling, no-compute, specificity).
- 2026-07-18 (D2, **open — user raised**) — the 5.1 manual loop (read scores+rationales → propose next
  version → re-run) is exactly what the user wants *embedded* and *iterating* in-app. That is a
  **semi-autonomous iterate loop**, currently in 2.2's "Out". **Decision pending:** keep draft-only
  (human runs each version) or add an opt-in bounded auto-iterate (propose → run → evaluate → stop on
  budget/plateau, human accepts the winner). Needs a user call before it's scoped in.
