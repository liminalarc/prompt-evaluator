## Problem
Building a *good* test dataset is an artform, and the tool gives no proactive help with it. LitmusAI
has the **mechanics** (capture-first fixtures, synthetic-fills-gaps — [[1.2]]) and, when [[2.2]] lands,
a **reactive** critique (given eval history, "this dataset caps scores → split it / make the rubric
data-conditional"). But nothing helps an operator *assemble a sound dataset in the first place*. A
poorly-curated set silently sabotages the whole loop — 5.1 proved it repeatedly: a single mixed-richness
dataset **capped the sparse fixture** regardless of prompt quality ([[2.12]] R3), and the number alone
never revealed it (R7). This is the dataset-side sibling of [[2.7]] (proactive prompt authoring): craft
guidance, at the point of building the set, before any run exists to diagnose.

## Value
As an operator I want proactive guidance on assembling a representative, well-balanced test set — so my
evals measure the **prompt's** quality, not artifacts of a lopsided or thin dataset, and so I learn the
craft instead of discovering the mistakes only after a misleading run.

## Scope
**Sibling split:** [[2.2]] owns the **reactive, eval-history-grounded** dataset/rubric critique (after
runs exist). This spec owns the **proactive, at-build-time** craft guidance (before runs). [[1.2]] owns
the fixture *mechanics*; this is guidance layered on top, not new storage.

**In (concept — detail later):**
- **Coverage & representativeness** — does the set span the real input distribution (rich vs sparse,
  common vs edge, the failure-prone shapes)? Surface gaps ("no sparse/short-input fixture", "all captured
  from one source").
- **Richness balance & homogeneity** — flag a **mixed-richness** set that a single rubric can't score
  fairly (R3), and offer the choice the walk taught: **split into homogeneous sets** vs a
  **data-conditional rubric**.
- **Capture-vs-synthetic ratio** — nudge toward captured ground truth; warn when synthetic dominates or
  isn't seeded from captures (drift from the real distribution — [[1.2]]).
- **Sufficiency / statistical power** — how many fixtures the current scorer actually warrants: tie to
  judge **determinism vs variance** (5.1: a near-deterministic Opus judge needs fewer repeats than a
  noisy one) and to regression confidence ([[1.11]] small-sample handling).
- **Edge-case prompting** — help enumerate the adversarial / sparse / boundary cases where prompts fail
  (data-starvation → fabrication was found *only* on the sparse edge, twice).
- **Labeling hygiene** — meaningful fixture labels/descriptions so a set reads as scenarios, not GUIDs.

**Out:** the reactive advisory (→ [[2.2]]); fixture storage/capture/synthesis mechanics (→ [[1.2]]);
multimodal / tool-loop fixtures (→ [[1.17]] / [[1.18]]); structured/severity-tagged *judging* (finding R7,
its own future spec) — related, since "good coverage" is partly defined by which failure-modes a set can
provoke, but a separate capability.

## Acceptance criteria
*(Concept stage — sharpen when picked up.)*
- [ ] Given a dataset, the tool surfaces concrete design feedback (coverage gaps, richness imbalance,
      capture/synthetic ratio, fixture-count sufficiency) — proactive, no eval run required.
- [ ] It offers the R3 remedy explicitly on a mixed-richness set: split vs data-conditional rubric.
- [ ] Guidance is actionable and specific (names the missing shape / the offending fixtures), never generic.
- [ ] The craft heuristics are captured as durable reference (guidance content), not only UI.
