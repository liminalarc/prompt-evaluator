## Problem
With prompts (1.1) and datasets (1.2) in place, we need the harness that actually runs a
prompt version over a dataset and scores each output. Scoring must combine fast
deterministic checks with LLM-as-judge for fuzzy quality, under one abstraction so scorers
compose per dataset.

## Value
As a prompt owner I want to run a prompt version against a dataset and get scored results
so that I have objective evidence of how well that version performs.

## Scope
**In:**
- `EvalRun` aggregate: a `Prompt × Version × Dataset` execution producing one `Score` per
  fixture per scorer. Every run persisted; nothing overwritten.
- `IScorer` abstraction with two implementations this spec:
  - **Deterministic** (in-process): regex match, JSON-schema validation, exact/fuzzy
    match, latency/cost capture.
  - **LLM-judge** (→ eval-runner): rubric-based structured scoring via Claude, returned as
    a structured verdict (tool-use / structured output, never free-text parsing).
- Scorer selection/config per dataset (which scorers, which rubric).
- **Judge model selection**: the LLM-judge's Claude model is configurable (Opus 4.8 default;
  Sonnet 5 / Haiku 4.5 for cheaper high-volume judging) and is **part of the Scorer's
  identity**. Changing the judge model yields a distinct scorer — and therefore a distinct
  score series — so trend/regression comparisons (1.4) only ever compare like-for-like.
- API to trigger a run and fetch results; Angular view of a run's per-fixture scores.
- The prompt output itself is produced by calling the prompt's target model through the
  eval-runner (the same service that hosts the judge).

**Out:** human review as a scorer (2.1). Trend/regression analytics across runs (1.4).

## Acceptance criteria
- [ ] An `EvalRun` executes a prompt version over every fixture in a dataset and persists a `Score` per fixture per scorer — proven by Application + Infrastructure tests
- [ ] Deterministic scorers (regex, JSON-schema, exact/fuzzy, latency/cost) work and are pure-unit-tested — proven by Domain/Application tests
- [ ] The LLM-judge scorer returns a structured verdict from a rubric — proven by an eval-runner test (model mocked) + an Application test against a faked `IEvaluationRunner`
- [ ] Scorers are selected/configured per dataset and compose — proven by a run using both scorer types
- [ ] The Angular UI shows a run's per-fixture scores — proven by unit + e2e
- [ ] Runs are append-only; re-running never overwrites a prior run — proven by a test
- [ ] The judge model is configurable and forms part of the Scorer identity; two runs judged by different models record as different scorer series — proven by a test

## Decisions
- 2026-07-11 — One `IScorer` abstraction; deterministic in-process, LLM-judge delegated to
  eval-runner. Judge verdicts are structured (tool-use), never parsed from prose.
- 2026-07-12 — The judge model is part of Scorer identity (Prompt × Version × Dataset ×
  Scorer). Changing it starts a new score series rather than silently shifting an existing
  one, so regression detection only compares like-for-like. Subject-model selection lives on
  the `PromptVersion` (1.1); non-Claude judge/subject execution is spec 1.5. Consult the
  `claude-api` skill for current model ids — never hardcode from memory.
