## Problem
Every model call LitmusAI makes costs money — the **subject execution** (running a prompt), the
**LLM-judge** scoring, and **synthetic fixture generation** — but there is no workspace-wide view of
that spend. Per-fixture token/cost is already captured on `FixtureRun` for subject executions, yet
judge and generation calls aren't ledgered at all, nothing is attributable by org / user / feature,
and there's no way to slice spend by day / model / feature / user / org or track it against a budget.
All three call types funnel through **one seam** — the `.NET` `IEvaluationRunner` → `EvalRunnerClient`
→ Python eval-runner boundary, whose responses already echo tokens (and cost) — so there's a single
integration point to capture everything. Shipped = an **admin-only AI usage & budget tracker**: every
model call is recorded (tokens, model, cost, who / what / when / status), and a workspace admin can
slice that data granularly — date range, model, feature, user, org, status — see spend over time and
breakdowns, drill into individual calls, export, and set budgets with over-threshold alerts.

## Value
As a workspace admin, I want to see and slice exactly what our model usage is costing — by date,
model, feature, user, org, and individual call — and track it against a budget, so I can control
spend and catch runaway cost early.

## Scope
**In:**

_Capture (every call, at the eval-runner seam):_
- Record an **`AiUsageRecord`** on every call through `EvalRunnerClient` — **subject execution,
  LLM-judge, and synthetic generation**, on **success and failure** (a failed call can still incur
  cost and signals waste). Fields: the **model** (as echoed by the runner), token counts (**input,
  output, and — added here — cache-create / cache-read**), the provider **request id**, a UTC
  **timestamp**, **feature/use-case** (`subject-execution` | `llm-judge` | `synthetic-generation`),
  **organization id**, **user id** (the caller), **status** (success / refusal / error), **latency
  ms**, and the **max-tokens** requested.
- **The eval-runner must return the full usage block** (model echoed, cache tokens, request id,
  status) on the judge + generate responses too — `execute` already returns tokens/cost; judge and
  generate currently don't surface usage. (Python change in `eval-runner`.)
- Thread a small **`AiUsageContext`** (feature + org + user) from each Application handler
  (`RunEvaluationHandler` for execute/judge, `GenerateSyntheticFixturesHandler` for generate) down to
  the transport so calls are attributable (explicit param preferred; ambient `AsyncLocal` fallback).
  New **`IAiUsageRecorder`** port (Application); EF-backed adapter (Infrastructure), persisted by the
  main `EvalDbContext`.

_Cost:_
- A **versioned per-model pricing table** (USD per million tokens: input / output / cache-write /
  cache-read), from config. Compute each record's cost and **snapshot** it (+ the rate version) at
  write time, so a later price change never rewrites history. Unknown model → cost null + flagged,
  tokens still stored.
- **De-duplicate pricing.** Cost lives in **two** places today — the eval-runner's `_PRICING` table
  (authoritative) and the Model Catalog's display-only `InputPricePerMTokUsd`/`OutputPricePerMTokUsd`
  (1.13). Make one **.NET** table authoritative for the ledger (extended with cache rates + a version)
  and snapshot from it; the eval-runner keeps returning raw tokens. Reconciling the two is a
  plan-time decision.

_Query & filter (as granular as possible):_
- Filter records by any combination of: **date range**, **model(s)**, **feature(s)**, **user(s)**,
  **organization(s)**, **status**.
- Aggregate / group by **period** (day / week / month), **model**, **feature**, **user**, **org**.
- Per-slice metrics: **total cost**, **token totals** (in / out / cache), **call count**, **avg cost
  & tokens per call**, **success rate**, **latency p50 / p95**.
- A **time-series** (spend over time) + **breakdown tables** (by model / feature / user / org) +
  **top-N**.
- **Drill to individual calls** — a filterable, paginated **calls table** (timestamp, feature, model,
  tokens, cost, user, org, status, latency, request id).
- **CSV export** of the current filtered / aggregated view.

_Budgets:_
- Define **budgets** for a period (monthly to start): a **global** (workspace) budget + optional
  **scoped** budgets (per model / feature / org).
- Track **spend vs budget** (amount + % used) with **threshold alerts** (e.g. 80% / 100%) surfaced in
  the admin UI.

_Admin surface & authorization:_
- An **Admin → AI Usage** area: dashboard (time-series + breakdowns + budget status), the calls table
  with filters, budget config, export.
- Gated to the existing **workspace global-admin** flag (`OrgAccess.IsGlobalAdminAsync`), matching the
  Model Catalog (1.13) and admin surfaces (4.3 / 4.4). This is a **workspace-wide management surface**,
  not org-membership-scoped — spend is workspace-wide.

**Out:**
- **Hard enforcement** (blocking a run when over budget) — tracking + alerting only here; enforcement
  changes the run path and is a follow-on decision.
- **Storing prompt / response content** — metadata + token counts only. (Note: `FixtureRun` already
  stores subject model *output* for eval runs; the usage **ledger** stores no prompt/response content
  and never keys/secrets.)
- **Multi-provider beyond what the eval-runner already routes** — the port is provider-neutral and the
  runner already routes Anthropic + OpenAI by model id (1.5); no new provider wired here.
- **Automated pricing sync** from the provider — pricing is config/seed, updated by hand for now.

## Decisions
- 2026-07-19 — **Adapted from another product's spec 6.1** (origin below). The source captured at a
  .NET `AnthropicMessagesClient` and introduced a new system-admin role; LitmusAI's shape differs on
  three points, folded in here: (a) the choke point is the **Python eval-runner via `EvalRunnerClient`**
  (not a .NET Anthropic client); (b) a **workspace global-admin already exists** so the source's
  "admin authorization is a real gap" decision is resolved — gate on `OrgAccess.IsGlobalAdminAsync`;
  (c) tenancy is **Organization**, not "project".
- 2026-07-19 — **Capture at the eval-runner seam.** Subject/judge/generate all funnel through
  `EvalRunnerClient`; recording there (via `IAiUsageRecorder`) catches 100% of usage with one
  integration point. `execute` already returns tokens/cost; judge + generate need their responses
  extended to surface usage (model echoed, cache tokens, request id, status).
- 2026-07-19 — **Immutable cost via snapshot**, from a **versioned .NET pricing table** made
  authoritative for the ledger — resolving the current split between the eval-runner `_PRICING` table
  and the catalog's display-only price (1.13). Store token counts always; freeze cost + rate version
  on the record.
- 2026-07-19 — **Metadata only, no content** in the ledger, keeping the "never log prompts or keys"
  rule. (`FixtureRun` separately keeps subject *output* for eval-run inspection; the ledger does not.)
- 2026-07-19 — **Relationship to `FixtureRun` token/cost** is a plan-time decision — the ledger is the
  cross-cutting workspace record across all three call types; `FixtureRun` keeps its per-fixture
  cost for run analytics. Likely complementary, not superseding.
- 2026-07-19 — **(plan-time, user-chosen) Pricing reconciliation = ledger table only.** One new
  versioned, config-backed .NET pricing table (input/output/cache-write/cache-read + rate version) is
  authoritative for the ledger. The eval-runner `_PRICING` (execution cost → `FixtureRun`) and the
  Model Catalog's display-only price (1.13) are left as-is; the residual three-way divergence is
  re-homed to **6.2** (deferral recorded). Rejected fuller option: make the catalog display price
  derive from the same table (edits 1.13).
- 2026-07-19 — **(plan-time, user-chosen) Full 6.1 in one spec** — all six slices incl. Budgets ship
  under 6.1; nothing split out except the 6.2 pricing-unification follow-on above.
- 2026-07-19 — **Attribution threading.** `AiUsageContext` (feature + org + user) passed as an explicit
  param from `RunEvaluationHandler` (execute) and `GenerateSyntheticFixturesHandler` (generate); the
  judge path runs indirectly through `LlmJudgeScorer` (no org/user), so an ambient `AsyncLocal`
  carries context across that seam. Org id derives from `Prompt.OrganizationId`; user from `ICurrentUser`.

## Origin & adaptation (porting note)
Ported from a sibling product's spec 6.1, which was written to adapt to any product with a **single
AI-call choke point**:
- **Capture seam** → wherever all provider calls funnel. Here: the eval-runner boundary
  (`EvalRunnerClient` / `IEvaluationRunner`), *not* a direct .NET Anthropic client — LitmusAI's model
  calls live in the Python eval-runner (`eval-runner/app`, routed by model id — 1.5).
- **Usage fields** come from the provider response (`model`, `usage.*_tokens`, request `id`), relayed
  through the eval-runner's HTTP responses.
- **Attribution** (org/user) → LitmusAI's Organization tenancy (1.9) + identity (4.1).
- **Admin boundary** → the existing workspace global-admin flag (1.13 / 4.3), no new role needed.
- **Pricing table** is provider-agnostic (USD per million tokens per model); keep it versioned +
  snapshotted, and reconcile with the eval-runner `_PRICING` + catalog price.
Everything else (filter/aggregate read model, calls table, CSV, budgets, alerts) is domain-neutral.
