## Problem
Dogfooding the eval loop on the prompt workspace (`/prompts/:id`) surfaced UX rough edges that
make the page harder to use than it should be. This spec collects those fixes. It descends from
2.4 (the app-shell / design-system overhaul) as follow-on polish; new findings from continued
dogfooding get appended here rather than scattered. "Shipped" means the workspace reads cleanly and
each action sits with the thing it acts on.

## Value
As an operator building up the eval testbed I want the prompt workspace controls to sit where I
expect them and lay out cleanly, so that adding versions/datasets isn't confusing.

## Scope
**In:**
- **Add-version control placement** — the `+ Add version` toggle currently renders at the very
  bottom of the page (below the Analytics section), disconnected from the **Version history** table
  it belongs to. Move it adjacent to that section (its logical home).
- **Add-version form layout** — the revealed form (`.add-version` in `web/src/app/prompts/prompt-detail.ts`)
  lays out with the fields crammed into the right half and a dead empty left column; controls read
  as misaligned. Fix to a clean, aligned form.
- Sweep the workspace for the same toggle-at-the-bottom / reveal-layout pattern on the other
  progressive-disclosure sections (datasets, add-scorer, capture/generate) and fix consistently.

**Out:** Functional changes to versioning/datasets/analytics (this is presentation only). New
features. Backend changes.

## Acceptance criteria
- [ ] The `+ Add version` control sits with the Version history section, not below Analytics — proven by inspection + the workspace e2e still green
- [ ] The add-version reveal form lays out cleanly (no dead column; fields + submit aligned) in light and dark — proven by inspection
- [ ] Other reveal forms on the page follow the same corrected pattern — proven by inspection
