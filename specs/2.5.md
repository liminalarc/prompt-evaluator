## Problem
Dogfooding the eval loop end-to-end (prompt workspace → dataset → run → eval-run detail) surfaced
UX rough edges that make the flow harder to use than it should be. This spec collects those fixes.
It descends from 2.4 (the app-shell / design-system overhaul) as follow-on polish; new findings
from continued dogfooding get appended here rather than scattered. "Shipped" means the eval loop
reads cleanly, each action sits with the thing it acts on, and run output is legible.

## Value
As an operator building up the eval testbed I want the prompt workspace controls to sit where I
expect them and lay out cleanly, so that adding versions/datasets isn't confusing.

## Scope
**In:**
- **Add-version control placement** — the `+ Add version` toggle currently renders at the very
  bottom of the page (below the Analytics section), disconnected from the **Version history** table
  it belongs to. Move it adjacent to that section (its logical home).
- **Add-version form layout** — the revealed form (`.add-version` in `web/src/app/prompts/prompt-detail.ts`)
  lays out with the fields crammed into the right half and a dead empty left column; controls read
  as misaligned. Fix to a clean, aligned form.
- Sweep the workspace for the same toggle-at-the-bottom / reveal-layout pattern on the other
  progressive-disclosure sections (datasets, add-scorer, capture/generate) and fix consistently.
- **Eval-run detail (`/eval-runs/:id`, `web/src/app/eval-runs/`) output rendering.** Each fixture's
  result is a raw dump: a literal `output` prefix, the model's ```` ```json … ``` ```` fences left
  unparsed, and `latency NNNNms` concatenated onto the same wrapped text blob. Pretty-print the
  model output (JSON when it parses), and present latency / cost / tokens as distinct labeled
  fields; show the fixture input alongside its output.
- **Runs list doesn't refresh after a run.** After clicking Run evaluation the new run doesn't
  appear in the Runs table until a re-navigation (Back); refresh the list on run completion.

**Also observed (needs a product decision, possibly its own spec):** the dataset **capture-fixture
form** (`web/src/app/datasets/dataset-detail.ts`) exposes only *Prompt input* and *Upstream SLM
output* — there's no field to set an **expected/reference output**, even though the capture API and
the `Fixture.expectedOutput` field support it. That's fine for capture-first fixtures (expected
output often unknown) but blocks authoring **labeled** test sets from the UI (so ExactMatch/
FuzzyMatch scoring has nothing to compare against; only Config-driven scorers like Regex/JsonSchema/
LlmJudge work). Decide whether to add an optional "expected output" field to the capture form.

**Out:** Functional changes to versioning/datasets/analytics beyond the above (this is mostly
presentation). Deeper fixture-authoring features. Backend changes.

## Acceptance criteria
- [ ] The `+ Add version` control sits with the Version history section, not below Analytics — proven by inspection + the workspace e2e still green
- [ ] The add-version reveal form lays out cleanly (no dead column; fields + submit aligned) in light and dark — proven by inspection
- [ ] Other reveal forms on the page follow the same corrected pattern — proven by inspection
- [ ] The eval-run detail page pretty-prints model output and shows latency/cost/tokens as labeled fields (no raw `output```json…``` blob) — proven by inspection
- [ ] The Runs list refreshes when a run completes (no Back required) — proven by inspection + e2e
