## Problem
The Model Catalog (from [[1.13]]) is a single workspace-wide list managed by a global admin. As
soon as more than one organization ([[1.9]]) shares the workspace, they may want different model
sets — an org restricted to a provider, an org with an extra model, different defaults. [[1.13]]
scoped this out ("a single workspace-wide catalog for now; per-org customization is future").

Origin: re-homed from [[1.13]] as an open deferral (`what: per-org catalogs`).

## Value
As an org owner I want to control which catalog models my organization can use — so each org's
droplists reflect its own allowed/available set rather than one shared list.

## Scope
**In (to refine at implement-time):**
- A per-org overlay on the workspace catalog: an org can enable/disable (and possibly add) models
  for its members; the droplists (target/judge) filter to the org's effective set.
- Gating tied to org roles ([[4.1]] `OrgRole.Owner`) rather than the global-admin flag 1.13 added
  for the workspace-wide catalog — decide the exact model at plan-time.
- Backward-compat: an org with no customization inherits the full workspace catalog.

**Out:**
- The base workspace catalog + global-admin management — delivered in [[1.13]].
- Live discovery — see [[1.14]].

## Acceptance criteria
- [ ] An org owner can restrict/extend their org's catalog set; members' droplists reflect it — proven by a test
- [ ] An org with no customization still sees the full workspace catalog — proven by a test
- [ ] Per-org changes don't affect other orgs — proven by a test

## Plan (thin slices)
1. [ ] Domain + persistence for a per-org catalog overlay (enable/disable per org). -> <sha>
2. [ ] Org-scoped `GET /api/models` (effective set) + org-owner-gated management endpoints. -> <sha>
3. [ ] Web: org-scoped droplists + per-org management UI. -> <sha>
