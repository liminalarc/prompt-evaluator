## Problem
Subject model is currently **pinned per version** (`PromptVersion.TargetModel`, immutable), so "the same
prompt on two models" means two versions — which is exactly the confound **R9** (2.9a) guards against by
*excluding* cross-model versions from the backport comparison. That keeps the *accidental* case honest but
leaves the *intentional* one unserved: there is **no first-class way to deliberately compare one prompt
across models** ("which model runs this prompt best / cheapest?"). R5 (2.12) only *warns* on a cross-model
comparison; R9 (2.9a) *excludes* it; neither *enables* it.

## Value
As a prompt owner I want to run one prompt+dataset across several subject models and compare quality/cost/
latency side by side, so I can pick the best (or cheapest-good-enough) model for it — a deliberate bake-off,
not an accidental confound.

## Scope
**In (when picked up):** make the subject model a **comparison axis** rather than a baked-in version
property — run a version across N models, a model-comparison view (per-model score/cost/latency over the
same dataset+scorers), honest same-prompt-different-model deltas. Interaction with score identity
(`Prompt × Version × Dataset × Scorer`) and with R9's same-model backport rule is a plan-time design call.
**Out:** the backport recommender's same-model discipline (that's R9/2.9a — the inverse concern); provider/
model catalog plumbing ([[1.13]]/[[1.5]], already shipped).

## Acceptance criteria
- [ ] One prompt version can be evaluated across multiple subject models over the same dataset+scorers
- [ ] A comparison view shows per-model quality + cost + latency for the same prompt, without the R9 confound
      (the axis is explicit, not smuggled through per-version model drift)

## Decisions
- 2026-07-20 — Created from a question raised during the R9 build (5.1 dogfooding): R9 *excludes* cross-model
  versions from the backport pick; this spec is the *intentional* counterpart. Origin:
