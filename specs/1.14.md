## Problem
The Model Catalog (from [[1.13]]) is managed manually — an admin adds/edits entries by hand and
the seed is a fixed set. When a provider ships a new model (or retires one), the catalog drifts
until someone updates it. [[1.13]] deliberately scoped discovery out ("manual/seeded management
here; discovery is a future spec") — this is that spec.

Origin: re-homed from [[1.13]] as an open deferral (`what: live model discovery`).

## Value
As an admin I want the catalog to stay in sync with what each provider actually offers — new
models appear automatically and retired ones are flagged — so I don't have to hand-maintain the
list or discover a stale entry only when an eval fails.

## Scope
**In (to refine at implement-time):**
- A discovery path that lists a provider's available models (Anthropic Models API, OpenAI models
  endpoint) and reconciles against the catalog: surface new models, flag missing/retired ones.
- Admin-reviewed sync (propose changes, admin confirms) vs. fully automatic — decide at plan-time.
- The eval-runner likely owns the provider calls (it holds the credentials), exposing discovery
  results that .NET reconciles into the catalog — consistent with 1.13's availability seam.

**Out:**
- The catalog aggregate / droplists / admin CRUD themselves — delivered in [[1.13]].
- Per-org catalogs — see [[1.15]].

## Acceptance criteria
- [ ] Discovery lists a provider's current models and diffs them against the catalog — proven by a test
- [ ] New provider models can be added to the catalog without hand-typing ids — proven by a test
- [ ] A retired/absent provider model is flagged (not silently dropped) — proven by a test

## Plan (thin slices)
1. [ ] Provider model-listing in the eval-runner (Anthropic + OpenAI), behind the existing seam. -> <sha>
2. [ ] .NET reconciliation of discovery results against the catalog (diff: new / retired). -> <sha>
3. [ ] Admin sync UI (review + confirm) on the Model Catalog page. -> <sha>
