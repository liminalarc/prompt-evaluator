## Problem
The prompts we want to evaluate are embedded in three (and growing) applications, with no
stable identity or version history. Scoring is meaningless without a stable
`Prompt × Version` identity to attach scores to. We need a registry that holds prompts
and their versions, seeded by copying prompts in — while leaving the door open for
Zatomic (our own prompt/version tool) to become the backing store later.

## Value
As a prompt owner I want each embedded prompt registered with a version history so that I
can attach eval scores to a specific version and compare versions over time.

## Scope
**In:**
- `Prompt` aggregate with an ordered `PromptVersion` history (content, created-at, label,
  optional source-app reference). Invariants: versions are append-only; content is
  immutable once a version exists.
- **Target (subject) model** on each `PromptVersion` — the model the prompt is meant to run
  against, stored as an opaque identifier (a Claude tier today; a non-Claude/SLM id later).
  It is version data, not global config, so the same registry holds Claude and non-Claude
  prompts. Not validated against a live provider here; multi-provider execution is spec 1.5.
- `IPromptRepository` port in Application; an EF Core (Postgres) implementation in
  Infrastructure.
- Copy-in ingestion: create a prompt and add versions via API; import a prompt's text
  from a pasted value or file.
- API endpoints + DTOs to create prompts, add versions, list/browse.
- Angular browsing: list prompts, view a prompt's version history and diff between versions.

**Out:** Zatomic integration (future adapter behind the same port — noted, not built).
Datasets/fixtures (1.2). Running/scoring (1.3).

## Acceptance criteria
- [ ] A prompt can be created and given multiple versions; versions are append-only and immutable — proven by Domain + Application tests
- [ ] `IPromptRepository` persists and retrieves prompts with full version history — proven by an Infrastructure test against real Postgres
- [ ] Prompts and versions are creatable/browsable via the API — proven by API tests
- [ ] The Angular UI lists prompts and shows version history + a version diff — proven by unit tests + one e2e
- [ ] The port is documented as the Zatomic seam; no domain type depends on the storage choice — proven by inspection
- [ ] A `PromptVersion` records its target model and round-trips through the repository and API — proven by tests

## Decisions
- 2026-07-11 — Copy-in first, `IPromptRepository` port from day one so Zatomic can back it
  later without touching Domain/Application.
- 2026-07-12 — Each `PromptVersion` carries a target (subject) model identifier as version
  data, stored as an opaque id so the registry is provider-agnostic from the start. The
  harness (1.3) reads it to know what to run the prompt against; live non-Claude execution
  is deferred to spec 1.5.
