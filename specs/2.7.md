## Problem
[[2.2]] (Advisory) is deliberately *evidence-grounded* — it needs eval history to suggest a better
version. There is no home for **proactive, best-practice authoring help**: critique and improve a
prompt *while you write it*, before any run exists. That is the assistance a prompt owner wants at the
keyboard — and exactly what accelerates [[5.1]]'s manual "author v2" step, 54× over.

## Value
As a prompt owner I want AI critique and rewrite suggestions grounded in prompt-engineering best
practices while I author any version, so I can draft a stronger prompt without waiting on eval history.

## Scope
**In:**
- An **"improve / critique this prompt"** action on a prompt version: given the prompt text (+ optional
  light context — intended task, target model, expected I/O shape), return (a) a **structured critique**
  across best-practice dimensions (clarity, specificity, role framing, explicit output-format, ambiguity,
  examples/few-shot need, guardrails) and (b) a **concrete rewrite**. Dimensions include the ones 5.1
  dogfooding proved matter: **length instruction as ceiling vs target**, **anti-fabrication / no-compute**,
  and **guidance specificity** (see Decisions).
- Optionally help author the **judge rubric**, not just the prompt — in 5.1 the rubric was as load-bearing
  as the prompt itself.
- Suggestions land as a **draft new version** — reuse [[2.2]]'s draft-only rule; never auto-applied.
- Runs through the **`IEvaluationRunner`** seam (Claude default). The advisory/meta-prompts are
  **first-class LitmusAI prompts** — versioned, not hardcoded strings — and themselves onboardable into
  LitmusAI's own registry + scored by its own harness (dogfood-your-own-advisor; a natural [[5.1]]-style
  third target).
- Angular **authoring-assist panel**, distinct from [[2.2]]'s eval-history advisory panel.

**Out:** evidence-grounded, run-derived advice (→ [[2.2]]); autonomous tuning loops; editing source-app
prompts directly; authoring-form visual polish (→ [[2.3]]).

**Sequencing:** not gated behind Phase 1 — needs only the registry ([[1.1]] ✅) and the LLM seam (✅),
not eval history — so it can land earlier than [[2.2]].

## Acceptance criteria
- [ ] Critique returns structured, dimensioned feedback on a prompt's text — proven by tests
- [ ] Accepting a suggestion creates a **draft** `PromptVersion`, never auto-applied — proven by Application tests
- [ ] The advisory/meta-prompt is a versioned LitmusAI prompt (not a hardcoded string) — proven by inspection/tests
- [ ] Authoring-assist panel renders critique + rewrite and can create a draft — proven by unit + e2e

## Decisions
- 2026-07-18 — Created as a sibling to [[2.2]] (user decision): split proactive best-practice authoring
  help from 2.2's evidence-grounded advisory — different trigger (at authoring time vs post-run) and
  grounding (best practices + the text vs real fixtures/scores). Origin: idea raised while onboarding
  5.1 prompts.
- 2026-07-18 — Advisory/meta-prompts are first-class versioned LitmusAI prompts (not inline strings), so
  the assistant can be dogfooded through LitmusAI itself.
- 2026-07-18 — Concrete critique dimensions learned from the 5.1 daily-briefing v1→v2 walk (0.55→0.88):
  (1) **length as ceiling vs target** — "N words *maximum*" makes models undershoot; flag ceiling wording
  and suggest a firm target reached by citing real data; (2) **anti-fabrication / no-compute** — models
  invent or miscompute derived stats to fill space; flag missing "use only provided data, don't compute
  new figures" guardrails; (3) **guidance specificity** — flag vague/generic advice not tied to the input.
  These are real, high-signal checks the assistant should ship with.
