## Problem
Organizations are the permission boundary ([[1.9]], [[4.1]]), but there's no admin surface to
manage the orgs **themselves**. Today an org is *created* as a side effect of the Prompts page
(creating one grants the creator Owner); there is no way to list all orgs, rename one, or delete
one from an admin view. [[4.3]] added an Admin folder (Users, Models) and manages a user's
*membership* in orgs, but deliberately left org-entity management to this spec.

Origin: split from [[4.3]] (user decision — 4.3 stays user-focused).

## Value
As a global admin I want to manage organizations — see them all, create, rename, and delete — from
the Admin area, rather than only creating them implicitly on the Prompts page.

## Scope
**In (to refine at implement-time):**
- Admin **Organizations** page under the Admin folder ([[4.3]]): list all orgs (with member counts),
  create, rename, delete (delete cascades folders/prompts/datasets/runs — behind a clear confirm).
  Gated by the global-admin flag ([[1.13]]).
- Admin-gated org endpoints (list-all / rename / delete); the `Organization` aggregate already
  supports `Rename` and the repository supports cascade-delete ([[1.10]]).
- Optionally an org-first member view (who belongs to this org) — decide vs. leaving membership on
  the [[4.3]] Users page.

**Out:**
- User admin-flag / membership / passwords — [[4.3]].
- Moving org *creation* off the Prompts page — keep the existing create action too unless it's
  clearly redundant (decide at plan-time).

## Acceptance criteria
- [ ] An admin can list all organizations with member counts — proven by a test
- [ ] An admin can create, rename, and delete an organization (delete confirmed + cascades) — proven by Api + web tests
- [ ] The page is global-admin gated (non-admin 403 / no nav) — proven by a test
