## Problem
The registry **copies prompts in** today (spec 1.1), persisted by an EF Core (Postgres)
adapter behind the `IPromptRepository` port. That port was introduced precisely so
[Zatomic](#) — our own prompt/version tool — can become the backing store later without
touching Domain or Application. This spec builds that Zatomic-backed adapter so prompts and
versions have a single source of truth in Zatomic rather than a copied-in duplicate. "Shipped"
means the app can run against Zatomic as the prompt store, selected by configuration, with the
domain unchanged.

## Value
As a prompt owner I want the registry backed by Zatomic so that prompt/version identity lives
in one system and I don't maintain a separate copied-in registry.

## Scope
**In:**
- A `IPromptRepository` implementation backed by Zatomic (behind the existing port).
- Configuration to select the backing store (EF/Postgres vs Zatomic) at composition root.
- Read/write mapping between Zatomic's model and our `Prompt`/`PromptVersion` aggregate; a
  migration/coexistence story for prompts already copied in.
**Out:** Building Zatomic itself. Any change to the `Prompt`/`PromptVersion` domain (the port
contract is fixed by 1.1). Scoring/harness concerns.

## Acceptance criteria
- [ ] The app runs against Zatomic as the prompt store, selected by config, with no Domain/Application change — proven by tests + a run
- [ ] `IPromptRepository` operations (add, get-with-history, list, save) round-trip through Zatomic — proven by an integration test
- [ ] A documented coexistence/migration path for already-copied-in prompts — proven by inspection + a test
