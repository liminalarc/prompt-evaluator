## Problem
[[1.5]] built the eval-runner's multi-provider abstraction (thin per-provider adapters
behind a `Provider` protocol + a model-id → provider registry) and shipped Anthropic + OpenAI
adapters. Some apps we evaluate run their SLMs on **Modal** (e.g. a Stormboard SLM), which is
a bespoke HTTP endpoint, not a standard chat-completions provider. 1.5 made the registry
accommodate a custom provider and documented how to add one, but deliberately did not build a
live Modal adapter — we had no reachable Modal endpoint or credentials to test against. This
spec builds and verifies that adapter once the endpoint contract is known.

Origin: re-homed from 1.5's deferral protocol (see 1.5 `deferrals:` → `to: 1.12`).

## Value
As a prompt owner I want to run and score prompts whose subject model is an app's Modal-hosted
SLM, so the harness covers those models the same way it covers Anthropic/OpenAI.

## Scope
**In (to be refined once the Modal endpoint contract is confirmed):**
- A Modal-backed `Provider` adapter plugged into the existing registry, selected by model id
  (prefix/config).
- Subject execution against the Modal endpoint; structured output only if the endpoint
  supports it, else scope structured judging/generation to providers that do.
- Modal credentials via env, documented in README / `.env.example`; missing creds fail clearly.

**Out:** the provider abstraction itself (done in 1.5); any Domain/Application change.

## Acceptance criteria
- [ ] eval-runner executes a prompt against the Modal endpoint selected by model id — proven by a test (client mocked)
- [ ] Modal credentials are env-configured and documented; missing creds fail clearly
