## Problem
The harness runs `system prompt + user input` against a model in one shot; it does not replay an app's
**tool loop** (`web_search` / `web_fetch`). Prompts whose real behavior *is* the tool loop can't be
faithfully evaluated. Surfaced dogfooding real prompts ([[5.1]] finding **F3**): it blocks **3 Cortex
Golf prompts** (`facility-enrichment`, `routing-hole-search`, `scorecard-search` — all `SearchModel` +
web tools). **Shipped** = a prompt that declares tools can be evaluated either by enabling the tool(s)
during the run, or by replaying **frozen** tool results captured into the fixture (evaluate the
reasoning, not live retrieval).

## Value
As a maintainer onboarding a tool-using prompt I want the harness to account for its tools so its score
reflects real behavior instead of a crippled one-shot run.

## Scope
**In:** a way to evaluate tool-augmented prompts — declare tools on a version and/or capture frozen
tool-call/result pairs into a fixture that the runner replays; the eval-runner honors it (Anthropic
tool-use). Decide per prompt: live tools vs frozen replay (frozen = deterministic, preferred for
regression tracking).
**Out:** building app-specific tools (we consume the provider's `web_search`/`web_fetch` or replay
frozen results, not reimplement the apps' `HttpClient` loops); arbitrary custom-tool execution
sandboxes; non-web tools unless a prompt needs them.

## Acceptance criteria
- [ ] A version can declare tool(s) and/or a fixture can carry frozen tool results — test
- [ ] Eval-runner runs a tool-using prompt (live or frozen-replay) end-to-end and scores — pytest + integration
- [ ] Frozen replay is deterministic (same fixture → same tool inputs) — test
- [ ] Non-tool prompts unaffected — test
