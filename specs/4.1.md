## Problem
LitmusAI is single-user with no login today. To become a multi-user product it needs
**authentication** (login/logout, sessions, password reset + forgot) and **authorization**
(access scoped to the **organization** boundary from 1.9 — which superseded 1.7's top-level-folder
boundary). We model the flows on **Cortex Golf**, but adapt its architecture: Cortex Golf is
micro-services, whereas we are a **distributed monolith** — so Identity is a **bounded context
inside the single `Api` composition root** (one identity store, in-process authZ), and the
eval-runner is treated as an **internal trusted service**, not a user-auth boundary. "Shipped"
means users authenticate and their access to prompts/folders is enforced by **per-organization**
permissions (resolved from `Prompt.OrganizationId`) across API and SPA.

## Value
As an operator I want users to log in with access scoped to the organizations they're permitted, so
that multiple people can use LitmusAI safely.

## Scope
**In:**
- **Identity bounded context**: user accounts, hashed credentials, sessions/tokens.
- **Auth flows**: login / logout, password **reset** and **forgot** (email), and initial user
  provisioning / **self-service registration**.
- **Authorization**: permissions granted per **organization** (from 1.9); its folders/prompts
  inherit. Enforced in the API; Angular route guards + conditional UI. The org switcher lists only
  orgs the user can access (1.9 ships it unfiltered).
- **Service auth**: eval-runner as an internal trusted service (network/service credentials),
  distinct from user auth.
- Architecture: update the DDD layering docs (`CLAUDE.md`) for the new Identity context when built.

**Out:** Fine-grained per-prompt/per-folder ACLs (the organization is the boundary). Billing/teams
beyond org permissions. **SSO / OAuth** → re-homed to **4.2**. **Concrete hosted email provider**
(the `IEmailSender` seam + reset flow ship here; wiring a real provider) → re-homed to **3.2**.

## Acceptance criteria
- [ ] A user can register, log in/out, and reset / recover a forgotten password — proven by tests + a run
- [ ] The API enforces organization-scoped permissions on prompt/folder access — proven by API tests (allowed vs forbidden)
- [ ] The SPA guards routes and hides unauthorized actions/UI (incl. the org switcher) — proven by unit tests + one e2e
- [ ] The eval-runner is reachable only via service auth, not user credentials — proven by inspection + a test

## Decisions
- 2026-07-12 — Modeled on **Cortex Golf**, **adapted to our distributed monolith**: Identity is
  an in-process bounded context in the `Api` composition root (one identity store), not a
  separate auth service; eval-runner is an internal trusted service. Own credentials
  (reset/forgot implies email+password).
- 2026-07-13 — Permission boundary is the **organization** (per 1.9), resolved from
  `Prompt.OrganizationId` — 1.9 superseded the earlier top-level-folder boundary (1.7).
- 2026-07-14 — **Reference reconciliation** (from Cortex Golf `C:\Development\code\Golfstat\golfstat`
  + Stormboard `C:\Development\stormboard`): Golfstat is a Duende OIDC/JWT microservice stack — too
  heavy and the wrong shape (separate token server) for our in-process context. **Adopt
  Stormboard's architecture** — cookie-based **ASP.NET Core Identity in the single `Api`**, two-tier
  authZ (route `[Authorize]` + in-handler org-membership check), Data-Protection keys in the DB.
  **Adopt Golfstat's practices** — enumeration-resistant forgot/reset + security-stamp bump, and the
  env-toggled `IEmailSender` (SMTP / NoOp) seam.
- 2026-07-14 — **ASP.NET Core Identity behind a port** (user decision): `UserManager`/token
  providers live in Infrastructure behind `IUserDirectory`; Domain stays framework-free; cookie
  sign-in/out is an Api concern (never in Application). Chose Identity over hand-rolling for proven
  PBKDF2 hashing + reset/confirm tokens; both reference apps do the same.
- 2026-07-14 — **Cookie session, not JWT** (user deferred to best practice): same-origin SPA + a
  single in-process `Api` means cookies avoid signing-key/refresh/token-storage machinery and match
  "in-process authZ". JWT's cross-service advantage doesn't apply to a monolith.
- 2026-07-14 — **Self-service registration** (user decision): public register creates an account
  with no org access; org membership is granted separately. Seeded bootstrap admin + Default org
  solves first-run.
- 2026-07-14 — **Deferrals (user decisions at plan-time):** concrete hosted email provider →
  **3.2** (build the seam + flow here); **SSO / OAuth → new spec 4.2**. Both cross-linked.
- 2026-07-14 — **Auth-hardening deferrals (user decision mid-build, slices 2–3):** immediate
  live-session invalidation on reset (SignInManager + SecurityStampValidator) and Data-Protection
  key persistence to the DB both → **3.2**. Both only bite in a deployed multi-instance environment;
  4.1 ships per-process cookies + a password-changing reset (AC met). Sign-in issues the cookie
  manually via `HttpContext.SignInAsync`; reset rotates the security stamp but it isn't validated on
  the cookie yet.
