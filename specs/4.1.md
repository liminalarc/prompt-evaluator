## Problem
LitmusAI is single-user with no login today. To become a multi-user product it needs
**authentication** (login/logout, sessions, password reset + forgot) and **authorization**
(access scoped to the **top-level folder** boundary from 1.7). We model the flows on **Cortex
Golf**, but adapt its architecture: Cortex Golf is micro-services, whereas we are a **distributed
monolith** — so Identity is a **bounded context inside the single `Api` composition root** (one
identity store, in-process authZ), and the eval-runner is treated as an **internal trusted
service**, not a user-auth boundary. "Shipped" means users authenticate and their access to
prompts/folders is enforced by top-level-folder permissions across API and SPA.

## Value
As an operator I want users to log in with access scoped to the folders they're permitted, so
that multiple people can use LitmusAI safely.

## Scope
**In:**
- **Identity bounded context**: user accounts, hashed credentials, sessions/tokens.
- **Auth flows**: login / logout, password **reset** and **forgot** (email), and initial user
  provisioning / registration.
- **Authorization**: permissions granted per **top-level folder** (from 1.7); subfolders
  inherit. Enforced in the API; Angular route guards + conditional UI.
- **Service auth**: eval-runner as an internal trusted service (network/service credentials),
  distinct from user auth.
- Architecture: update the DDD layering docs (`CLAUDE.md`) for the new Identity context when built.

**Out:** Fine-grained per-prompt ACLs (top-level folder is the boundary). Org/billing/teams
beyond folder permissions. SSO / OAuth (possible later addition — see open questions).

## Acceptance criteria
- [ ] A user can register, log in/out, and reset / recover a forgotten password — proven by tests + a run
- [ ] The API enforces top-level-folder-scoped permissions on prompt/folder access — proven by API tests (allowed vs forbidden)
- [ ] The SPA guards routes and hides unauthorized actions/UI — proven by unit tests + one e2e
- [ ] The eval-runner is reachable only via service auth, not user credentials — proven by inspection + a test
