## Problem
Origin: re-homed from **1.7**. In 1.7 a `Dataset` became owned by exactly one `Prompt`
(datasets "live together" with the prompt). That was the right default — fixtures are inputs
shaped to a single prompt's contract, and the genuinely reusable part (scorers) is already
composable per dataset. But it forecloses one real case: a **cross-app standard benchmark**
(e.g. a safety / PII / JSON-validity suite) that an owner wants to apply to *many* prompts.

This spec exists to hold that need until it's real. It is **not** started speculatively — it
is the deliberate home for the sharing capability so 1.7 could exclude it without losing it.

## Value
As a prompt owner I want to reuse a curated dataset across multiple prompts so that I can run
a common benchmark (safety, format-validity, etc.) against several prompts without rebuilding
its fixtures each time.

## Scope
**In:** _to be decided when the need is real._ Two candidate shapes (leaning to the first, per
the 1.7 discussion):
- **Clone** — copy a dataset (and its fixtures) under a target prompt as a new owned dataset.
  Keeps 1.7's single-owner model intact; no many-to-many. Simplest.
- **Many-to-many association** — a dataset attachable to multiple prompts. More powerful, more
  invariants (input-contract compatibility, deletion semantics, whose analytics).

**Out:** Anything already delivered by 1.7 (prompt-owned datasets, the unified workspace).

## Acceptance criteria
- [ ] _drafted when scheduled_ — a dataset's fixtures can be reused against a second prompt
  (via clone or association, per the chosen shape).
