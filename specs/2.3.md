## Problem
Spec 1.1 shipped a deliberately **plain** authoring UI — bare forms for creating a prompt and
adding a version, a raw textarea for content. It works, but it isn't pleasant or guided. This
spec is the agreed "improve the flow/visuals later" home for that UI: make authoring clean and
guided **without changing the API or domain**. "Shipped" means creating prompts and versions
feels polished and validated, on the same endpoints.

## Value
As a prompt owner I want a clean, guided UI for creating prompts and versions so that authoring
is fast and pleasant rather than filling in a bare form.

## Scope
**In:**
- Refined create / add-version forms: inline validation, friendly error messages, clear
  empty / loading / success states.
- A better content editor (monospace, comfortable sizing, character count / placeholder-token
  hints as useful).
- Layout, spacing, and affordance polish — all via **brand-tokens** (`--sb-*`, no hex).
- Optionally revisit 1.1's "create + first version are separate calls" **at the UI layer only**
  (a combined new-prompt-with-first-version flow) — the API contract is unchanged.

**Out:** Any API/domain change. File / bulk import (**1.6**). Folder navigation (**1.7**).
Human-review UI (2.1).

## Acceptance criteria
- [ ] The create / add-version flow is redesigned with inline validation and clear states — proven by unit tests + one e2e
- [ ] Styling is brand-tokens only (no hardcoded hex); light/dark both work — proven by inspection + tests
- [ ] No change to the prompts API or domain — proven by the existing 1.1 tests staying green
