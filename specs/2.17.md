## Problem
Eval runs execute **synchronously**: the run endpoint runs the whole batch inline (every fixture's subject
generation + judge call) and only responds when done, holding an HTTP request open. 5.1 hit the ceiling live —
a round-debrief run **502'd twice** at the ~100s boundary. The interim band-aid (HttpClient timeout 100s → 5
min, [[2.12]]) bought headroom for today's sizes, but a synchronous request can't hold open for a multi-minute
run behind App Runner's router, and it blocks bigger datasets, heavier prompts, and repeated runs ([[2.14]]).

## Value
As an operator I want to kick off a run and have it execute as a background job I poll for status/results — so
runs never time out, and bigger datasets, heavier prompts, and N-repeat variance runs all become practical.

## Scope
**In:** kick off a run as a **job** (return immediately with an id), execute the batch server-side, **poll**
for status + results; the run page shows progress and fills in as fixtures complete. Removes the synchronous
timeout ceiling. **Repeat-N runs (F4):** a **run-count** control on the run form fires N runs of the same
`version × dataset × scorers` in one action (async makes this practical), each persisted as its own run,
feeding the [[2.14]] variance view directly — so run-to-run stability ("one run lies", R4) is one click, not
hand-repetition.
**Out:** distributed job queues / horizontal workers beyond what dev needs (keep it simple first); the
interim timeout band-aid already shipped ([[2.12]]).

## Acceptance criteria
*(Concept — sharpen when picked up.)*
- [ ] A run returns immediately with a job id; status/results are polled; a run longer than the old sync ceiling completes.
- [ ] The run page shows progress and finalizes without a held-open request or a gateway timeout.
- [ ] The run form accepts a **run count N**; submitting fires N runs (each persisted separately) and the
  variance view aggregates them; progress reflects all N. (F4)
