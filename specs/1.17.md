## Problem
`Fixture.Input` is a `string` (`src/Domain/Fixture.cs`), so a fixture can only hold text. Prompts whose
real input is an **image** (satellite tiles, scorecard photos) can't be represented, so they can't be
walked end-to-end. Surfaced dogfooding real prompts ([[5.1]] finding **F2**): it blocks **6 Cortex Golf
prompts** (`auto-map-centerline-detection`, `course-layout-analysis`, `overview-map-registration`,
`per-hole-polygon-detection`, `scorecard-extraction`, `routing-review`). **Shipped** = a fixture can
carry image(s) alongside/instead of text, the eval-runner sends them as image content blocks to the
provider, and the UI has an image-upload affordance.

## Value
As a maintainer onboarding a vision prompt I want fixtures that carry images so I can evaluate and
improve image-input prompts, not just text ones.

## Scope
**In:** multimodal `Fixture` input (text + one or more images; images stored as blobs/refs, not inline
JSONB bloat); eval-runner passes image blocks to the provider (Anthropic default) per its multimodal
API; capture/UI affordance to attach images to a fixture; scorers operate on the (text) model output as
today.
**Out:** generating synthetic *images* (synthetic fixtures stay text-seeded — 1.2); non-image modalities
(audio/PDF-as-native) unless a prompt needs it; vision **output** scoring beyond existing scorer kinds.

## Acceptance criteria
- [ ] A fixture can carry one or more images (+ optional text); persisted without corrupting text fixtures — test
- [ ] Eval-runner sends images as provider image blocks; a vision prompt runs end-to-end and scores — pytest + integration
- [ ] UI attaches/preview images on a fixture; image fixtures render in the fixtures table — unit + e2e
- [ ] Text-only fixtures/behavior unchanged (backward compatible) — test
