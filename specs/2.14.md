## Problem
A single eval run is **noisy**, and the aggregate score **hides the signal that actually decides a call** —
which lives in the judge's *rationale*, not the number. 5.1 proved both, hard: round-debrief v6's sparse
fixture scored **0.82 then 0.72** across two identical runs, and the whole v6-vs-v7 (and v1-vs-v5) decision
turned on *why* the judge deducted, not the delta (tied scores hid a prediction-risk difference). The tool
shows one run's scalar + a score delta, so an operator (me, during the walk) **made a wrong backport call off
a single run**, and every real comparison required **manually re-running 2–3×** and **hand-reading each
rationale**. That toil compounds over 40+ remaining prompts.

## Value
As an operator I want to see how stable a score is across repeated runs and to diff the judge's *reasoning*
between versions — so I don't mistake noise for signal, don't read a flat number as "no change," and don't
have to re-run and hand-read rationales by hand every time.

## Scope
**In:**
- **Variance view.** Aggregate **repeated runs** of the same `Version × Dataset × Scorer` into **mean ±
  spread** (per fixture and overall). A one-click "run N×" (or aggregate existing repeats) and a chart/table
  that shows the spread, so a wobble reads as noise. Surfaces the distribution, not a lone point.
- **Rationale-diff in Compare.** For each fixture, show the LLM-judge **rationale for each version side by
  side** (the "why"), not only the score delta — the view that would have made v6-vs-v7 obvious.
- **Variance-aware comparison.** A version delta that sits **within the measured spread** is flagged as
  *within noise*, not a confident move (ties into [[1.11]] small-sample regression confidence).

**Out:** restructuring what the judge emits — per-criterion / failure-mode / severity tagging is [[2.15]]
(R7); this spec surfaces the **existing free-text rationale** and repeats existing runs, no judge change.
Data-conditional rubrics → [[2.16]]. Async run execution → [[2.17]] (though repeated runs make async more
valuable, they don't depend on it).

## Acceptance criteria
- [x] Repeating a run N× shows **per-version mean ± spread** (per fixture + aggregate) in Analytics — unit (e2e in the e2e/UX pass)
- [x] Compare shows the **judge rationale for each side, per fixture** (rationale-diff), not just the score delta — unit
- [x] A version-over-version delta **within the measured spread** is labeled *within noise* (not a confident change) — unit
- [x] Existing single-run flows are unchanged when only one run exists (graceful n=1) — test

## Backfill (no migration needed)
Both features read data **already present** on every existing run — nothing to backfill (user asked):
- **Variance** aggregates each version's existing per-fixture scores; the round-debrief v6/v7 double-runs
  already produce spread today.
- **Rationale-diff** surfaces `Score.Detail`, which `RunEvaluationHandler` has persisted from the judge's
  rationale (`LlmJudgeScorer` → `ScoreOutcome.Detail`) since the eval harness shipped ([[1.3]]) — so every
  past run already carries it. Opening Analytics/Compare on any existing prompt just works.
- **Principle for future features** (e.g. [[2.15]] structured judging, which *does* need new judge output):
  the subject **outputs are stored** on each `FixtureRun`, so a backfill can **re-judge stored outputs**
  (re-run only the judge, no re-generation) rather than a full re-run — cheaper, and it's the intended
  backfill path when a feature needs data not already captured.
