## Problem
The app grew slice-by-slice and now reads as disconnected. Concrete failures found in a
full sweep of `web/`:

- **Navigation is flat and incomplete.** Topbar is `Home · Prompts · Datasets · Analytics`.
  `Home` is the walking-skeleton echo toy (`/api/echo`) — a dev artifact is the landing page.
  Eval runs have no nav entry (only reachable by drilling into a dataset; the run page's
  back-link points at `/datasets`, not its own dataset/prompt).
- **Organization** — the top-level boundary (1.9) — exists only as a dropdown inside `/prompts`;
  Datasets and Analytics ignore it, so the mental model breaks off the Prompts page.
- **The same Prompt×Dataset relationship is expressed three ways** — embedded in the prompt
  workspace (1.7), as a top-level `/datasets` list, and again behind `/analytics`'s own
  pickers — forcing re-selection and page-bouncing.
- **Every page is a long scroll of stacked bare forms** with no chrome: no shared header,
  no breadcrumb standard, no loading states (pages render blank until data arrives), and a
  generic "is the stack running?" box for every error.
- **The rich brand vocabulary is unused** — `.sb-badge`, `.sb-chip`, `.sb-card`, button
  variants/sizes, `.sb-field--error` all ship in `brand-tokens` but status is raw emoji and
  plain text, which is why it reads flat.
- **The core eval loop is invisible** — nothing narrates *author → dataset → scorers → run →
  scores → regressions → improve*, and there's no "how are my prompts doing?" view.

"Shipped" means the app is coherent and consistently navigable, reflects the brand, and
narrates the eval loop — **with no API or domain change**.

## Value
As an operator (prompt engineer) I want a coherent, consistently-navigable app that narrates
the eval loop and uses our brand system, so that I can move from prompt → dataset → run →
scores → regressions without re-orienting on every page.

## Scope
**In:**
- **App shell & navigation.** Redesigned topbar; a **persistent global organization context**
  (small signals-based org-context store) that scopes prompts, datasets, analytics, and the
  dashboard consistently; org switcher moves to the topbar. Consistent breadcrumb/back across
  detail pages. Runs made discoverable; the run page links back to its dataset **and** prompt.
- **Landing dashboard** (`/`) replacing the echo skeleton: for the selected org — prompts with
  their latest scores, recent eval runs, and open regression flags. Built on **existing read
  APIs**; net-new presentational components, container fetches.
- **Consistent page chrome.** A shared UI kit under `web/src/app/shared/`: PageHeader
  (title/subtitle/primary-action slot), Breadcrumb, and standard Loading / Empty / Error
  states, applied to every page.
- **Design-system pass.** Adopt brand primitives consistently — badges/chips for origin
  (Captured/Synthetic), scorer kind, pass/fail, target/judge model, regression severity;
  `.sb-card` grouping; button variants/sizes; `.sb-field--error` for validation. Replace raw
  emoji/plain-text status. No hardcoded hex; light + dark both verified.
- **IA reconciliation & progressive disclosure.** Resolve the three-way Prompt×Dataset
  duplication — the prompt workspace is the hub; `/datasets` and `/analytics` become clearly
  scoped secondary views (exact treatment decided in the design slice). Group the long stacked
  pages into setup / data / action, revealing secondary forms rather than always showing them.
- Retire the echo skeleton from product nav (keep the round-trip only if a smoke test needs it,
  off the primary path).

**Out (coordinated, not absorbed — cross-linked both ways):**
- **Any API/domain change.** UI-only. If the dashboard needs an aggregate the read APIs can't
  cheaply serve, it is **re-homed to a small API spec via the deferral protocol** — never added
  silently.
- **Authoring form redesign** (guided create/add-version, content editor) → stays **2.3**; this
  spec supplies the chrome/primitives 2.3 builds on.
- **Human-review UI** → stays **2.1**; this spec's shell/nav will host it.
- **New analytics content / new charts** → **1.4** owns analytics; this spec only restyles
  existing charts.

## Acceptance criteria
- [ ] Topbar redesign with a global org switcher scoping prompts/datasets/analytics/dashboard; changing org updates all surfaces — proven by org-context store unit tests + one e2e
- [ ] Landing dashboard at `/` shows the selected org's prompts+latest scores, recent runs, and open regressions; echo skeleton removed from nav — proven by unit + e2e
- [ ] Shared chrome (header/breadcrumb/loading/empty/error) applied to every page; a loading state renders before data arrives — proven by unit tests + inspection
- [ ] Brand primitives adopted (badges/chips/cards, button variants, field-error) replacing raw emoji/plain status; no hardcoded hex; light + dark verified — proven by inspection + a hex-scan of web sources + tests
- [ ] Runs are discoverable and a run links back to its dataset and prompt — proven by unit + e2e
- [ ] Consistent breadcrumb/back across all detail pages — proven by inspection + tests
- [ ] No prompts API/domain change and no new endpoints — proven by existing backend + web tests staying green (any needed aggregate re-homed via deferral)

  any aggregate that would need a new endpoint is re-homed via the deferral protocol, never
  added silently (keeps the "no API/domain change" guarantee intact).
- 2026-07-13 — Design constraint: **brand-tokens only** (`--sb-*`, no hex); the overhaul leans
  on already-shipped-but-unused primitives (`.sb-badge`, `.sb-chip`, `.sb-card`, button
  variants, `.sb-field--error`) rather than inventing new styles.
