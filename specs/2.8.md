## Problem
Dogfooding LitmusAI on real prompts ([[5.1]]) surfaced a batch of eval-loop UI bugs and UX
inconsistencies — the first real onboarding run failed **silently**, the run picker **leaks across orgs**,
and add/edit affordances differ per entity. [[2.5]] was round 1 of the eval-loop UX; this is **round 2**,
driven by the 5.1 findings log (`specs/5.1/findings.md`, B1–B5 + U1–U9). "Shipped" = the eval loop fails
loudly, stays org-scoped, and offers a consistent add/edit-with-description surface across versions,
datasets, fixtures, and scorers.

## Value
As an operator onboarding prompts I want the eval-loop UI to surface errors clearly, stay within my org,
and let me name/annotate and edit what I create, so the register→dataset→score→run loop is smooth and
trustworthy.

## Scope
**In (5.1 findings B1–B5, U1–U9):**
- **Loud failures** — a failed run shows a clear error (banner/toast), not a bare 500; a missing/invalid
  provider key reports "eval-runner: <provider> not configured" (ideally a preflight before the run). [B1,B2]
- **Run picker scoped to the dataset's prompt** — a dataset belongs to one prompt, so the run form fixes
  the prompt to `Dataset.PromptId` (pick a version only); removes the cross-org `listPrompts()` leak. [B3]
- **Rubric is a `<textarea>`** (multi-line); **scorer config required per kind** (Regex/JsonSchema). [B4,B5]
- **Create-prompt navigates to the new prompt's workspace** (`/prompts/:id`). [U1]
- **Version history = collapsible list** (progressive disclosure, 2.4 pattern); **version label/description
  editable** — content + target model stay immutable (change the model by adding a version). [U2,U3,U4]
- **Descriptions** — dataset **create** form gains Description [U5]; **fixtures gain a label/description**
  (`Fixture.Label`), shown in the table instead of the raw input. [U7]
- **Consistent add/edit table pattern** for fixtures [U6] and scorers [U9] (add or edit a row → reveal
  fields → save).
- **Fixture origin honesty** — manual entry can mark **Synthetic**, not only Captured. [U8]
- **LLM-judge budgets for thinking-on-by-default judge models** *(eval-runner)* — size the judge's
  output budget so adaptive-thinking judges (Sonnet 5 / Fable 5) don't truncate the verdict JSON →
  500. Model-agnostic (no per-model `thinking`/`effort` flags). [B6]

**Out:** system-prompt vs templated prompt *shape* (D1 — decision logged in 5.1 findings, no spec yet);
the dev-deploy provider-key check (O1 → [[3.2]] / infra runbook); any change to the immutable content/model
rule.

## Acceptance criteria
- [ ] A failed run surfaces a clear error (not a silent 500); a missing provider key is named — test + manual
- [ ] Run form is fixed to the dataset's owning prompt; no cross-org prompts appear — test
- [ ] Rubric is multi-line; Regex/JsonSchema config is required (validated) — test
- [ ] Create-prompt lands on the new prompt's workspace — test
- [ ] Version history is a collapsible list; label/description editable, content/model not — test
- [ ] Dataset create form + fixtures carry a description/label, shown in tables — test
- [ ] Fixtures and scorers use the add/edit-then-details pattern — test
- [ ] Manual fixture entry can be marked Synthetic — test
- [x] LLM-judge doesn't 500 on a thinking-on-by-default judge model (Sonnet 5) — proven by an eval-runner pytest [B6]

## Decisions
- 2026-07-18 — Created from the 5.1 dogfood findings (round 2 of [[2.5]]'s eval-loop UX). Content and
  target model stay **immutable** (run identity); only version label/description become editable. Run picker
  is scoped to the dataset's owning prompt (fixes both the org leak and the conceptual oddity of a free
  prompt choice on a prompt-owned dataset).
