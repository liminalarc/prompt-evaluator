## Problem
There's no way to remove registry data once created. The **organization** delete endpoint exists
in the API (`DELETE /api/organizations/{id}`, cascades to folders/prompts/datasets) but is **not
exposed in the UI**, and there is **no delete at all** — API or UI — for **prompts**, **folders**,
or **datasets**. So a mistaken org, a throwaway prompt, or an experimental dataset is permanent,
which bites immediately when building up a testbed (surfaced during dogfooding: a leftover empty
"Test Organization" can't be removed). "Shipped" means an operator can delete an org, prompt,
folder, or dataset from the UI, with clear cascade semantics and a confirmation guard.

## Value
As an operator I want to delete organizations, prompts, folders, and datasets I no longer need, so
that the registry stays clean and mistakes aren't permanent.

## Scope
**In:**
- **Organization delete (UI)** — expose the existing cascade delete on the org (topbar or Prompts
  page), behind a confirmation. On deleting the current org, clear/repoint the org context.
- **Prompt delete** — new `DELETE /api/prompts/{id}` (cascade its versions, datasets, and those
  datasets' fixtures/scorer-configs/eval-runs/scores) + UI on the prompt workspace / list.
- **Dataset delete** — new `DELETE /api/datasets/{id}` (cascade fixtures, scorer configs, eval-runs,
  scores) + UI on the dataset page / prompt workspace.
- **Folder delete** — new `DELETE /api/folders/{id}`; decide contents behaviour (cascade vs. reparent
  children/prompts to the folder's parent/root) — pick reparent-to-parent unless a decision says
  otherwise + UI.
- **Confirmation UX** — destructive actions confirm first and state what cascades (e.g. "Deletes the
  prompt and its N versions, M datasets, and all runs").
- **AuthZ** — every delete is org-scoped per 4.1 (a non-member gets 403; enforce on the new endpoints
  the same way reads/writes are).

**Out:** Soft-delete / trash / restore (hard delete only for now). Bulk delete. Audit log of deletions.

## Acceptance criteria
- [ ] An operator can delete an organization from the UI (with confirmation); its folders/prompts/datasets cascade — proven by API test + e2e
- [ ] An operator can delete a prompt, a dataset, and a folder from the UI; cascades/reparenting behave as specified — proven by API tests (allowed + cascade) + e2e
- [ ] Every delete endpoint enforces org-scoped authorization (non-member → 403) — proven by API tests
- [ ] Deletes confirm first and state what will cascade — proven by unit test + e2e

## Decisions
- 2026-07-14 — Created during dogfooding: no way to delete a leftover org/prompt/dataset. Org delete
  already cascades server-side (from 1.9); this spec exposes it and adds prompt/folder/dataset delete.
  Hard delete only (no trash) for now.

## Open questions
- Folder delete: cascade its contents, or reparent children + prompts to the folder's parent/root?
  (Leaning reparent — less destructive.) Resolve at plan-time.
