## Problem
There's no way to remove registry data once created. The **organization** delete endpoint exists
in the API (`DELETE /api/organizations/{id}`, cascades to folders/prompts/datasets) but is **not
exposed in the UI**, and there is **no delete at all** ‚Äî API or UI ‚Äî for **prompts**, **folders**,
or **datasets**. So a mistaken org, a throwaway prompt, or an experimental dataset is permanent,
which bites immediately when building up a testbed (surfaced during dogfooding: a leftover empty
"Test Organization" can't be removed). "Shipped" means an operator can delete an org, prompt,
folder, or dataset from the UI, with clear cascade semantics and a confirmation guard.

## Value
As an operator I want to delete organizations, prompts, folders, and datasets I no longer need, so
that the registry stays clean and mistakes aren't permanent.

## Scope
**In:**
- **Organization delete (UI)** ‚Äî expose the existing cascade delete on the org (topbar or Prompts
  page), behind a confirmation. On deleting the current org, clear/repoint the org context.
- **Prompt delete** ‚Äî new `DELETE /api/prompts/{id}` (cascade its versions, datasets, and those
  datasets' fixtures/scorer-configs/eval-runs/scores) + UI on the prompt workspace / list.
- **Dataset delete** ‚Äî new `DELETE /api/datasets/{id}` (cascade fixtures, scorer configs, eval-runs,
  scores) + UI on the dataset page / prompt workspace.
- **Folder delete** ‚Äî new `DELETE /api/folders/{id}`; decide contents behaviour (cascade vs. reparent
  children/prompts to the folder's parent/root) ‚Äî pick reparent-to-parent unless a decision says
  otherwise + UI.
- **Confirmation UX** ‚Äî destructive actions confirm first and state what cascades (e.g. "Deletes the
  prompt and its N versions, M datasets, and all runs").
- **AuthZ** ‚Äî every delete is org-scoped per 4.1 (a non-member gets 403; enforce on the new endpoints
  the same way reads/writes are).

**Out:** Soft-delete / trash / restore (hard delete only for now). Bulk delete. Audit log of deletions.

## Decisions
- 2026-07-14 ‚Äî Created during dogfooding: no way to delete a leftover org/prompt/dataset. Org delete
  already cascades server-side (from 1.9); this spec exposes it and adds prompt/folder/dataset delete.
  Hard delete only (no trash) for now.
- 2026-07-14 ‚Äî **Folder delete = reparent** (resolved): a folder's child folders and filed prompts
  move up to its parent (or the org root when top-level), then the empty folder is removed. Least
  destructive.
- 2026-07-14 ‚Äî **Cascade is explicit for analytics, FK-driven for the aggregate.** `datasets`
  (FK ‚Üí `prompts`, cascade) and the owned collections (`prompt_versions`, `fixtures`, `fixture_runs`,
  `scores`) already cascade at the DB level. `eval_runs` and `scorer_configs` reference their
  prompt/dataset by id **without an FK**, so prompt/dataset delete removes them explicitly (in a
  transaction) rather than adding new FK constraints + a migration. Delete on `eval_runs` still
  cascades to `fixture_runs`/`scores` via their real FKs.
- 2026-07-14 ‚Äî **Confirmation is a tokenized in-app dialog** (`ConfirmService` + `ConfirmDialog` in
  `shared/`, mounted once in the app shell), not `window.confirm` ‚Äî consistent, themed, and testable.

## Known follow-up ‚Äî RESOLVED in this spec
- **Org delete orphaned `eval_runs` / `scorer_configs`** (pre-existing since 1.9: those tables have
  no FK to their prompt/dataset, so a bare org FK-cascade left them behind). Fixed:
  `OrganizationRepository.DeleteAsync` now explicitly removes the org's prompts' eval-runs and
  scorer-configs (in a transaction, mirroring `PromptRepository`) before the org cascade ‚Äî so AC1's
  "‚Ä¶datasets cascade" is truly complete. Covered by a new Infrastructure round-trip test
  (`Deleting_an_org_also_removes_its_orphan_eval_runs_and_scorer_configs`). No new deferral.
