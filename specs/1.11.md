## Problem
Regression detection (1.4) flags a version only when the mean score drop exceeds a threshold (0.05)
**and** a paired t-test finds it significant (p < 0.05). The significance guard returns `null` for
fewer than 2 matched fixtures, so a **catastrophic drop on a single fixture** (e.g. 1.0 → 0.0, a
pass flipping to a fail) is **not flagged at all** — surfaced during dogfooding, where a deliberately
worse prompt version dropped the only labeled case from 1.0 to 0.0 and the Regressions panel stayed
empty. Two issues: (1) the empty-state copy is misleading, and (2) whether severe small-sample drops
should be surfaced is an open product call. "Shipped" means the analytics view communicates
small-sample drops honestly (and, if decided, surfaces them as unverified).

## Value
As an operator I want the analytics to tell me clearly when a version got worse but the sample is
too small to be statistically certain, so that a real regression on one or two fixtures isn't
silently hidden.

## Scope
**In:**
- **Fix the misleading empty state.** "No regressions beyond the threshold" renders even when the
  drop far exceeds the threshold but significance couldn't be established (n < 2). Distinguish the
  cases: "no meaningful drop" vs. "drop detected but not statistically significant — add more
  fixtures to confirm."
- **Small-sample surfacing (product decision — see open question).** Decide whether a drop that
  clears the threshold but lacks significance (n < 2, or n ≥ 2 with p ≥ alpha) is surfaced as an
  **unverified / possible** regression (distinct styling) rather than omitted. Keep the confirmed
  flag semantics unchanged.
- Wherever the trend/regression/comparison views disagree in what they imply (the trend line and
  Compare view both show the drop; Regressions shows nothing), reconcile the messaging so they read
  consistently.

**Out:** Changing the core detector math (threshold + paired t-test + zero-variance→significant
short-circuit) unless the product decision requires it. New scorers. Backend perf.

## Acceptance criteria
- [ ] When a drop exceeds the threshold but significance can't be established, the UI says so explicitly (not "no regressions beyond the threshold") — proven by unit test + inspection
- [ ] The chosen small-sample behavior (surface-as-unverified or documented-omit) is implemented and tested — proven by Application tests over the detector/handler + a web unit test
- [ ] Confirmed-regression flagging (drop + significant) is unchanged — proven by existing 1.4 tests still green

## Decisions
- 2026-07-14 — Raised during dogfooding: v2 (naive prompt) dropped the single labeled fixture 1.0→0.0;
  Regressions panel said "No regressions beyond the threshold" though the drop was total. Root cause:
  `RegressionDetector.OneSidedDropPValue` returns null for n < 2, so no flag. Note the existing
  zero-variance short-circuit means adding a 2nd consistently-regressing fixture *would* flag it.

## Open questions
- Should a threshold-clearing drop with insufficient significance be surfaced as an **unverified**
  regression, or just messaged as "not enough fixtures"? (Leaning: surface as unverified, visually
  distinct from a confirmed flag.) Resolve at plan-time.
