## Problem
[[1.13]] added a workspace-level global-admin flag (`AppUser.IsAdmin`) but **no UI to manage it** —
today the only way to grant admin is the bootstrap-admin env var, and the only way to change org
membership is creating an org (grants the creator Owner). There's no admin surface to see users,
toggle their admin access, or adjust their org memberships/roles. Password reset is **email-only**
([[4.1]] `forgot-password`), which this deployment doesn't use, and there is **no self-service
change-password** at all. Admins need to manage people and access from inside the app, without email.

## Value
As a global admin I want to manage users — grant/revoke global-admin, add/remove their org
memberships and role, and reset a password — from an admin page. And as any signed-in user I want
to change my own password. All without any email flow.

## Scope
**In:**
- **Admin user-management page** (`/admin/users`, gated by the [[1.13]] global-admin flag; non-admins
  403 / no nav): lists users (email, display name, `isAdmin`, org memberships + role); lets an admin
  **toggle global-admin**, **grant/revoke org membership and set role** (Owner/Member — [[4.1]]
  `OrgRole`), and **set a new password** for a user directly.
- **Self-service change-password** for any signed-in user (enter current + new); wrong current is
  rejected. Replaces the email path for routine rotation.
- **Lockout guard:** the last remaining global admin cannot have their admin flag removed (prevents
  locking the workspace out of admin).
- Backend: admin-gated endpoints + a self change-password endpoint; `IUserDirectory` grows the needed
  operations (list users with memberships, revoke org membership, admin set-password, verify-and-
  change own password). Admin writes reuse `OrgAccess.IsGlobalAdminAsync`.

**Out:**
- **Email** of any kind — invites, email-based reset (the [[4.1]] `forgot-password` seam stays as-is).
- **Admin-created accounts** — users still self-register at `/register` (decided); admins manage
  existing users only.
- Deleting / deactivating user accounts (future if needed).
- SSO / OAuth — [[4.2]]. Fine-grained per-resource permissions beyond org membership + global-admin.

## Acceptance criteria
- [ ] An admin sees a list of users with their admin flag and org memberships — proven by Api + web tests
- [ ] An admin can grant/revoke global-admin; a non-admin gets 403 and no nav/page — proven by Api + web tests
- [ ] An admin can add and remove a user's org membership and set its role — proven by an Api test
- [ ] An admin can set a new password for a user; that password then logs in — proven by an Api test
- [ ] A signed-in user can change their own password (current + new); a wrong current is rejected — proven by an Api test
- [ ] The last global admin's admin flag cannot be revoked — proven by an Api test
