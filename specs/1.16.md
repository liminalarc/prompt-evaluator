## Problem
LitmusAI tracks `Prompt × Version` but has no concept of **which version the source app is actually
running**, so after an improvement wins in the harness there's no in-tool signal that a better prompt
is sitting unshipped. Surfaced dogfooding real prompts ([[5.1]] finding **F1**): the deployed/shipped
state lived only in 5.1's fill sheets, invisible inside the tool. **Shipped** = a noticeable per-version
status — `Current in source` · `Backport-eligible` · `Regressed` — plus a manual action to move the
Current marker after a backport, which clears the eligibility flag. LitmusAI **signals; it never edits a
source repo** — the backport itself is a human action in the source app's own process.

## Value
As a maintainer improving a prompt I want LitmusAI to show which version my app runs and flag when a
better one exists, so I can see at a glance what needs backporting and mark it done once shipped.

## Scope
**In:**
- **`Current in source` marker** — exactly one version per prompt, set manually (optional commit
  SHA + timestamp). Nullable until first set.
- **`Backport-eligible` flag** — derived: a version scores higher than `Current` on a shared dataset.
- **Mark-as-backported action** — move `Current` to a chosen version; the eligibility flag re-derives
  (clears when Current is the top scorer).
- **Version-status badges on the version itself** (Version history), unifying with **regression**
  marking — today regressions live only in the Analytics table (1.11); this surfaces `Regressed` as a
  badge on the version too.
**Out:** automated backporting / any write to a source repo (LitmusAI signals only). Registry-backed
deployment tracking → [[3.1]] (this is the standalone signal; 3.1 may later own the deployment source
of truth). "Better" = which scorer/aggregate decides eligibility beyond a simple higher-aggregate rule
→ revisit with [[2.9]] (weighted composite).

## Acceptance criteria
- [ ] A version can be marked `Current in source` (one per prompt); optional SHA/timestamp recorded — test
- [ ] `Backport-eligible` is derived and shown when a higher-scoring version exists above Current — test
- [ ] Mark-as-backported moves Current and the eligibility flag re-derives (clears when appropriate) — test
- [ ] Version history shows status badges (`Current` · `Backport-eligible` · `Regressed`) on the version — unit + e2e
- [ ] LitmusAI performs no source-repo write (signal-only) — verified by scope/review

## Plan (thin slices)
1. [ ] **Domain + persistence**: `Prompt.CurrentVersionId` (`Guid?`, nullable) + optional
   `CurrentVersionSha` / `CurrentVersionSetAt`; `SetCurrentVersion(versionId, sha?, at?)` invariant
   (version belongs to this prompt; exactly one current). Nullable-column migration on `EvalDbContext`
   (no DB FK to the owned `prompt_versions`, matching `EvalRun.PromptVersionId`). Set-current and
   mark-as-backported are the **same mechanic** (move Current to a chosen version); the UI labels it
   contextually. -> <sha>
2. [ ] **Derivation (Application)**: per-version status — `Current` (== pointer), `Backport-eligible`
   (interim rule below), `Regressed` (any [[1.11]] `RegressionFlag` with `ToVersionId == v`). Reuse
   `AnalyticsProjection` + trend means, keyed off `ScorerDescriptor.Identity`. -> <sha>
3. [ ] **API + UI**: `POST /api/prompts/{id}/versions/{versionId}/set-current` (optional SHA);
   version-status fields on the response; Versions-tab **badges** (`Current · Backport-eligible ·
   Regressed`) + a per-row set-current / mark-backported action + a compact **Deployment summary**
   card. -> <sha>

## Decisions
- 2026-07-18 — Standalone spec (user), not folded into [[3.1]]; the signal stands alone inside LitmusAI
  and assumes nothing about the source system (which may not be on our flow/spec process).
- 2026-07-18 — LitmusAI signals, never executes a backport (F1 design stance).
- 2026-07-18 (from 5.1 round-debrief) — the **`Regressed`** badge (and the derived flags) must compare runs
  that share the **same scorer config** — a mid-stream rubric change gives a version runs on two different
  yardsticks, and averaging across them hides a real version-over-version drop (v2→v3 regression didn't
  surface). Key regression/eligibility off same-scorer-config runs, not a blended per-version mean.
- 2026-07-19 (plan sign-off) — **Interim backport-eligibility rule:** a version is eligible when, over a
  shared dataset, it is **≥ Current on every same-scorer-config series and strictly higher on at least
  one** — surfaces real improvements, never flags a version that regressed on any scorer, and keys off
  `ScorerDescriptor.Identity` (no blending). The user noted scorers *should* be weighted (LLM judge >
  RegEx > JSON-content); that **weighted determination is deferred to [[2.9]]** (weighted composite feeds
  eligibility once weights exist) rather than a new spec — 2.9 already owns per-dataset scorer weights.
- 2026-07-19 (plan sign-off) — **Set-current == mark-as-backported** (one `SetCurrentVersion` mechanic /
  one `set-current` endpoint); the UI labels it contextually ("Set as current in source" on any version,
  "Mark backported → vN" on the eligible one). **UI = both** (user): status badges + per-row actions on
  the version-history rows **and** a compact Deployment summary card at the top of the Versions tab.

## Verification / evidence
Set Current → improve → see `Backport-eligible` → mark-backported → flag clears; badges visible on versions.
